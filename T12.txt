## 0. Executive Summary (TL;DR)

Проект демонстрирует **исключительный уровень для студенческой работы**: реализованы порты ключевых алгоритмов XCSoar, EKF для ветра, IGC-логгер, PCM-варио на AudioTrack, OSM-карта под компасом. Это не "лабораторная на коленке", а осмысленный продукт с пониманием доменной области.

**Критические проблемы:**

1. **Компас**: несмотря на использование `TYPE_ROTATION_VECTOR` (исправлено в v0.2.13), фильтрация heading всё ещё вызывает дрожание ±2-3° в статике из-за неправильных коэффициентов α-β фильтра и слишком агрессивного deadband
2. **Архитектура**: `MainActivity.java` — 2700+ строк God Object, смешивающий отрисовку Canvas, бизнес-логику, sensor wiring, GPS-трек, TTS, диалоги
3. **Потокобезопасность**: race conditions в `rotationMatrix` и `headingFilter` между sensor thread и UI thread
4. **Производительность**: аллокации в `onDraw()` (Paint, float[], Rect, String.format) создают GC-pressure 30-60 раз в секунду
5. **Отсутствие тестов**: критичные алгоритмы (HeadingFilter, CirclingManager, ThermalDetector) не покрыты unit-тестами

**Вердикт**: 7/10 для студенческого проекта, 4/10 для production-полётов. Требуется 1-2 спринта рефакторинга.

## 2. Компас: главная проблема (Deep Dive)

### 2.1. Симптом

> "Прибор лежит на столе. Стрелка компаса дрожит в пределах 2-3 градусов. При резком повороте — компас плавно поворачивается."

Этот симптом — **классическая карта отказов α-β фильтра с завышенным α и слишком малым deadband**.

### 2.2. Цепочка обработки heading (v0.2.13)

```
TYPE_ROTATION_VECTOR (sensor thread, 25 Hz)
  ↓
SensorController.onSensorChanged(ROTATION_VECTOR)
  ↓
SensorManager.getRotationMatrixFromVector(rotationMatrixInternal, event.values)
  ↓
SensorManager.getOrientation(rotationMatrixInternal, orientationVals)
  ↓
rawHeading = Math.toDegrees(orientationVals[0])
  ↓
HeadingFilter.update(rawHeading, elapsedRealtime)
  ↓
  ├─ CLAMP (|Δheading| > 70°/с → limit)
  ├─ MEDIAN FILTER (окно 5)
  ├─ Alpha-Beta filter (α=0.15, β=0.02) с unwrap
  └─ DEADBAND (адаптивный: 2.5° в статике, 0.5° в движении)
  ↓
volatile float heading
  ↓
UI thread (30 FPS):
  MainActivity.onDraw()
    ↓
  getCompassHeading() → sensorController.getHeading()
    ↓
  SLERP-интерполяция для плавности (α=0.25)
    ↓
  RadarRenderer.draw(canvas, ..., headingDisplay, ...)
    ↓
  canvas.rotate(-heading, cx, cy)
```

### 2.3. Анализ HeadingFilter.java (построчно)

```java
// HeadingFilter.java:18-25
private static final float MAX_DELTA_DEG_PER_SEC = 70f;
private static final int MEDIAN_WINDOW = 5;
private final double[] medianBuf = new double[MEDIAN_WINDOW];
private int medianCount;
```

**Проблема:** `MAX_DELTA_DEG_PER_SEC = 70°/с` — слишком агрессивно. Параплан может поворачивать со скоростью до 30-40°/с в крутке. Если сенсор даёт выброс 80°/с (магнитная аномалия), clamp его обрежет до 70°/с, но это всё равно резкий скачок.

**Рекомендация:** `MAX_DELTA_DEG_PER_SEC = 40f` (реалистичный максимум для параплана).

```java
// HeadingFilter.java:28-33
private final double alpha; // вес позиции (0.15 — статика)
private final double beta; // вес скорости (0.02)
private static final double NOMINAL_DT = 1.0 / 50.0; // 50 Гц — ожидаемая частота
private double xk; // filtered heading (в unwrapped space)
private double vk; // turn rate (°/с)
```

**Проблема:** `NOMINAL_DT = 1.0 / 50.0` предполагает 50 Гц, но `TYPE_ROTATION_VECTOR` зарегистрирован на 25 Гц (`COMPASS_DELAY_US = 40000`). Это означает, что β будет работать неправильно — он делит innovation на `NOMINAL_DT`, но реальный dt = 40 мс.

**Критическая ошибка в формуле:**

```java
// HeadingFilter.java:108-109
xk = xkPred + alpha * innovation;
vk = vkPred + beta * innovation / NOMINAL_DT;
```

Здесь `beta * innovation / NOMINAL_DT` — это попытка нормализовать β по частоте. Но правильная формула α-β фильтра:

```
xk = xkPred + alpha * innovation
vk = vkPred + beta * innovation / dt
```

где `dt` — **реальный** интервал времени, а не номинальный. Если `dt` варьируется (а он варьируется на Android), то деление на `NOMINAL_DT` даёт неправильную оценку скорости.

**Решение:**

```java
// Правильная реализация
double dt = dtMs / 1000.0;
xk = xkPred + alpha * innovation;
vk = vkPred + beta * innovation / dt; // используем реальный dt
```

Но это создаёт другую проблему: при коротких `dt` (например, 10 мс) `vk` будет взрываться. Поэтому нужно ограничить `vk`:

```java
vk = vkPred + beta * innovation / dt;
vk = Math.max(-50.0, Math.min(50.0, vk)); // clamp turn rate
```

```java
// HeadingFilter.java:36-40
private static final double DEADBAND_STATIC_DEG = 2.5;
private static final double DEADBAND_MOVING_DEG = 0.5;
private static final double TURN_RATE_THRESHOLD = 1.0;
private double lastOutputDeg;
```

**Проблема:** `DEADBAND_STATIC_DEG = 2.5°` — это слишком много. Если пилот хочет точно центрировать термик, ему нужна точность ±1°. Deadband 2.5° означает, что компас может показывать 358° вместо 0° (или наоборот), и пользователь этого не увидит.

**Рекомендация:** `DEADBAND_STATIC_DEG = 1.0`, `DEADBAND_MOVING_DEG = 0.3`.

```java
// HeadingFilter.java:115-125
double displayDiff = Math.abs(outputDeg - lastOutputDeg);
if (displayDiff > 180) displayDiff = 360 - displayDiff;

double deadband = (Math.abs(vk) < TURN_RATE_THRESHOLD)
    ? DEADBAND_STATIC_DEG
    : DEADBAND_MOVING_DEG;

if (displayDiff < deadband) {
    // Подтягиваем состояние фильтра к видимому — не накапливаем отставание
    xk = lastOutputDeg;
    return lastOutputDeg;
}
```

**Критическая ошибка:** Когда `displayDiff < deadband`, код делает `xk = lastOutputDeg`. Это означает, что внутреннее состояние фильтра принудительно подтягивается к последнему выведенному значению. Но `xk` — это filtered heading в unwrapped space, а `lastOutputDeg` — в wrapped space (0-360). Если `xk = 359.5°`, а `lastOutputDeg = 0.5°` (из-за wrap), то после `xk = lastOutputDeg` фильтр теряет накопленную историю и начинает работать с нуля.

**Решение:** Не трогать `xk`, просто не обновлять `lastOutputDeg`:

```java
if (displayDiff < deadband) {
    // Не обновляем lastOutputDeg, но и не трогаем xk
    // Фильтр продолжает накапливать историю
    return lastOutputDeg;
}
lastOutputDeg = outputDeg;
return outputDeg;
```

### 2.4. Как правильно реализовать компас (Best Practices)

На основе анализа популярных компас-приложений и документации Android, вот правильная архитектура:

#### 2.4.1. Выбор сенсора

**Правильный выбор:** `TYPE_ROTATION_VECTOR` — это fusion gyro+accel+mag, сглаженный ядром Android. Это **де-факто стандарт** для компасов с 2014 года .

```java
// Правильная регистрация
Sensor rotationVector = sensorManager.getDefaultSensor(Sensor.TYPE_ROTATION_VECTOR);
if (rotationVector != null) {
    // 25 Гц достаточно для параплана (не 50 Гц)
    sensorManager.registerListener(this, rotationVector, 40000); // 25 Hz
} else {
    // Fallback: TYPE_ACCELEROMETER + TYPE_MAGNETIC_FIELD
    Sensor accel = sensorManager.getDefaultSensor(Sensor.TYPE_ACCELEROMETER);
    Sensor mag = sensorManager.getDefaultSensor(Sensor.TYPE_MAGNETIC_FIELD);
    sensorManager.registerListener(this, accel, 40000);
    sensorManager.registerListener(this, mag, 40000);
}
```

**Почему не `TYPE_MAGNETIC_FIELD` напрямую?** Потому что он не компенсирует magnetic inclination и даёт "jumpy compass needle" когда телефон держится вертикально .

#### 2.4.2. Фильтрация heading

**Правильный пайплайн:**

```
Raw heading (0-360)
  ↓
Unwrap (монотонная шкала без скачков 359→0)
  ↓
Low-pass filter (α=0.1-0.2 для статики, α=0.5-0.8 для динамики)
  ↓
Deadband (1-2° в статике, 0.3-0.5° в движении)
  ↓
Wrap (0-360)
  ↓
UI smoothing (SLERP с α=0.2-0.3 для плавности)
```

**Пример правильной реализации:**

```java
public class CompassFilter {
    private static final double ALPHA_STATIC = 0.15;
    private static final double ALPHA_MOVING = 0.6;
    private static final double DEADBAND_STATIC = 1.0;
    private static final double DEADBAND_MOVING = 0.3;
    private static final double TURN_RATE_THRESHOLD = 2.0; // °/s
    
    private double filteredHeading;
    private double lastOutput;
    private double turnRate;
    private long lastTimeMs;
    
    public double update(double rawHeading, long timeMs) {
        if (lastTimeMs == 0) {
            filteredHeading = rawHeading;
            lastOutput = rawHeading;
            lastTimeMs = timeMs;
            return rawHeading;
        }
        
        double dt = (timeMs - lastTimeMs) / 1000.0;
        lastTimeMs = timeMs;
        
        // Unwrap
        double diff = rawHeading - filteredHeading;
        while (diff > 180) diff -= 360;
        while (diff < -180) diff += 360;
        
        // Адаптивный α в зависимости от turn rate
        turnRate = Math.abs(diff / dt);
        double alpha = (turnRate < TURN_RATE_THRESHOLD) 
            ? ALPHA_STATIC 
            : ALPHA_MOVING;
        
        // Low-pass filter
        filteredHeading += alpha * diff;
        
        // Wrap
        double output = (filteredHeading % 360 + 360) % 360;
        
        // Deadband
        double displayDiff = Math.abs(output - lastOutput);
        if (displayDiff > 180) displayDiff = 360 - displayDiff;
        
        double deadband = (turnRate < TURN_RATE_THRESHOLD)
            ? DEADBAND_STATIC
            : DEADBAND_MOVING;
        
        if (displayDiff < deadband) {
            return lastOutput; // не обновляем UI
        }
        
        lastOutput = output;
        return output;
    }
}
```

#### 2.4.3. UI smoothing (SLERP для углов)

В `MainActivity.java` используется SLERP-интерполяция для плавности отображения:

```java
// MainActivity.java — UI smoothing
float headingDisplay = sensorController.getHeading();
float diff = ((headingDisplay - headingDisplaySmoothed + 540f) % 360f) - 180f;
headingDisplaySmoothed = (headingDisplaySmoothed + diff * 0.25f + 360f) % 360f;
```

**Проблема:** `0.25f` — это фиксированный коэффициент, не зависящий от frame rate. Если FPS падает с 30 до 15, сглаживание становится в 2 раза сильнее.

**Решение:** Использовать time-based interpolation:

```java
long frameTimeMs = SystemClock.elapsedRealtime();
double dtSec = (frameTimeMs - lastFrameTimeMs) / 1000.0;
lastFrameTimeMs = frameTimeMs;

// Time-based smoothing: за 100 мс проходим 90% расстояния
double smoothingFactor = 1.0 - Math.pow(0.1, dtSec / 0.1);
float diff = ((headingDisplay - headingDisplaySmoothed + 540f) % 360f) - 180f;
headingDisplaySmoothed = (float)((headingDisplaySmoothed + diff * smoothingFactor + 360f) % 360f);
```

#### 2.4.4. Magnetometer calibration

**Критическое упущение:** В коде отсутствует калибровка магнитометра (hard-iron и soft-iron). На столе у телефона магнитный фон искажён железом стола/ноутбука, что добавляет ±5-10° ошибки.

**Решение:** Добавить калибровку:

```java
public class MagnetometerCalibrator {
    private double[] hardIronOffset = new double[3];
    private double[][] softIronMatrix = new double[3][3];
    
    // Hard-iron: среднее значение при вращении телефона
    public void calibrateHardIron(List<double[]> samples) {
        double sumX = 0, sumY = 0, sumZ = 0;
        for (double[] s : samples) {
            sumX += s[0];
            sumY += s[1];
            sumZ += s[2];
        }
        hardIronOffset[0] = sumX / samples.size();
        hardIronOffset[1] = sumY / samples.size();
        hardIronOffset[2] = sumZ / samples.size();
    }
    
    public double[] applyCalibration(double[] raw) {
        return new double[]{
            raw[0] - hardIronOffset[0],
            raw[1] - hardIronOffset[1],
            raw[2] - hardIronOffset[2]
        };
    }
}
```

Пользователь должен вращать телефон в воздухе (figure-8 pattern) для калибровки.

## 3. Потокобезопасность (Concurrency)

### 3.1. Race condition в rotationMatrix

```java
// SensorController.java — текущая реализация
private final float[] rotationMatrixInternal = new float[9];
private volatile float[] rotationMatrixSnapshot = new float[9];

private void publishRotationMatrixSnapshot() {
    synchronized (rotMatrixLock) {
        float[] snapshot = new float[9];
        System.arraycopy(rotationMatrixInternal, 0, snapshot, 0, 9);
        rotationMatrixSnapshot = snapshot;
    }
}

public float[] getRotationMatrix() { 
    return rotationMatrixSnapshot; 
}
```

**Проблема:** `rotationMatrixSnapshot` — это `volatile float[]`, но `volatile` гарантирует только атомарность ссылки, а не содержимого массива. Если UI thread читает `rotationMatrixSnapshot[0]`, а sensor thread в этот момент делает `System.arraycopy`, UI может прочитать наполовину обновлённый массив.

**Решение:** Возвращать копию:

```java
public float[] getRotationMatrix() {
    float[] snapshot = rotationMatrixSnapshot;
    float[] copy = new float[9];
    System.arraycopy(snapshot, 0, copy, 0, 9);
    return copy;
}
```

Или использовать `AtomicReference<float[]>`:

```java
private final AtomicReference<float[]> rotationMatrixRef = 
    new AtomicReference<>(new float[9]);

private void publishRotationMatrixSnapshot() {
    float[] snapshot = new float[9];
    System.arraycopy(rotationMatrixInternal, 0, snapshot, 0, 9);
    rotationMatrixRef.set(snapshot);
}

public float[] getRotationMatrix() {
    return rotationMatrixRef.get(); // атомарно
}
```

### 3.2. Race condition в HeadingFilter

```java
// SensorController.java
private final HeadingFilter headingFilter = new HeadingFilter();
private final Object headingLock = new Object();

private void updateFilteredHeading(float newHeading) {
    synchronized (headingLock) {
        heading = (float) headingFilter.update(newHeading, SystemClock.elapsedRealtime());
    }
}

public void updateGpsHeading(float gpsHeading) {
    synchronized (headingLock) {
        heading = (float) headingFilter.update(gpsHeading, SystemClock.elapsedRealtime());
    }
}
```

**Проблема:** `headingLock` защищает `headingFilter`, но `heading` — это `volatile float`, который читается из UI thread без блокировки. Если sensor thread пишет `heading`, а UI thread читает в этот момент, UI может прочитать старое значение.

**Решение:** Использовать `AtomicReference<Float>` или `AtomicInteger` (для битового представления float):

```java
private final AtomicInteger headingBits = new AtomicInteger();

private void updateFilteredHeading(float newHeading) {
    synchronized (headingLock) {
        float filtered = (float) headingFilter.update(newHeading, SystemClock.elapsedRealtime());
        headingBits.set(Float.floatToIntBits(filtered));
    }
}

public float getHeading() {
    return Float.intBitsToFloat(headingBits.get());
}
```

### 3.3. Карта потоков

| Поток | Что делает | Проблема |
|-------|-----------|----------|
| Main (UI) | `onDraw`, `onResume/onPause`, handlers | Читает `heading`, `rotationMatrix` без блокировки |
| Sensor thread | `onSensorChanged` → `SensorController` | Пишет `heading`, `rotationMatrix` |
| FusedLocationProvider | `onLocationResult` → `GpsManager` | Пишет `gpsLat`, `gpsLon` (volatile) |
| AudioTrack thread | `VarioSoundManager` playback | Читает `currentVario` (volatile) |
| TTS thread | `TextToSpeech` callbacks | Читает `ttsReady` (volatile) |

**Рекомендация:** Выделить весь sensor-fusion в `HandlerThread` с одной очередью сообщений:

```java
private final HandlerThread fusionThread = new HandlerThread("sensor-fusion");
private Handler fusionHandler;

@Override
public void onCreate() {
    fusionThread.start();
    fusionHandler = new Handler(fusionThread.getLooper());
    
    // Все обновления heading — через fusionHandler.post(...)
    sensorManager.registerListener(new SensorEventListener() {
        @Override
        public void onSensorChanged(SensorEvent event) {
            fusionHandler.post(() -> processSensorEvent(event));
        }
    }, rotationVector, 40000);
}
```

## 5. Алгоритмы и бизнес-логика

### 5.1. ThermalDetector — верификация blip

```java
// ThermalDetector.java:150-180
if (confirmCount >= confirmTarget && !blipConfirmed) {
    float avgFirst = (confirmHalfTarget > 0) ? confirmSumFirst / confirmHalfTarget : 0f;
    float avgSecond = (confirmTarget - confirmHalfTarget > 0)
        ? confirmSumSecond / (confirmTarget - confirmHalfTarget) : 0f;
    float ratio = (avgFirst > AVG_FIRST_MIN) ? avgSecond / avgFirst : 1f;
    
    if (ratio >= CONFIRM_GROW_RATIO) {
        blipConfirmed = true;
    } else if (ratio <= CONFIRM_SHRINK_RATIO) {
        consecutiveRejections++;
        if (consecutiveRejections >= MAX_REJECTIONS) {
            pilotAdvice = "ЛЕТАЙ ЗМЕЙКОЙ";
        }
    }
}
```

**Проблема:** `AVG_FIRST_MIN = 0.0001f` — это слишком маленький порог. Если `avgFirst = 0.00005`, то `ratio = avgSecond / avgFirst` может дать огромное значение (например, 1000), даже если `avgSecond` тоже маленький.

**Решение:** Использовать абсолютные пороги:

```java
if (avgFirst < 0.01f) {
    // Сигнал слишком слабый, не подтверждаем
    confirmCount = 0;
    return;
}

float ratio = avgSecond / avgFirst;
if (ratio >= CONFIRM_GROW_RATIO && avgSecond > 0.02f) {
    blipConfirmed = true;
}
```

### 5.2. CirclingManager — state machine

```java
// CirclingManager.java:180-220
switch (state) {
    case CRUISE:
        if (turning) {
            state = CirclingState.SUSPECT_CLIMB;
            stateEnterMs = nowMs;
        }
        break;
    
    case SUSPECT_CLIMB:
        if (!turning) {
            state = CirclingState.CRUISE;
        } else if (nowMs - stateEnterMs >= CRUISE_TO_CLIMB_MS) {
            state = CirclingState.CLIMB;
        }
        break;
    // ...
}
```

**Проблема:** `CRUISE_TO_CLIMB_MS = 4000` (4 секунды) — это слишком долго. Если пилот входит в термик и начинает крутить, он ожидает подтверждения через 1-2 секунды, а не 4.

**Рекомендация:** `CRUISE_TO_CLIMB_MS = 2000` (2 секунды).

### 5.3. WindEKF — не EKF

```java
// WindEKF.java — текущая реализация
public class WindEKF {
    private double windU, windV; // компоненты ветра
    
    public void update(double airspeed, double gpsVx, double gpsVy) {
        // Простое EMA, не Kalman filter
        double alpha = 0.1;
        windU += alpha * (gpsVx - airspeed * sin(heading) - windU);
        windV += alpha * (gpsVy - airspeed * cos(heading) - windV);
    }
}
```

**Проблема:** Это **не Extended Kalman Filter**, а просто Exponential Moving Average. Настоящий EKF требует:
- State vector: `[windU, windV, scaleFactor]`
- Covariance matrix `P`
- Prediction step: `P = F * P * F' + Q`
- Update step: `K = P * H' * (H * P * H' + R)^-1`

**Решение:** Переименовать в `WindEstimator` или реализовать настоящий EKF:

```java
public class WindKalmanFilter {
    private double[] x = new double[3]; // state: [windU, windV, scale]
    private double[][] P = new double[3][3]; // covariance
    private double[][] Q = new double[3][3]; // process noise
    private double[][] R = new double[2][2]; // measurement noise
    
    public void predict(double dt) {
        // x_pred = F * x
        // P_pred = F * P * F' + Q
    }
    
    public void update(double gpsVx, double gpsVy, double heading, double airspeed) {
        // y = [gpsVx - airspeed*sin(heading), gpsVy - airspeed*cos(heading)]
        // H = [[1, 0, 0], [0, 1, 0]]
        // K = P * H' * (H * P * H' + R)^-1
        // x = x_pred + K * (y - H * x_pred)
        // P = (I - K * H) * P_pred
    }
}
```

## 6. Android Lifecycle и утечки памяти

### 6.1. Handler.postDelayed утечки

```java
// MainActivity.java:354
Handler handler = new Handler();
handler.postDelayed(() -> {
    sensorController.startTiltCalibration();
}, 1000);
```

**Проблема:** Если Activity уничтожится за 1 секунду, `Handler` удержит `Runnable`, который удерживает `MainActivity` → утечка памяти.

**Решение:** Использовать `view.postDelayed` (привязка к View) или хранить `Runnable` в поле:

```java
private Runnable calibrationRunnable;

@Override
protected void onCreate(Bundle savedInstanceState) {
    calibrationRunnable = () -> sensorController.startTiltCalibration();
    findViewById(R.id.root).postDelayed(calibrationRunnable, 1000);
}

@Override
protected void onDestroy() {
    findViewById(R.id.root).removeCallbacks(calibrationRunnable);
    super.onDestroy();
}
```

### 6.2. Foreground service lifecycle

```java
// MainActivity.java:374
startForegroundService(new Intent(this, ThermalRadarService.class));
```

**Проблема:** `startForegroundService` требует, чтобы сервис в течение 5 секунд вызвал `startForeground(...)`. Если сервис не успеет (например, из-за тяжёлой инициализации), Android кидает `ForegroundServiceDidNotStartInTimeException` (CRASH на Android 12+).

**Решение:** Вызывать `startForeground` в `onCreate` сервиса:

```java
public class ThermalRadarService extends Service {
    @Override
    public void onCreate() {
        super.onCreate();
        // Немедленно показать notification
        startForeground(NOTIFICATION_ID, buildNotification());
    }
    
    @Override
    public int onStartCommand(Intent intent, int flags, int startId) {
        // Инициализация...
        return START_STICKY;
    }
}
```

public class AppContainer {
    public final SensorHub sensorHub;
    public final FlightHub flightHub;
    public final UiController uiController;
    public final LogHub logHub;
    
    public AppContainer(Context context) {
        sensorHub = new SensorHub(context);
        flightHub = new FlightHub(sensorHub);
        uiController = new UiController();
        logHub = new LogHub(context, sensorHub, flightHub);
    }
}
```

        // Прямой доступ к private полям MainActivity
        float heading = sensorController.getHeading();
        List<ThermalBlip> thermals = MainActivity.this.thermals;
        // ...
    }
}
```

**Проблема:** Tight coupling, невозможно переиспользовать `RadarView` в другом Activity, невозможно тестировать.

**Решение:** Вынести в отдельный файл, передавать данные через публичный API:

```java
public class RadarView extends View {

                           float vario, String status) {
        renderer.setHeading(heading);
        renderer.setThermals(thermals);
        invalidate();
    }
    
    @Override
    protected void onDraw(Canvas canvas) {
        renderer.draw(canvas, ...);
    }
}
```

## 2. Компас: главная проблема (Deep Dive)

### 2.1. Симптом

> "Прибор лежит на столе. Стрелка компаса дрожит в пределах 2-3 градусов. При резком повороте — компас плавно поворачивается."

Этот симптом — **классическая карта отказов α-β фильтра с завышенным α и слишком малым deadband**.

### 2.2. Цепочка обработки heading (v0.2.13)

```
TYPE_ROTATION_VECTOR (sensor thread, 25 Hz)
  ↓
SensorController.onSensorChanged(ROTATION_VECTOR)
  ↓
SensorManager.getRotationMatrixFromVector(rotationMatrixInternal, event.values)
  ↓
SensorManager.getOrientation(rotationMatrixInternal, orientationVals)
  ↓
rawHeading = Math.toDegrees(orientationVals[0])
  ↓
HeadingFilter.update(rawHeading, elapsedRealtime)
  ↓
  ├─ CLAMP (|Δheading| > 70°/с → limit)
  ├─ MEDIAN FILTER (окно 5)
  ├─ Alpha-Beta filter (α=0.15, β=0.02) с unwrap
  └─ DEADBAND (адаптивный: 2.5° в статике, 0.5° в движении)
  ↓
volatile float heading
  ↓
UI thread (30 FPS):
  MainActivity.onDraw()
    ↓
  getCompassHeading() → sensorController.getHeading()
    ↓
  SLERP-интерполяция для плавности (α=0.25)
    ↓
  RadarRenderer.draw(canvas, ..., headingDisplay, ...)
    ↓
  canvas.rotate(-heading, cx, cy)
```

### 2.3. Анализ HeadingFilter.java (построчно)

```java
// HeadingFilter.java:18-25
private static final float MAX_DELTA_DEG_PER_SEC = 70f;
private static final int MEDIAN_WINDOW = 5;
private final double[] medianBuf = new double[MEDIAN_WINDOW];
private int medianCount;
```

**Проблема:** `MAX_DELTA_DEG_PER_SEC = 70°/с` — слишком агрессивно. Параплан может поворачивать со скоростью до 30-40°/с в крутке. Если сенсор даёт выброс 80°/с (магнитная аномалия), clamp его обрежет до 70°/с, но это всё равно резкий скачок.

**Рекомендация:** `MAX_DELTA_DEG_PER_SEC = 40f` (реалистичный максимум для параплана).

```java
// HeadingFilter.java:28-33
private final double alpha; // вес позиции (0.15 — статика)
private final double beta; // вес скорости (0.02)
private static final double NOMINAL_DT = 1.0 / 50.0; // 50 Гц — ожидаемая частота
private double xk; // filtered heading (в unwrapped space)
private double vk; // turn rate (°/с)
```

**Проблема:** `NOMINAL_DT = 1.0 / 50.0` предполагает 50 Гц, но `TYPE_ROTATION_VECTOR` зарегистрирован на 25 Гц (`COMPASS_DELAY_US = 40000`). Это означает, что β будет работать неправильно — он делит innovation на `NOMINAL_DT`, но реальный dt = 40 мс.

**Критическая ошибка в формуле:**

```java
// HeadingFilter.java:108-109
xk = xkPred + alpha * innovation;
vk = vkPred + beta * innovation / NOMINAL_DT;
```

Здесь `beta * innovation / NOMINAL_DT` — это попытка нормализовать β по частоте. Но правильная формула α-β фильтра:

```
xk = xkPred + alpha * innovation
vk = vkPred + beta * innovation / dt
```

где `dt` — **реальный** интервал времени, а не номинальный. Если `dt` варьируется (а он варьируется на Android), то деление на `NOMINAL_DT` даёт неправильную оценку скорости.

**Решение:**

```java
// Правильная реализация
double dt = dtMs / 1000.0;
xk = xkPred + alpha * innovation;
vk = vkPred + beta * innovation / dt; // используем реальный dt
```

Но это создаёт другую проблему: при коротких `dt` (например, 10 мс) `vk` будет взрываться. Поэтому нужно ограничить `vk`:

```java
vk = vkPred + beta * innovation / dt;
vk = Math.max(-50.0, Math.min(50.0, vk)); // clamp turn rate
```

```java
// HeadingFilter.java:36-40
private static final double DEADBAND_STATIC_DEG = 2.5;
private static final double DEADBAND_MOVING_DEG = 0.5;
private static final double TURN_RATE_THRESHOLD = 1.0;
private double lastOutputDeg;
```

**Проблема:** `DEADBAND_STATIC_DEG = 2.5°` — это слишком много. Если пилот хочет точно центрировать термик, ему нужна точность ±1°. Deadband 2.5° означает, что компас может показывать 358° вместо 0° (или наоборот), и пользователь этого не увидит.

**Рекомендация:** `DEADBAND_STATIC_DEG = 1.0`, `DEADBAND_MOVING_DEG = 0.3`.

```java
// HeadingFilter.java:115-125
double displayDiff = Math.abs(outputDeg - lastOutputDeg);
if (displayDiff > 180) displayDiff = 360 - displayDiff;

double deadband = (Math.abs(vk) < TURN_RATE_THRESHOLD)
    ? DEADBAND_STATIC_DEG
    : DEADBAND_MOVING_DEG;

if (displayDiff < deadband) {
    // Подтягиваем состояние фильтра к видимому — не накапливаем отставание
    xk = lastOutputDeg;
    return lastOutputDeg;
}
```

**Критическая ошибка:** Когда `displayDiff < deadband`, код делает `xk = lastOutputDeg`. Это означает, что внутреннее состояние фильтра принудительно подтягивается к последнему выведенному значению. Но `xk` — это filtered heading в unwrapped space, а `lastOutputDeg` — в wrapped space (0-360). Если `xk = 359.5°`, а `lastOutputDeg = 0.5°` (из-за wrap), то после `xk = lastOutputDeg` фильтр теряет накопленную историю и начинает работать с нуля.

**Решение:** Не трогать `xk`, просто не обновлять `lastOutputDeg`:

```java
if (displayDiff < deadband) {
    // Не обновляем lastOutputDeg, но и не трогаем xk
    // Фильтр продолжает накапливать историю
    return lastOutputDeg;
}
lastOutputDeg = outputDeg;
return outputDeg;
```

### 2.4. Как правильно реализовать компас (Best Practices)

На основе анализа популярных компас-приложений и документации Android, вот правильная архитектура:

#### 2.4.1. Выбор сенсора

**Правильный выбор:** `TYPE_ROTATION_VECTOR` — это fusion gyro+accel+mag, сглаженный ядром Android. Это **де-факто стандарт** для компасов с 2014 года.

```java
// Правильная регистрация
Sensor rotationVector = sensorManager.getDefaultSensor(Sensor.TYPE_ROTATION_VECTOR);
if (rotationVector != null) {
    // 25 Гц достаточно для параплана (не 50 Гц)
    sensorManager.registerListener(this, rotationVector, 40000); // 25 Hz
} else {
    // Fallback: TYPE_ACCELEROMETER + TYPE_MAGNETIC_FIELD
    Sensor accel = sensorManager.getDefaultSensor(Sensor.TYPE_ACCELEROMETER);
    Sensor mag = sensorManager.getDefaultSensor(Sensor.TYPE_MAGNETIC_FIELD);
    sensorManager.registerListener(this, accel, 40000);
    sensorManager.registerListener(this, mag, 40000);
}
```

**Почему не `TYPE_MAGNETIC_FIELD` напрямую?** Потому что он не компенсирует magnetic inclination и даёт "jumpy compass needle" когда телефон держится вертикально.

#### 2.4.2. Фильтрация heading

**Правильный пайплайн:**

```
Raw heading (0-360)
  ↓
Unwrap (монотонная шкала без скачков 359→0)
  ↓
Low-pass filter (α=0.1-0.2 для статики, α=0.5-0.8 для динамики)
  ↓
Deadband (1-2° в статике, 0.3-0.5° в движении)
  ↓
Wrap (0-360)
  ↓
UI smoothing (SLERP с α=0.2-0.3 для плавности)
```

**Пример правильной реализации:**

```java
public class CompassFilter {
    private static final double ALPHA_STATIC = 0.15;
    private static final double ALPHA_MOVING = 0.6;
    private static final double DEADBAND_STATIC = 1.0;
    private static final double DEADBAND_MOVING = 0.3;
    private static final double TURN_RATE_THRESHOLD = 2.0; // °/s
    
    private double filteredHeading;
    private double lastOutput;
    private double turnRate;
    private long lastTimeMs;
    
    public double update(double rawHeading, long timeMs) {
        if (lastTimeMs == 0) {
            filteredHeading = rawHeading;
            lastOutput = rawHeading;
            lastTimeMs = timeMs;
            return rawHeading;
        }
        
        double dt = (timeMs - lastTimeMs) / 1000.0;
        lastTimeMs = timeMs;
        
        // Unwrap
        double diff = rawHeading - filteredHeading;
        while (diff > 180) diff -= 360;
        while (diff < -180) diff += 360;
        
        // Адаптивный α в зависимости от turn rate
        turnRate = Math.abs(diff / dt);
        double alpha = (turnRate < TURN_RATE_THRESHOLD) 
            ? ALPHA_STATIC 
            : ALPHA_MOVING;
        
        // Low-pass filter
        filteredHeading += alpha * diff;
        
        // Wrap
        double output = (filteredHeading % 360 + 360) % 360;
        
        // Deadband
        double displayDiff = Math.abs(output - lastOutput);
        if (displayDiff > 180) displayDiff = 360 - displayDiff;
        
        double deadband = (turnRate < TURN_RATE_THRESHOLD)
            ? DEADBAND_STATIC
            : DEADBAND_MOVING;
        
        if (displayDiff < deadband) {
            return lastOutput; // не обновляем UI
        }
        
        lastOutput = output;
        return output;
    }
}
```

#### 2.4.3. UI smoothing (SLERP для углов)

В `MainActivity.java` используется SLERP-интерполяция для плавности отображения:

```java
// MainActivity.java — UI smoothing
float headingDisplay = sensorController.getHeading();
float diff = ((headingDisplay - headingDisplaySmoothed + 540f) % 360f) - 180f;
headingDisplaySmoothed = (headingDisplaySmoothed + diff * 0.25f + 360f) % 360f;
```

**Проблема:** `0.25f` — это фиксированный коэффициент, не зависящий от frame rate. Если FPS падает с 30 до 15, сглаживание становится в 2 раза сильнее.

**Решение:** Использовать time-based interpolation:

```java
long frameTimeMs = SystemClock.elapsedRealtime();
double dtSec = (frameTimeMs - lastFrameTimeMs) / 1000.0;
lastFrameTimeMs = frameTimeMs;

// Time-based smoothing: за 100 мс проходим 90% расстояния
double smoothingFactor = 1.0 - Math.pow(0.1, dtSec / 0.1);
float diff = ((headingDisplay - headingDisplaySmoothed + 540f) % 360f) - 180f;
headingDisplaySmoothed = (float)((headingDisplaySmoothed + diff * smoothingFactor + 360f) % 360f);
```

#### 2.4.4. Magnetometer calibration

**Критическое упущение:** В коде отсутствует калибровка магнитометра (hard-iron и soft-iron). На столе у телефона магнитный фон искажён железом стола/ноутбука, что добавляет ±5-10° ошибки.

**Решение:** Добавить калибровку:

```java
public class MagnetometerCalibrator {
    private double[] hardIronOffset = new double[3];
    private double[][] softIronMatrix = new double[3][3];
    
    // Hard-iron: среднее значение при вращении телефона
    public void calibrateHardIron(List<double[]> samples) {
        double sumX = 0, sumY = 0, sumZ = 0;
        for (double[] s : samples) {
            sumX += s[0];
            sumY += s[1];
            sumZ += s[2];
        }
        hardIronOffset[0] = sumX / samples.size();
        hardIronOffset[1] = sumY / samples.size();
        hardIronOffset[2] = sumZ / samples.size();
    }
    
    public double[] applyCalibration(double[] raw) {
        return new double[]{
            raw[0] - hardIronOffset[0],
            raw[1] - hardIronOffset[1],
            raw[2] - hardIronOffset[2]
        };
    }
}
```

Пользователь должен вращать телефон в воздухе (figure-8 pattern) для калибровки.

## 3. Потокобезопасность (Concurrency)

### 3.1. Race condition в rotationMatrix

```java
// SensorController.java — текущая реализация
private final float[] rotationMatrixInternal = new float[9];
private volatile float[] rotationMatrixSnapshot = new float[9];

private void publishRotationMatrixSnapshot() {
    synchronized (rotMatrixLock) {
        float[] snapshot = new float[9];
        System.arraycopy(rotationMatrixInternal, 0, snapshot, 0, 9);
        rotationMatrixSnapshot = snapshot;
    }
}

public float[] getRotationMatrix() { 
    return rotationMatrixSnapshot; 
}
```

**Проблема:** `rotationMatrixSnapshot` — это `volatile float[]`, но `volatile` гарантирует только атомарность ссылки, а не содержимого массива. Если UI thread читает `rotationMatrixSnapshot[0]`, а sensor thread в этот момент делает `System.arraycopy`, UI может прочитать наполовину обновлённый массив.

**Решение:** Возвращать копию:

```java
public float[] getRotationMatrix() {
    float[] snapshot = rotationMatrixSnapshot;
    float[] copy = new float[9];
    System.arraycopy(snapshot, 0, copy, 0, 9);
    return copy;
}
```

Или использовать `AtomicReference<float[]>`:

```java
private final AtomicReference<float[]> rotationMatrixRef = 
    new AtomicReference<>(new float[9]);

private void publishRotationMatrixSnapshot() {
    float[] snapshot = new float[9];
    System.arraycopy(rotationMatrixInternal, 0, snapshot, 0, 9);
    rotationMatrixRef.set(snapshot);
}

public float[] getRotationMatrix() {
    return rotationMatrixRef.get(); // атомарно
}
```

### 3.2. Race condition в HeadingFilter

```java
// SensorController.java
private final HeadingFilter headingFilter = new HeadingFilter();
private final Object headingLock = new Object();

private void updateFilteredHeading(float newHeading) {
    synchronized (headingLock) {
        heading = (float) headingFilter.update(newHeading, SystemClock.elapsedRealtime());
    }
}

public void updateGpsHeading(float gpsHeading) {
    synchronized (headingLock) {
        heading = (float) headingFilter.update(gpsHeading, SystemClock.elapsedRealtime());
    }
}
```

**Проблема:** `headingLock` защищает `headingFilter`, но `heading` — это `volatile float`, который читается из UI thread без блокировки. Если sensor thread пишет `heading`, а UI thread читает в этот момент, UI может прочитать старое значение.

**Решение:** Использовать `AtomicReference<Float>` или `AtomicInteger` (для битового представления float):

```java
private final AtomicInteger headingBits = new AtomicInteger();

private void updateFilteredHeading(float newHeading) {
    synchronized (headingLock) {
        float filtered = (float) headingFilter.update(newHeading, SystemClock.elapsedRealtime());
        headingBits.set(Float.floatToIntBits(filtered));
    }
}

public float getHeading() {
    return Float.intBitsToFloat(headingBits.get());
}
```

### 3.3. Карта потоков

| Поток | Что делает | Проблема |
|-------|-----------|----------|
| Main (UI) | `onDraw`, `onResume/onPause`, handlers | Читает `heading`, `rotationMatrix` без блокировки |
| Sensor thread | `onSensorChanged` → `SensorController` | Пишет `heading`, `rotationMatrix` |
| FusedLocationProvider | `onLocationResult` → `GpsManager` | Пишет `gpsLat`, `gpsLon` (volatile) |
| AudioTrack thread | `VarioSoundManager` playback | Читает `currentVario` (volatile) |
| TTS thread | `TextToSpeech` callbacks | Читает `ttsReady` (volatile) |

**Рекомендация:** Выделить весь sensor-fusion в `HandlerThread` с одной очередью сообщений:

```java
private final HandlerThread fusionThread = new HandlerThread("sensor-fusion");
private Handler fusionHandler;

@Override
public void onCreate() {
    fusionThread.start();
    fusionHandler = new Handler(fusionThread.getLooper());
    
    // Все обновления heading — через fusionHandler.post(...)
    sensorManager.registerListener(new SensorEventListener() {
        @Override
        public void onSensorChanged(SensorEvent event) {
            fusionHandler.post(() -> processSensorEvent(event));
        }
    }, rotationVector, 40000);
}
```

## 5. Алгоритмы и бизнес-логика

### 5.1. ThermalDetector — верификация blip

```java
// ThermalDetector.java:150-180
if (confirmCount >= confirmTarget && !blipConfirmed) {
    float avgFirst = (confirmHalfTarget > 0) ? confirmSumFirst / confirmHalfTarget : 0f;
    float avgSecond = (confirmTarget - confirmHalfTarget > 0)
        ? confirmSumSecond / (confirmTarget - confirmHalfTarget) : 0f;
    float ratio = (avgFirst > AVG_FIRST_MIN) ? avgSecond / avgFirst : 1f;
    
    if (ratio >= CONFIRM_GROW_RATIO) {
        blipConfirmed = true;
    } else if (ratio <= CONFIRM_SHRINK_RATIO) {
        consecutiveRejections++;
        if (consecutiveRejections >= MAX_REJECTIONS) {
            pilotAdvice = "ЛЕТАЙ ЗМЕЙКОЙ";
        }
    }
}
```

**Проблема:** `AVG_FIRST_MIN = 0.0001f` — это слишком маленький порог. Если `avgFirst = 0.00005`, то `ratio = avgSecond / avgFirst` может дать огромное значение (например, 1000), даже если `avgSecond` тоже маленький.

**Решение:** Использовать абсолютные пороги:

```java
if (avgFirst < 0.01f) {
    // Сигнал слишком слабый, не подтверждаем
    confirmCount = 0;
    return;
}

float ratio = avgSecond / avgFirst;

    blipConfirmed = true;
}
```

### 5.2. CirclingManager — state machine

```java
// CirclingManager.java:180-220
switch (state) {
    case CRUISE:
        if (turning) {
            state = CirclingState.SUSPECT_CLIMB;
            stateEnterMs = nowMs;
        }
        break;
    
    case SUSPECT_CLIMB:
        if (!turning) {
            state = CirclingState.CRUISE;
        } else if (nowMs - stateEnterMs >= CRUISE_TO_CLIMB_MS) {
            state = CirclingState.CLIMB;
        }
        break;
    // ...
}
```

**Проблема:** `CRUISE_TO_CLIMB_MS = 4000` (4 секунды) — это слишком долго. Если пилот входит в термик и начинает крутить, он ожидает подтверждения через 1-2 секунды, а не 4.

**Рекомендация:** `CRUISE_TO_CLIMB_MS = 2000` (2 секунды).

### 5.3. WindEKF — не EKF

```java
// WindEKF.java — текущая реализация
public class WindEKF {
    private double windU, windV; // компоненты ветра
    
    public void update(double airspeed, double gpsVx, double gpsVy) {
        // Простое EMA, не Kalman filter
        double alpha = 0.1;
        windU += alpha * (gpsVx - airspeed * sin(heading) - windU);
        windV += alpha * (gpsVy - airspeed * cos(heading) - windV);
    }
}
```

**Проблема:** Это **не Extended Kalman Filter**, а просто Exponential Moving Average. Настоящий EKF требует:
- State vector: `[windU, windV, scaleFactor]`
- Covariance matrix `P`
- Prediction step: `P = F * P * F' + Q`
- Update step: `K = P * H' * (H * P * H' + R)^-1`

**Решение:** Переименовать в `WindEstimator` или реализовать настоящий EKF:

```java
public class WindKalmanFilter {
    private double[] x = new double[3]; // state: [windU, windV, scale]
    private double[][] P = new double[3][3]; // covariance
    private double[][] Q = new double[3][3]; // process noise
    private double[][] R = new double[2][2]; // measurement noise
    
    public void predict(double dt) {
        // x_pred = F * x
        // P_pred = F * P * F' + Q
    }
    
    public void update(double gpsVx, double gpsVy, double heading, double airspeed) {
        // y = [gpsVx - airspeed*sin(heading), gpsVy - airspeed*cos(heading)]
        // H = [[1, 0, 0], [0, 1, 0]]
        // K = P * H' * (H * P * H' + R)^-1
        // x = x_pred + K * (y - H * x_pred)
        // P = (I - K * H) * P_pred
    }
}
```

## 6. Android Lifecycle и утечки памяти

### 6.1. Handler.postDelayed утечки

```java
// MainActivity.java:354
Handler handler = new Handler();
handler.postDelayed(() -> {
    sensorController.startTiltCalibration();
}, 1000);
```

**Проблема:** Если Activity уничтожится за 1 секунду, `Handler` удержит `Runnable`, который удерживает `MainActivity` → утечка памяти.

**Решение:** Использовать `view.postDelayed` (привязка к View) или хранить `Runnable` в поле:

```java
private Runnable calibrationRunnable;

@Override
protected void onCreate(Bundle savedInstanceState) {
    calibrationRunnable = () -> sensorController.startTiltCalibration();
    findViewById(R.id.root).postDelayed(calibrationRunnable, 1000);
}

@Override
protected void onDestroy() {
    findViewById(R.id.root).removeCallbacks(calibrationRunnable);
    super.onDestroy();
}
```

### 6.2. Foreground service lifecycle

```java
// MainActivity.java:374
startForegroundService(new Intent(this, ThermalRadarService.class));
```

**Проблема:** `startForegroundService` требует, чтобы сервис в течение 5 секунд вызвал `startForeground(...)`. Если сервис не успеет (например, из-за тяжёлой инициализации), Android кидает `ForegroundServiceDidNotStartInTimeException` (CRASH на Android 12+).

**Решение:** Вызывать `startForeground` в `onCreate` сервиса:

```java
public class ThermalRadarService extends Service {
    @Override
    public void onCreate() {
        super.onCreate();
        // Немедленно показать notification
        startForeground(NOTIFICATION_ID, buildNotification());
    }
    
    @Override
    public int onStartCommand(Intent intent, int flags, int startId) {
        // Инициализация...
        return START_STICKY;
    }
}
```

### 6.3. Permission flow

```java
// MainActivity.java:250-260
if (android.os.Build.VERSION.SDK_INT >= android.os.Build.VERSION_CODES.M) {
    if (checkSelfPermission(...) != GRANTED) {
        requestPermissions(...);
    }
}
```

**Проблема:** `Build.VERSION.SDK_INT >= M` (23) — `minSdk = 26`, всегда true. Дед-код.

**Решение:** Убрать проверку:

```java
if (checkSelfPermission(Manifest.permission.ACCESS_FINE_LOCATION) != GRANTED) {
    requestPermissions(new String[]{Manifest.permission.ACCESS_FINE_LOCATION}, 1001);
}
```

**Дополнительно:** `POST_NOTIFICATIONS` (Android 13+) не запрашивается runtime. На Android 13+ foreground service notification не покажется.

```java
if (Build.VERSION.SDK_INT >= 33) {
    if (checkSelfPermission(Manifest.permission.POST_NOTIFICATIONS) != GRANTED) {
        requestPermissions(new String[]{Manifest.permission.POST_NOTIFICATIONS}, 1002);
    }
}

## 7. Безопасность и разрешения

### 7.1. OSM User-Agent

```java
// StaticMapLoader.java
URL url = new URL("https://tile.openstreetmap.org/" + zoom + "/" + x + "/" + y + ".png");
HttpURLConnection conn = (HttpURLConnection) url.openConnection();
// ← отсутствует User-Agent!
```

**Проблема:** OSM Tile Usage Policy требует:

```
User-Agent: TERMO1-Radar/0.2.13 (contact@example.com)
```

Без UA OSM может забанить IP после нескольких сотен запросов.

**Решение:**

```java
HttpURLConnection conn = (HttpURLConnection) url.openConnection();
conn.setRequestProperty("User-Agent", "TERMO1-Radar/0.2.13 (https://github.com/isemaster/ThermalRadar)");
```

### 7.2. Cleartext traffic

`AndroidManifest.xml` не содержит `android:usesCleartextTraffic`. По умолчанию на `targetSdk = 34` cleartext запрещён, что правильно. Но `StaticMapLoader` должен ходить только по HTTPS — проверить.
