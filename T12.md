## 0. Исполнительное резюме

Приложение **ThermalRadar (TERMO1)** — амбициозная попытка одного разработчика-студента создать «карманный тепловизор для парапланериста»: определить направление на восходящий поток по микрораскачке крыла, вариометру, гироскопу, GPS-треку, и подсказать пилоту, в какую сторону крутить. Декларируется портирование алгоритмов из **XCSoar** (эталонного open-source варно/навигатора для планеристов).

Положительные стороны:

- Чистая layered-структура пакетов (`core/`, `flight/`, `sensors/`, `gps/`, `logging/`, `map/`, `ui/`, `model/`).
- Реализован полный pipeline: сенсоры → фильтры → детектор → состоянийный автомат → UI + TTS + звук варно.
- IGC-логгер соответствует формату FAI, есть параллельный CSV-лог 50 Гц для отладки.
- Есть симуляция, сценарий, трек-реплей — инструменты для наземной отладки.
- Используются `HandlerThread`, `WakeLock`, foreground service — стандартный набор для долговременной работы в полёте.

Критические проблемы:

1. **Жалобы пользователя на проигрывание чужих IGC-треков подтверждаются кодом.** Найдено 7 критических багов, прямо объясняющих симптомы: «теряется трек», «неполностью прорисован», «мерцает», «неверные высоты», «ошибки в показаниях».
2. **Safety-critical баги в алгоритмах ветра.** В `WindDriftCalculator.java` знаки в формулах сноса инвертированы — пилот, следующий подсказке приложения, будет лететь **в обратную сторону** от нужной для компенсации сноса. Это прямая угроза безопасности полёта.
3. **`HeadingFilter.computeMedian()` ломает временной буфер** — компас для всех устройств без `ROTATION_VECTOR` (старые/дешёвые телефоны, fallback при плохом магнетометре) после первого сэмпла работает на отсортированных данных, а не на временном ряде.
4. **Отсутствует `SensorManager.remapCoordinateSystem()`** — компас показывает произвольный курс при нестандартной ориентации крепления телефона (а парапланеристы крепят кого угодно и как угодно).
5. **Архитектурный долг:** `MainActivity.java` — 3591 строка в одном файле, смешаны UI, бизнес-логика, сенсорные колбэки, алгоритмы, gesture handling. Это немейнтейнабо.
6. **Производительность onDraw:** десятки аллокаций на кадр (`new double[]`, `new float[]`, `String.format`, `new Date()`, `new Path()`, `NotificationCompat.Builder`) → minor GC каждые 3–5 секунд → мерцание.
7. **Threading data races** в `ThermalLocator`, `CirclingManager`, `LiftDatabase`, `ThermalBlip` — чтение из UI-треда без volatile/synchronized, на ARM возможны torn reads и NaN.

Оценка полезности прибора (краткий ответ, подробности в разделе 9):

> Приложение **нельзя использовать как основной прибор в реальном полёте** до исправления safety-critical багов в `WindDriftCalculator` (T-25, T-26), `HeadingFilter` (T-64), `SensorController` (T-52), и в отображении MSL-высоты (BUG MA-1). После исправлений — это неплохой учебный/вспомогательный инструмент для парапланеристов-любителей, но не замена сертифицированным варно (XCSoar + специальный Bluetooth-варно, Flytec, Brauniger).

### 1.2. Архитектурные наблюдения

**Положительные:**

- Разделение по пакетам по domain-признаку (`flight/`, `sensors/`, `gps/`, `logging/`, `map/`, `ui/`) — читается лучше, чем типичный «студенческий» проект.
- Использование `SensorManager`, `LocationManager` (через FusedLocationProvider), foreground service, `WakeLock`, `HandlerThread` — соответствует best practices Android.
- IGC-логгер реализован по FAI-стандарту (HFAPIRECORDTYPE, HFPLTPILOT, B-record с `A/V` validity, pressure + GPS altitude).
- Есть unit-тесты (`Termo1UnitTests.java`), хотя покрытие минимальное.

**Отрицательные (архитектурный долг):**

- **MainActivity.java — 3591 строка.** Это god class. В нём: lifecycle, sensor listener, GPS callback, render loop, touch handling, dialog management, TTS, file picker, simulation, scenario, track replay. Стандартное правило «экран Activity ≤ 800 строк» нарушено в 4.5 раза.
- **Нет ViewModel / Jetpack MVVM.** Начиная с min SDK 26, можно было использовать хотя бы `ViewModel` + `LiveData` (даже без AndroidX — через `androidx.lifecycle` как dependency). Сейчас весь state в полях Activity — любой поворот или process death теряет данные.
- **Нет инъекции зависимостей.** Все модули создаются вручную в `onCreate()`. Это OK для проекта такого размера, но делает unit-тестирование невозможным (нельзя подсунуть mock GPS или mock sensors).
- **Java вместо Kotlin.** На min SDK 26 и Java 17 можно уже мигрировать на Kotlin с null-safety, data classes, coroutines. Текущий код полон `if (x == null)` и ручных defensive checks, которые Kotlin бы выловил на этапе компиляции.
- **Нет AndroidX.** `import android.app.Activity` вместо `androidx.appcompat.app.AppCompatActivity`, `import android.app.AlertDialog` вместо `androidx.appcompat.app.AlertDialog`. Это означает: нет поддержки back-port-ов, нет AppCompat-тёмной темы, нет inline-контента в dialog-ах на старых устройствах, нет `OnBackPressedDispatcher`.
- **Compile SDK 34, target SDK 34, но `GpsStatus.Listener` deprecated с API 24** — код использует устаревший API.
- **Нет ProGuard-настроек** (`proguard-rules.pro` пустой или дефолтный, а `isMinifyEnabled = true` включён — есть риск что релизная сборка что-то вырежет).

## 2. Разбор модуля `TrackReplayer.java` (парсер и проигрыватель IGC)

Файл: `app/src/main/java/com/termo1/radar/core/TrackReplayer.java`, 963 строки.

Это центральный модуль для жалобы пользователя «плохо работает проигрывание чужих IGC треков». Разбираю построчно критические участки.

### 2.1. Парсер B-record — критические баги

#### BUG TR-1 (CRITICAL): Hardcoded time filter применяется к чужим трекам

**Файл:** `TrackReplayer.java:310-336`

```java

            try {

                String t = line.substring(1, 7);
                int hh = Integer.parseInt(t.substring(0, 2));
                int mm = Integer.parseInt(t.substring(2, 4));
                int ss = Integer.parseInt(t.substring(4, 6));
                float timeSec = hh * 3600f + mm * 60f + ss;
                // FIX: 5Hz IGC — односекундные записи получают суб-секунды

                    sameSecCount++;
                    timeSec += sameSecCount * 0.2f;

                    sameSecCount = 0;

                if (applyTimeFilter) {
                    if (timeSec < 13 * 3600f + 17 * 60f) continue;          // ← 13:17 hardcoded
                    if (timeSec > 13 * 3600f + 40 * 60f) continue;          // ← 13:40 hardcoded
                }
                ...
```

**Проблема:** Параметр `applyTimeFilter` управляет применением окна 13:17–13:40. Это окно захардкожено под конкретный тестовый файл `17-18_17-30.igc` (упомянутый в javadoc класса, строка 14). Для собственного встроенного ресурса `R.raw.track_replay` это допустимо. Но `loadFile(String)` (строка 274) вызывает `loadFromIGC(..., false)`, что корректно.

Однако:
- Время в коде — device time UTC+4, как сказано в javadoc. Если чужой IGC записан в UTC (стандарт!), `13:17 UTC+4 = 09:17 UTC`. Код не делает этой конвертации для встроенного трека, но и не документирует, что `timeSec` — это UTC или локальное.
- Внешние IGC-файлы могут содержать полёты в любое время суток, в любой timezone. Жалоба «неверно показаны высоты» может косвенно связана с тем, что `vario` (вычисляется из `altDeltaPerFrame`) базируется на `timeSec`, а `timeSec` имеет baggage 5Hz-эвристики (см. следующий баг).

#### BUG TR-2 (CRITICAL): 5Hz-эвристика ломает 1Hz / 10Hz IGC

**Файл:** `TrackReplayer.java:324-332`

```java
if (timeSec == prevTimeSec) {
    sameSecCount++;
    timeSec += sameSecCount * 0.2f;          // ← всегда 0.2s!
} else {
    if (prevTimeSec > 0 && timeSec < prevTimeSec) timeSec += 86400f;
    prevTimeSec = timeSec;
    sameSecCount = 0;
}
```

**Проблема:** Код предполагает, что если несколько B-records имеют одинаковое `HHMMSS`, то это **5Hz IGC** (интервал 0.2с). Но:

1. Многие логгеры (LXNAV, Flarm, XCSoar internal) пишут **1Hz** — каждый `HHMMSS` уникален. Если случайно появилась пара одинаковых секунд (clock drift, дубликат из-за buffered write), код добавит 0.2с между ними → расстояние между точками растягивается во времени, скорость пилота занижается в 5 раз.
2. Логгеры **10Hz** (OpenVario, некоторые Flarm-версии) — код добавит 0.2с вместо реальных 0.1с → скорость завышается в 2 раза, варно-скачки в 2× чаще.
3. Логгеры **2Hz** — растянет на 0.2с вместо 0.5с → скорость завышается в 2.5 раза.

Все эти эффекты прямо вызывают «ошибки в показаниях для пилота»: скорость не соответствует реальной (по IGC), варно выглядит скачущим, интерполяция сужается/растягивается.

**Фикс:** Парсить I-records для получения fix accuracy / extension, либо измерять фактическую частоту пост-фактум:

```java
// После загрузки всех точек — оценить частоту
if (track.size() > 10) {
    float dtSum = 0;
    int dtCount = 0;
    for (int i = 1; i < Math.min(track.size(), 50); i++) {
        float dt = track.get(i).timeSec - track.get(i-1).timeSec;
        if (dt > 0 && dt < 5) { dtSum += dt; dtCount++; }
    }
    float medianDt = dtCount > 0 ? dtSum / dtCount : 1f;
    // Если медианный dt < 0.5, значит файл high-rate — оставляем как есть.
    // Если >= 0.5, то файл 1Hz или 2Hz — суб-секунды не нужны.
}
```

Или полностью отказаться от суб-секундной эвристики и округлять до 0.1с:

```java
// Округлить до 0.1с от времени первого валидного фикса
float baseTimeSec = track.isEmpty() ? 0 : track.get(0).timeSec;
float timeSec = Math.round((rawTimeSec - baseTimeSec) * 10f) / 10f + baseTimeSec;
```

#### BUG TR-3 (CRITICAL): Не проверяется `A`/`V` fix validity в B-record

**Файл:** `TrackReplayer.java:314-350`

```java
if (line.isEmpty() || line.charAt(0) != 'B') continue;
try {
    if (line.length() < 30) continue;
    String t = line.substring(1, 7);
    // ...
    // НЕТ ПРОВЕРКИ line.charAt(24) == 'A'
    // ...
    track.add(new TrackPoint(lat, lon, Math.max(alt, 0), timeSec));
}
```

**Проблема:** IGC B-record имеет формат:

```
B HHMMSS DDMMmmm N DDDMMmmm E A PPPPP GGGGG
                              ^
                              позиция 24: A = fix valid, V = no fix
```

Код не проверяет `fixValid`. Невалидные фиксы (когда GPS у пилота пропал — часто встречается в горах, под облаками, при сильном крене) добавляются в трек как валидные точки. Это даёт «прыжки» трека, мусорные координаты (0,0), пропуски. Внешние IGC часто имеют V-фиксы в начале (до взлёта) и в конце (после посадки).

Это прямо объясняет жалобу «траектория теряется»: если 30% точек невалидны и содержат (0,0), трек выглядит как паутина от САО (0,0 — точка в Гвинейском заливе).

**Фикс:**

```java
char fixValid = line.length() > 24 ? line.charAt(24) : 'V';
if (fixValid != 'A') continue;   // пропускаем невалидные фиксы

// Дополнительно — проверка на (0,0):
double lat = latDeg + latMin / 60.0;
if (line.charAt(14) == 'S') lat = -lat;
double lon = lonDeg + lonMin / 60.0;
if (line.charAt(23) == 'W') lon = -lon;
if (lat == 0.0 && lon == 0.0) continue; // null island
```

#### BUG TR-4 (CRITICAL): Pressure altitude = 0 → fallback на GPS altitude

**Файл:** `TrackReplayer.java:345-350`

```java
int alt = 0;
try { alt = Integer.parseInt(line.substring(25, 30).trim()); } catch (Exception ignored) {}
if (alt == 0 && line.length() >= 35) {
    try { alt = Integer.parseInt(line.substring(30, 35).trim()); } catch (Exception ignored) {}
}
track.add(new TrackPoint(lat, lon, Math.max(alt, 0), timeSec));
```

**Проблема:** В IGC B-record:
- позиция 25–29 = **pressure altitude** (PPPPP, в метрах от QNE=1013.25)
- позиция 30–34 = **GPS altitude** (GGGGG, в метрах от эллипсоида WGS84)

Код берёт pressure altitude, и если оно 0, **переключается на GPS altitude**. Но:

1. Pressure altitude = 0 означает **уровень моря** — это валидное значение (полёт над океаном, взлёт с пляжа).
2. GPS altitude ≠ MSL altitude. Разница до ±100м (геоид-undulation).
3. Если баро-датчик логгера на секунду выдал 0 (глюк), та точка получит GPS altitude, отличающуюся на десятки метров от соседних pressure altitude → «ступенька» в треке.

Это прямая причина жалобы «неверно показаны высоты» для чужих IGC: трек, записанный на берегу моря, покажет +30м вместо 0 в начале полёта. Любой глюк баро даст ступеньку.

**Фикс:**

```java
int pressAlt = 0, gpsAlt = 0;
try { pressAlt = Integer.parseInt(line.substring(25, 30).trim()); } catch (Exception ignored) {}
if (line.length() >= 35) {
    try { gpsAlt = Integer.parseInt(line.substring(30, 35).trim()); } catch (Exception ignored) {}
}
// Стандарт IGC: всегда используем pressure altitude для анализа полёта.
// GPS altitude — только если pressure отсутствует (поле пустое, не "00000").
int alt = pressAlt;
if (alt == 0 && gpsAlt != 0) {
    // Только в случае явного отсутствия pressure (что редкость) — fallback.
    alt = gpsAlt;
}
// НЕ делаем Math.max(alt, 0) — отрицательная pressure altitude валидна
// (полёт над уровнем моря в низком давлении).
track.add(new TrackPoint(lat, lon, alt, timeSec));
```

#### BUG TR-5 (HIGH): Линейная интерполяция lat/lon вместо great-circle

**Файл:** `TrackReplayer.java:557-558`

```java
pilotLat = prevLat + (nextLat - prevLat) * frac;
pilotLon = prevLon + (nextLon - prevLon) * frac;
```

**Проблема:** Линейная интерполяция широты/долготы не учитывает кривизну Земли. Для коротких сегментов (<1км, 1Hz) ошибка < 1м — незначительна. Но:

- На высоких широтах (>60°) при сегментах >1км ошибка 5–10м.
- Если IGC имеет большие gap-ы (10-секундные интервалы между B-records при 9 м/с = 90м сегмент), пилот визуально «срезает» углы на поворотах.
- При пересечении линии смены даты (±180° долготы) линейная интерполяция даст «трансконтинентальный прыжок».

**Фикс:** Использовать great-circle Slerp для долготы, либо `Location.interpolate()` (Android API):

```java
// Простой способ через mid-point formula:
double lat1 = Math.toRadians(prevLat);
double lat2 = Math.toRadians(nextLat);
double dLat = Math.toRadians(nextLat - prevLat);
double dLon = Math.toRadians(nextLon - prevLon);

// Slerp на сфере
```

## 6. Safety-critical разбор: алгоритмы ветра и компаса

Эта секция — о багах, которые **прямо угрожают безопасности полёта**. Парапланерист, полагающийся на эти алгоритмы, может быть дезинформирован.

### 6.1. `WindDriftCalculator.java` — знаки инвертированы

Файл: `app/src/main/java/com/termo1/radar/flight/WindDriftCalculator.java`.

#### BUG WC-1 (SAFETY-CRITICAL): Знак heading перевёрнут — пилот летит ВНИЗ по ветру

**Файл:** `WindDriftCalculator.java:75-86`

```java
double windAngleRad = Math.toRadians(windDirection - desiredTrack);
double sinArg = (windSpeed / airspeed) * Math.sin(windAngleRad);
if (Math.abs(sinArg) > 1.0) {
    // Ветер сильнее воздушной скорости — компенсация невозможна
    return new WindCorrected(desiredTrack, airspeed, 0f, 0f, "Ветер слишком сильный");
}

double driftRad = Math.asin(sinArg);
float driftDeg = (float) Math.toDegrees(driftRad);

// Курс для компенсации
float heading = (desiredTrack - driftDeg + 360f) % 360f;
```

**Проблема:** Стандартная формула (Wind Correction Angle):

```
WCA = asin((Ws/Tas) × sin(Wd − TC))
```

- Знак WCA: **+ = ветер справа** → нужно довернуть ВПРАВО (увеличить heading).
- Heading = TC + WCA.

Код делает `heading = TC - WCA` → пилот направляется **В противоположную сторону** от нужной компенсации.

**Пример:** Ветер с севера (Wd=0), летим на восток (TC=90), Ws=0.5·Tas.

- WCA = asin(0.5 × sin(0 − 90)) = asin(0.5 × (−1)) = asin(−0.5) = **−30°** (ветер слева → нужно довернуть ВЛЕВО, уменьшить heading).
- Правильно: `heading = 90 + (−30) = 60°` (северо-восток, нос в ветер).
- Код: `heading = 90 − (−30) = 120°` (юго-восток, нос ПО ветру).

Пилот, следующий подсказке приложения, **будет снесён вдвое сильнее** ожидаемого.

**Фикс:**

```java
float heading = (desiredTrack + driftDeg + 360f) % 360f;
```

#### BUG WC-2 (SAFETY-CRITICAL): Ground speed: headwind отображается как tailwind

**Файл:** `WindDriftCalculator.java:89-90`

```java
double gs = airspeed * Math.cos(driftRad)
          + windSpeed * Math.cos(windAngleRad);
```

**Проблема:** `windSpeed × cos(Wd − TC)` — это **НЕ forward-компонента ветра**.

- Forward-компонента ветра = `−windSpeed × cos(Wd − TC)` (ветер ВПЕРЁД = положительный при tailwind, отрицательный при headwind).
- В коде `+windSpeed × cos` → headwind (cos > 0) увеличивает gs.

**Пример:** Ветер в лоб (Wd=0, TC=0, Ws=5, Tas=10):

- `cos(0 − 0) = 1`
- Код: `gs = 10 × 1 + 5 × 1 = 15 м/с` (WRONG — должно быть 5 м/с)
- Правильно: `gs = 10 × 1 − 5 × 1 = 5 м/с`

Пилот видит на экране «ground speed 15 м/с» при встречном ветре, когда реальная путевая — 5 м/с. Это критическая ошибка для планирования маршрута и расчёта времени до следующего поворотного пункта.

**Фикс:**

```java

          - windSpeed * Math.cos(windAngleRad);
```

### 6.3. `HeadingFilter.java` — сломан median filter

Файл: `app/src/main/java/com/termo1/radar/sensors/HeadingFilter.java`.

#### BUG HF-1 (SAFETY-CRITICAL): `computeMedian()` сортирует буфер IN-PLACE

**Файл:** `HeadingFilter.java:162-178`

```java
private double computeMedian() {
    double[] b = medianBuf;
    for (int i = 1; i < MEDIAN_WINDOW; i++) {
        double key = b[i];
        int j = i - 1;
        while (j >= 0 && b[j] > key) {
            b[j + 1] = b[j];
            j--;
        }
        b[j + 1] = key;
    }
    return b[2];
}
```

**Проблема:** `b = medianBuf` — сортировка IN-PLACE разрушает временной порядок буфера. Следующий `update()` делает `System.arraycopy(medianBuf, 1, medianBuf, 0, 4)` (строка 107) — удаляет `b[0]` (**НАИМЕНЬШИЙ** элемент после сортировки), а не самый старый.

**Пример:**

1. Окно: `[10, 20, 30, 40, 50]` (временной порядок).
2. New = 35 → shift → `[20, 30, 40, 50, 35]`.
3. sort → `[20, 30, 35, 40, 50]`, return 35 (median of `[20, 30, 40, 50, 35]` = 35 ✓).
4. Buffer теперь `[20, 30, 35, 40, 50]`.
5. Next new = 25 → shift removes `b[0] = 20` (smallest, NOT oldest=30) → `[30, 35, 40, 50, 25]`.
6. sort → `[25, 30, 35, 40, 50]`, return 35. Median of `[30, 40, 50, 35, 25]` = 35 (ок, совпадение).
7. Next new = 15 → shift removes `b[0] = 25` (smallest, NOT oldest=40) → `[30, 35, 40, 50, 15]`.
8. **Правильное окно должно быть `[40, 50, 35, 25, 15]` (median = 25).**
9. Код даёт median of `[30, 35, 40, 50, 15]` = 35. **НЕВЕРНО.**

**Фикс:**

```java
    double[] b = medianSortBuf;
    for (int i = 1; i < MEDIAN_WINDOW; i++) {
        double key = b[i];
```

**Проблема:** Если сервис убит системой без `onDestroy`, WakeLock удерживается до перезагрузки устройства. Во время реплея IGC (когда логирование остановлено) WakeLock всё равно активен — батарея тратится впустую.

**Фикс:**

```java
// Опция 1: таймаут 6 часов (максимальная длительность полёта)
wakeLock.acquire(6 * 60 * 60 * 1000L);

// Опция 2: не держать WakeLock во время реплея (экран и так включён через FLAG_KEEP_SCREEN_ON)
public void setReplayMode(boolean replay) {
    if (replay && wakeLock != null && wakeLock.isHeld()) {
        wakeLock.release();
    } else if (!replay && (wakeLock == null || !wakeLock.isHeld())) {
        wakeLock = pm.newWakeLock(PowerManager.PARTIAL_WAKE_LOCK, "TERMO1:RadarSvc");
        wakeLock.acquire();
    }
}
```

### 8.1. `ThermalDetector.java`

### 8.2. `SignalProcessor.java`

#### BUG SP-1 (HIGH): Направление турбулентности из `mean(bandpass) ≈ 0`

**Файл:** `SignalProcessor.java:243-246`


**Проблема:** `meanX/meanY` — среднее за окно 64 сэмпла bandpass-отфильтрованного сигнала. Полосовой фильтр (0.25-2.5 Гц) удаляет DC → `meanX ≈ 0, meanY ≈ 0`. `atan2(0, 0) = 0` → направление всегда ~север. ### 8.4. `WindStore.java`

#### BUG WS-1 (CRITICAL): Взвешенное среднее подшипов (wraparound bug)

**Файл:** `WindStore.java:122-135`

```java

    long age = nowMs - m.timestampMs;
    double ageFactor = 1.0 - (double) age / MAX_AGE_MS;
    if (ageFactor < 0.1) ageFactor = 0.1;

    weightedBearing += m.bearing * weight;   // ← WRONG

...
```

**Проблема:** Углы нельзя усреднять линейно. Два измерения: 350° и 10°, weight=1. `weightedBearing = 360`. `avg = 180°` (юг!). Правильный ответ: 0° (север). Ветер из `WindStore` разворачивается на 90-180° при измерениях по разные стороны 0/360.

**Фикс:** Векторное усреднение:

```java

for (WindMeasurement m : candidates) {
    long age = nowMs - m.timestampMs;
    double ageFactor = Math.max(0.1, 1.0 - (double) age / MAX_AGE_MS);
    double w = m.quality * ageFactor;
    double rad = Math.toRadians(m.bearing);
    sumSin += Math.sin(rad) * w;
    sumCos += Math.cos(rad) * w;
    totalWeight += w;
    weightedSpeed += m.speed * w;
}
double avgBearing = Math.toDegrees(Math.atan2(sumSin, sumCos));
if (avgBearing < 0) avgBearing += 360;
```

## 10. Оценка полезности прибора

### 10.1. Что приложение делает хорошо

1. **Архитектурный замысел** — порт XCSoar на Android-телефон с минимумом зависимостей. Это правильно: XCSoar — золотой стандарт для парапланеристов-планеристов, и его алгоритмы (CirclingComputer, ThermalLocator, LiftDatabase) многократно проверены.

2. **Полный pipeline сенсоров** — акселерометр + гироскоп + баро + магнетометр + GPS, с фильтрацией (баттерворт, медиана, alpha-beta) и state machine крутки. Для учебного проекта это очень сильный набор.

3. **Логирование** — параллельно CSV 50 Гц (для отладки) и IGC 1 Гц (для совместимости с XCSoar/SeeYou/Leonardo). Это позволяет анализировать полёты постфактум в стандартных инструментах.

4. **TTS голосовые подсказки** — правильная идея для параплана, где пилоту нельзя смотреть в телефон. Дебаунс 8 секунд, очереди TTS, локализация RU.

5. **PCM-генератор звука варно** — порт XCSoar VarioSynthesiser, 6 режимов, anti-click. Лучше, чем MediaPlayer с MP3.

6. **Foreground service** — корректно держит приложение в фоне, foreground notification видна.

7. **Документация** — README, manual.md, calibration.md, SIMULATION.md, algo.md, T5/T6/T9.md — студент явно старался описать, что делает код.

### 10.2. Что приложение делает плохо

1. **Safety-critical баги в алгоритмах ветра.** WindDriftCalculator инвертирует знаки — пилот, следующий подсказке приложения, будет лететь В ОБРАТНУЮ сторону. CirclingManager.estimateWindFromGps разворачивает ветер на 180°. WindStore усредняет подшипы линейно (350° + 10° → 180°). **Эти три бага independently делают показания ветра полностью недостоверными.**

2. **Компас сломан для всех нестандартных ориентаций крепления.** Отсутствует `remapCoordinateSystem()`. Парапланерист с телефоном на груди видит heading, отличный от реального на 90-180°. HeadingFilter.computeMedian ломает временной буфер после первого сэмпла.

3. **Баро-калибровка бесполезна.** `getAltCalibrated()` не exposed через SensorController. GPS калибрует baroOffset, но UI показывает `gpsManager.getAltitude()` или `altRaw` без offset.

4. **Производительность onDraw.** Десятки аллокаций на кадр → minor GC 5-15мс каждые 3-5 секунд → мерцание. Для прибора, который пилот смотрит в полёте, это недопустимо.

5. **Threading data races.** `ThermalLocator`, `CirclingManager`, `LiftDatabase`, `ThermalBlip` — чтение из UI без volatile/synchronized. На ARM возможны torn reads, NaN, крэши.

6. **Баро-deadband 0.2 м/с** скрывает реальное снижение 9 м/мин. Пилот не видит слабых нисходящих потоков.

7. **Vario-канал термо-детекции дезактивирован.** `NOISE_FLOOR_FIXED = 3.5м` слишком высок для реальной турбулентности.

8. **Реплей чужих IGC — основной источник жалоб пользователя — полностью сломан.** MSL-высота из живого GPS, trail-буфер рассчитан на live-режим, GPS не остановлен, 5Hz-эвристика ломает 1Hz/10Hz файлы, fix validity не проверяется, pressure altitude путается с GPS altitude.

### 10.4. Итоговая оценка

**В текущем состоянии (v0.2.74):**

- **Нельзя использовать как основной прибор в реальном полёте.** Safety-critical баги в ветре (WC-1, WC-2, WC-3, CM-1, WS-1) и компасе (HF-1, SC-1) могут дезинформировать пилота критическим образом.
- **Нельзя использовать для анализа чужих IGC** — все жалобы пользователя подтверждаются кодом.
- **Можно использовать как учебный проект** для изучения архитектуры Android-сенсорных приложений и алгоритмов термо-детекции. Код хорошо структурирован (по пакетам), есть документация, есть симуляция для отладки.

**После исправления P0 багов (11 штук):**

- **Станет рабочим учебно-вспомогательным прибором** для парапланеристов-любителей в простых погодных условиях.
- Все ещё не replacement для сертифицированных варно из-за:
  - Отсутствия аппаратной кнопки (пилот в перчатках не может нажать сенсор)
  - Зависимости от батареи телефона (3-4 часа экрана включённым)
  - Качества телефонных датчиков (особенно магнетометр)
  - Отсутствия резервного питания
- Но может занять нишу «бесплатное дополнение к основному варно» для образовательных целей, особенно для студентов-парапланеристов, изучающих физику термиков.

**После исправления P0 + P1 (25 штук):**

- Станет **полноценной альтернативой Flyskyhy на Android** для тех, кто не готов платить.

## 11. Резюме для студента

Твой проект — заметное достижение для одного разработчика. Архитектура пакета, портирование XCSoar-алгоритмов, IGC-логгер, TTS, PCM-варно, foreground service — это серьёзный объём работы, выполненный на достойном уровне. Особенно впечатляет, что ты **сам** нашёл и попытался исправить многие баги (видны комментарии `BUG-5 FIX`, `BUG-6 FIX`, `NEW-01`, `NEW-02`, `NEW-03`, `H-08`, `BUG-23`, `BUG-A18`, `BUG-3 FIX` и т.д. — это зрелый подход).

Однако:

1. **Три safety-critical бага в ветре** (WC-1, WC-2, CM-1) — это не просто «ошибки вывода», это **прямая угроза безопасности полёта**. Пилот, следующий твоим подсказкам, может быть снесён в препятствие. **Отключи wind guidance в UI до исправления этих багов**, или выведи предупреждение «ветро-расчёт в разработке, не полагаться».

2. **Жалобы пользователя на IGC-реплей полностью подтверждаются кодом.** Это не «странное поведение», а конкретные баги, которые **гарантированно** проявляются для любого чужого IGC. Исправление TR-1..TR-5 и MA-2..MA-5 решит 80% жалоб.

3. **MainActivity.java 3591 строк** — это god class, который невозможно поддерживать. Любой новый баг будет трудно локализовать. Разбей на 4-5 классов в первую очередь.

4. **Аллокации в onDraw** — это не «оптимизация», это correctness. Мерцание дисплея в полёте — это потеря информации в критический момент. Pre-allocate everything в `onSizeChanged()` и переиспользуй.

5. **Тесты.** У тебя 5+ сложных алгоритмов (WindDriftCalculator, CirclingManager, HeadingFilter, ThermalLocator, TrackReplayer). Каждый из них должен иметь unit-тесты с известными входными/выходными данными. Особенно для ветра — есть стандартные тестовые сценарии из XCSoar.

6. **Документация кода.** Уберите противоречивые комментарии (LiftDatabase 36 vs 12 секторов, HeadingFilter 30с vs 6с, TrackReplayer javadoc упоминает конкретный файл). Комментарий, который лжёт, хуже отсутствующего.

**Что у тебя получилось действительно хорошо:**

- Идея: термик-радар для параплана — это правильная ниша, не занятая на Android.
- Структура пакетов: domain-driven, читается хорошо.
- IGC-логгер: FAI-совместимый, правильный формат B-record с pressure + GPS altitude.
- PCM-генератор варно: порт XCSoar VarioSynthesiser — это правильно.
- Симуляция и scenario mode: инструменты наземной отладки.
- TTS с дебаунсом: правильная идея для aviation UI.
- StaticMapLoader с OSM-тайлами: бесплатно, без Google Maps API key.

**Что нужно переделать:**

- Алгоритмы ветра: три независимых бага, каждый ломает показания.
- Компас: remapCoordinateSystem обязателен для авиационного использования.
- Баро-калибровка: getAltCalibrated() должен быть в UI pipeline.
- OnDraw: zero allocations.
- Threading: volatile/synchronized для всех общих данных.
- MainActivity: разбить на 4-5 классов.

Если исправишь P0 баги и сделаешь unit-тесты на алгоритмы — у тебя будет **лучший бесплатный варно для параплана на Android**. Это того стоит.
