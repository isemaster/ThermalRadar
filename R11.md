
**Оценка:** правильно. Дыр в IGC-файле больше не будет — пишем B-record каждые 200 мс, помечаем `V` если фикса нет.

**Но есть тонкость с `hasFix`:**
```java
hasFix = !(lat == 0.0 && lon == 0.0);
```

Это **не проверяет** `fixAgeMs`. Если GPS-фикс был 30 секунд назад, но координаты не (0,0) — `hasFix = true`, пишем `A`. Это противоречит IGC-спецификации: `A` означает «3D fix valid now», а не «был когда-то».

**Правильно:**
```java
hasFix = !(lat == 0.0 && lon == 0.0) && fixAge <= 3000;
```

Или, ещё лучше, передавать `fixAge` через `formatBRecord` и там решать:

```java
boolean fixValid = !(lat == 0.0 && lon == 0.0) && fixAgeMs <= 3000 && accuracy < 50f;
```

(В исходном коде до v0.2.30 именно так и было, но студент при refactor'е упростил и потерял проверку `fixAge`.)

**5 Гц — хорошо, но есть проблема с GPS-координатами.** При 5 Гц B-records и 1 Гц GPS — 4 из 5 записей дублируют координаты с прошлой секунды. XCSoar/SeeYou это понимают, но если пользователь смотрит IGC в анализаторе, видит «ступеньки» в широте/долготе каждую секунду. Это норма для баро-IGC (5 Гц давления, 1 Гц GPS), но **правильнее** было бы во внутренних 4 записях ставить `V` (нет нового фикса) и `A` только в одной записи в секунду, когда пришёл новый GPS.

**Дополнительно.** `LOG_INTERVAL_MS = 200` означает децимацию с шагом 200 мс, но `recordSample()` вызывается из render loop с шагом 33 мс (30 Гц). Это значит, `recordSample()` вызывается ~6 раз до того, как пройдёт 200 мц и сэмпл запишется. Не критично (быстрая проверка `if (elapsed - lastLogElapsedMs < LOG_INTERVAL_MS) return;`), но создаёт лишнюю нагрузку. Можно децимировать на стороне вызова.

### 1.6. ✅ C-06: периодический flush + fsync

**Commit:** `cd18cb5 v0.2.27`
**Файл:** `IgcLogger.java:75-76, 84, 189-190, 384-404`

```java
private BufferedOutputStream currentOut;
private FileOutputStream currentFos; // для fsync
...
private long lastFlushMs;
...
// в startLogging:
currentFos = new FileOutputStream(file);
currentOut = new BufferedOutputStream(currentFos);
...
lastFlushMs = SystemClock.elapsedRealtime();
```

```java
private void writeLine(String line) {
    if (currentOut == null) return;
    try {
        byte[] bytes = (line + "\r\n").getBytes("ISO-8859-1");
        currentOut.write(bytes);
        if (crc32 != null) {
            crc32.update(bytes);
        }
        long now = SystemClock.elapsedRealtime();
        if (now - lastFlushMs > 5000) {
            currentOut.flush();
            if (currentFos != null) {
                currentFos.getFD().sync();
            }
            lastFlushMs = now;
        }
    } catch (IOException e) {
        Log.e(TAG, "IGC write error", e);
    }
}
```

**Оценка:** отлично. `getFD().sync()` — это именно то, что рекомендовалось в R9 §2.6. Теперь при краше максимум 5 секунд данных потеряно.

**Подводный камень:** `currentFos.getFD().sync()` может бросить `SyncFailedException` (extends `IOException`) на некоторых устройствах с read-only файловыми системами или при заполненном диске. Оно ловится общим `catch (IOException)`, но тогда `lastFlushMs = now` **не выполнится** (он в той же try-блоке после sync), и следующий вызов `writeLine()` снова попытается синкнуть. На заполненном диске это превратится в попытку `sync()` на каждой записи (50 раз в секунду), что замедлит приложение.

**Фикс:**
```java
long now = SystemClock.elapsedRealtime();
if (now - lastFlushMs > 5000) {
    try {
        currentOut.flush();
        if (currentFos != null) {
            currentFos.getFD().sync();
        }
    } catch (IOException syncEx) {
        Log.w(TAG, "sync failed (disk full?) — will retry next interval", syncEx);
    }
    lastFlushMs = now;   // ← всегда обновляем, даже при ошибке
}
```

#### v0.2.28: ridge-soaring fix

**Commit:** `bd1448a v0.2.28: fix false landing in ridge soaring`
**Файл:** `FlightStateMachine.java:42-48`

```java

private static final float LANDING_SPEED_MS = 1.0f;     // было 2.5

```

**Анализ:** `LANDING_SPEED 1.0` и `CONFIRM 60s` — это **критическое** улучшение для параплана. В ridge-soaring (динамический парящий полёт у склона) пилот часто летит со скоростью 2–4 м/с относительно земли (против ветра), при этом не «летит», а висит над склоном. Со старыми порогами (>2.5 м/с = летим) это давало ложную посадку — FSM думал, что пилот приземлился, хотя он висел у склона.

`LANDING_CONFIRM_MS = 60_000` — пилот должен **стоять** (скорость <1 м/с) **60 секунд** подряд, чтобы FSM решил, что приземлился. Это хорошо для параплана (нормальная посадка — это 5–15 секунд пробежки), но есть риск: если пилот приземлился и сразу выключил телефон — FSM не успеет сработать. Запись остановится только при `onDestroy()`. Это OK — `onDestroy` вызывает `igcLogger.destroy()` → `stopLogging()` → `closeCurrentFile()` с G-record.

`TAKEOFF_SPEED 3.0` — снизили порог, чтобы автостарт срабатывал при слабом ветре (пилот делает пару шагов и взлетает). `TAKEOFF_CONFIRM_MS = 5_000` — 5 секунд вместо 10. Это разумно, но создаёт риск ложного автостарта, если пилот идёт пешком со скоростью >3 м/с (беготня по старту). Например, побежал за потерянным шлемом — автостарт.

**Рекомендация:** добавить проверку высоты для взлёта: скорость >3 м/с И Δh > 3 м за 5 секунд. Это уберет ложные срабатывания от беготни по старту.

### 2.1. ❌ C-04: `bgHandler` всё ещё на main looper

**Файл:** `MainActivity.java:243`

```java
private Handler bgHandler = new Handler(Looper.getMainLooper());  // ← НЕ ИЗМЕНЕНО
```

Это означает, что **всё**, что бегает в `bgTask` (FSM, circlingManager, ThermalLocator, WindDrift, и теперь `updateSpeedBased`), выполняется на UI-треде.

Особенно болезненны:
1. `onFlightStarted()` → `igcLogger.startLogging()` → `new FileOutputStream()` + `writeHeader()` — 50–100 мс блокировки UI при разбеге пилота. Это **худший** момент для ANR, потому что пилот смотрит на экран и ждёт, что приложение покажет «полёт начался».

### 5.2. 🟡 NEW-02: `seekTo()` сбрасывает heading и vario

**Файл:** `TrackReplayer.java:533-543`

```java
public void seekTo(float timeSec) {
    totalSimSec = Math.max(0, Math.min(timeSec, getTotalTime()));
    currentIdx = 0;
    double oldLat = pilotLat, oldLon = pilotLon;
    if (track != null && track.size() > 0) {
        TrackPoint first = track.get(0);
        pilotLat = first.lat;
        pilotLon = first.lon;
        altitude = first.altMeters;
    }
    heading = 0;
    vario = 0;
}
```

После перемотки пилот «телепортируется» в первую точку трека, потом `update()` интерполирует до `totalSimSec`. Это даёт **визуальный скачок** — на одном кадре пилот в начале трека, на следующем — в точке перемотки. Плюс `heading = 0` (север) — радар развернётся на север.

Лучше — найти ближайшую точку трека к `timeSec` и установить `currentIdx` так, чтобы `update()` сразу интерполировал правильно:

```java
public void seekTo(float timeSec) {
    if (track == null || track.size() < 2) return;
    totalSimSec = Math.max(0, Math.min(timeSec, getTotalTime()));
    
    // Найти индекс ближайшей точки
    currentIdx = 0;
    for (int i = 0; i < track.size() - 1; i++) {
        if (track.get(i).timeSec - track.get(0).timeSec <= totalSimSec
                && track.get(i + 1).timeSec - track.get(0).timeSec > totalSimSec) {
            currentIdx = i;
            break;
        }
    }
    
    // Интерполировать позицию
    TrackPoint a = track.get(currentIdx);
    TrackPoint b = track.get(Math.min(currentIdx + 1, track.size() - 1));
    float segStart = a.timeSec - track.get(0).timeSec;
    float segEnd = b.timeSec - track.get(0).timeSec;
    float t = (segEnd > segStart)
            ? (totalSimSec - segStart) / (segEnd - segStart) : 0;
    pilotLat = a.lat + (b.lat - a.lat) * t;
    pilotLon = a.lon + (b.lon - a.lon) * t;
    altitude = a.altMeters + (b.altMeters - a.altMeters) * t;
    
    // Heading — по направлению к следующей точке
    heading = bearing(a.lat, a.lon, b.lat, b.lon);
    vario = (b.altMeters - a.altMeters) / Math.max(0.01f, (b.timeSec - a.timeSec));
}
```
