# ThermalRadar v0.2.5 — построчное ревью кода

**Репозиторий:** `https://github.com/isemaster/ThermalRadar` (тег `v0.2.5`)
**Объём:** ~4 300 строк Java в `app/src/main`, 1 тестовый класс
**Ревьюер:** senior-разработчик авионики и Android
**Жалобы пользователя:** не работает компас; не видна подгрузка карт.

---

## 0. Краткое резюме (executive summary)

Приложение написано с понятной архитектурой и хорошим набором сенсорной логики,
но содержит **два критических дефекта**, ровно соответствующих жалобам
пользователя, и около трёх десятков проблем среднего и низкого приоритета.

| Жалоба | Корневая причина | Файл / строка |
|---|---|---|
| **Не работает компас** | (a) DEADBAND в `HeadingFilter.update()` замораживает отображение при медленных поворотах (<4°/сэмпл). (b) `getCompassHeading()` возвращает `heading` даже когда `magAccuracy == LOW` и `getRotationMatrix()` регулярно возвращает `false`. (c) `calibrateHeading()` принудительно сбрасывает `compassReady=false` на 2 секунды, в течение которых радар использует GPS-курс со скоростью >2 м/с — иначе 0. | `HeadingFilter.java:133-141`, `SensorController.java:381-396, 608-613`, `MainActivity.java:1681-1692` |
| **Не подгружаются карты** | (a) Сервис `https://staticmap.openstreetmap.de/staticmap.php` **фактически мёртв** — DNS не резолвится (проверено: `curl` → `exit_code 6, Could not resolve host`). (b) Округление координат до `%.2f` (~1.1 км) при пороге обновления 500 м приводит к aliasing — карта может не обновляться даже при активном сервисе. | `StaticMapLoader.java:44-45, 124-125, 274-276` |

Ниже — детальный разбор каждого модуля. Все номера строк соответствуют
состоянию репозитория на теге `v0.2.5`.

## 2. Компас: почему он не работает

Компас в этом проекте — это цепочка из трёх компонентов:

1. `SensorController.updateCompass()` — берёт `gravity` + `magnetometer`, строит `rotationMatrix`, получает `heading`.
2. `HeadingFilter.update()` — фильтрует raw heading: CLAMP → MEDIAN(3) → α-β → DEADBAND.
3. `MainActivity.getCompassHeading()` — выбирает между компасом и GPS-fallback.

Баги есть в каждом из трёх, и они суммируются.

### 2.1. `HeadingFilter.java` — построчно

```java
// HeadingFilter.java:50-53
public HeadingFilter() {
    this(0.75, 0.3);
}
```

**Баг №1 — Несоответствие комментариев и кода.** Javadoc на строке 11-12 и
на строке 50 говорит о `α=0.6, β=0.2`, но конструктор использует `α=0.75, β=0.3`.
Если коэффициенты подбирались эмпирически — фиксируйте итог в комментарии, а не
оставляйте «рекомендуемые» значения из `algo.md`. Это не cosmetic: при `α=0.75`
фильтр ведёт себя значительно «жёстче», чем при `α=0.6`, и любой, кто читает
код и видит `0.6` в комментарии, будет думать, что фильтр более инерционный,
чем он есть на самом деле.

**Рекомендация:**
```java
/**
 * Конструктор по умолчанию.
 * α=0.75, β=0.3 — эмпирически подобрано для параплана (частота сенсора ~50 Гц,
 * типичная угловая скорость в спирали ~20°/с). См. testdata/heading-tune.csv.

public HeadingFilter() {
    this(0.75, 0.3);
```

```java
// HeadingFilter.java:62-95
public double update(double rawHeading, long timeMs) {
    if (!initialized) { /* ... */ }

    long dtMs = timeMs - lastTimeMs;
    if (dtMs <= 0) dtMs = 1;
    if (dtMs > 2000) dtMs = 2000; // cap
    double dt = dtMs / 1000.0;
    lastTimeMs = timeMs;

    // 1. CLAMP
    double clamped = rawHeading;
    if (initialized) {                 // ← (1)

        // нормализация в [-180, 180]
        while (diff > 180) diff -= 360;
        while (diff < -180) diff += 360;
        double maxDelta = MAX_DELTA_DEG_PER_SEC * dt;
        if (Math.abs(diff) > maxDelta) {
            clamped = lastOutputDeg + Math.signum(diff) * maxDelta;
            clamped = (clamped % 360 + 360) % 360;
        }
    }
    // ...
}
```

**Баг №2 — Мёртвый код на строке 84.** Условие `if (initialized)` ВСЕГДА
истинно в этой точке, потому что на строке 63 мы уже вышли через `return`,
если `!initialized`. Это либо артефакт рефакторинга, либо задуманная
защита от изменений — но ни одну из этих ролей код не выполняет.
Удалите проверку:

```java
double clamped = rawHeading;
double diff = rawHeading - lastOutputDeg;
while (diff > 180) diff -= 360;
while (diff < -180) diff += 360;
double maxDelta = MAX_DELTA_DEG_PER_SEC * dt;
if (Math.abs(diff) > maxDelta) {
    clamped = lastOutputDeg + Math.signum(diff) * maxDelta;
    clamped = (clamped % 360 + 360) % 360;
}
```

**Баг №3 — CLAMP использует `lastOutputDeg`, а не `xk`.** `lastOutputDeg`
обновляется ТОЛЬКО когда deadband разрешает обновление (строка 140).
Если дисплей «заморожен» deadband-ом на 4°, а сенсор прыгнул на 30°
(например, магнитометр словил помеху от магнита в кокпите), CLAMP
пропустит этот скачок через `lastOutputDeg + sign*maxDelta`, но
сам `lastOutputDeg` останется старым. Получается, что:
- фильтр (`xk`) получает «мягко ограниченный» скачок,
- но `lastOutputDeg` (который читает CLAMP на следующем шаге) не двигается,
- в итоге несколько подряд импульсов будут последовательно ограничиваться
  от одного и того же значения, накапливая ошибку в фильтре.

CLAMP должен сравнивать с `xk` (внутреннее состояние α-β), а не с
`lastOutputDeg` (display state):

```java
double diff = rawHeading - xk;            // было: lastOutputDeg
while (diff > 180) diff -= 360;
while (diff < -180) diff += 360;
double maxDelta = MAX_DELTA_DEG_PER_SEC * dt;
if (Math.abs(diff) > maxDelta) {
    clamped = xk + Math.signum(diff) * maxDelta;
    clamped = (clamped % 360 + 360) % 360;
}
```

**Баг №4 — Аллокация на горячем пути.** `medianBuffer.toArray(new Double[0])`
на каждом сэмпле (50 Гц) аллоцирует массив `Double[3]` + 3 boxed `Double`.
В GC-паузах это даёт микрофризы в 5-15 мс — как раз достаточно, чтобы
vario lag ощущался. Для окна 3 медиана считается тривиально без аллокации:

Или ещё проще — заменить `ArrayDeque<Double>` на `double[3]` + счётчик,
избавившись от бокса вообще.

**Баг №5 — Медиана игнорируется на коротких буферах.** Для первых двух
сэмплов после `reset()` возвращается `getLast()`, то есть raw. Если `reset()`
вызывается часто (например, после `calibrateHeading()`, который зовёт
`headingFilter.reset()` на строке 611), то первые 2 сэмпла пойдут в
α-β без медианы. Это не критично, но inconsistent.

```java
// HeadingFilter.java:130-141

if (displayDiff > 180) displayDiff = 360 - displayDiff;

if (displayDiff < DEADBAND_DEG) {
    return Double.NaN; // сигнал: не обновлять отображение
}

return outputDeg;
```

**Баг №6 (КРИТИЧЕСКИЙ — причина «не работает компас») — DEADBAND замораживает индикацию.**
Константа `DEADBAND_DEG = 4.0`. Логика: если изменение filtered heading
относительно последнего показанного меньше 4°, вернуть `NaN` и НЕ обновить
`lastOutputDeg`. В `SensorController.updateCompass()` (строка 391-394):

```java
double filtered = headingFilter.update(newHeading, SystemClock.elapsedRealtime());
if (!Double.isNaN(filtered)) {
    heading = (float) filtered;
}
```

При `NaN` поле `heading` не обновляется. То есть для UI компас «застывает»
до тех пор, пока накопленное изменение не превысит 4°.

Что это значит на практике для параплана:
- В прямолинейном полёте с рысканьем ±2° компас **вообще не сдвинется**.
- В слабой спирали с угловой скоростью 10°/с и частотой сенсора 50 Гц
  шаг изменения = 0.2°, а deadband = 4°. Чтобы компас сдвинулся на один
  шаг отображения, нужно накопить 4° в фильтре, что произойдёт через ~20
  сэмплов (~0.4 с). Но за эти 0.4 с α-β-фильтр с `α=0.75` «двигает»
  `xk` очень медленно (т.к. инновация 0.2° × 0.75 = 0.15° за шаг).
  Реально нужно ждать 4°/0.15° ≈ 27 шагов ≈ 0.54 с, прежде чем дисплей
  сдвинется на 4°. Это даёт характерный «ступенчатый» эффект, который
  пользователь воспринимает как «компас не работает / залипает».

Для авионики deadband на отображение — нормальная практика (чтобы не было
дрожания цифр), но:
- deadband должен быть не больше 1° (а лучше 0.5°),
- возвращать нужно не `NaN`, а ТЕКУЩЕЕ отфильтрованное значение, чтобы
  caller сам решал, обновлять ли дисплей.

**Исправление:**
```java
// Вариант A: уменьшить deadband и всегда возвращать число
private static final double DEADBAND_DEG = 0.5;
// ...
if (displayDiff < DEADBAND_DEG) {
    return lastOutputDeg;   // вместо NaN — возвращаем «последнее показанное»
}
lastOutputDeg = outputDeg;
return outputDeg;

// Вариант B (лучше): вынести deadband на уровень UI, а не фильтра.
// Фильтр должен всегда возвращать filtered heading.
return outputDeg;
```

Я настоятельно рекомендую **Вариант B**: deadband — это UI-концерн.
В авиационных EFIS deadband на цифровых показаниях есть, но на аналоговой
стрелке (которую вы рисуете на Canvas) его быть не должно.

```java
// HeadingFilter.java:165-167
public double getTurnRate() {
    return vk;
}
```

Замечание: `vk` — это оценка угловой скорости в °/с. Если фильтр только что
сделал `reset()`, `vk = 0`. Если deadband замораживает индикацию, `vk`
продолжает обновляться. Это правильное поведение, но в `CirclingManager`,
который использует turn rate для определения спирали, это означает, что
спираль детектируется даже когда дисплей «застыл». Нужно проверить, не
приводит ли это к ложным срабатываниям голосовых подсказок «СПИРАЛЬ».

**Баг №9 — Магнетометр сохраняется в volatile, но не блокируется.**
`latestMagX/Y/Z` — volatile, но операция `gravBuf = {latestMagX, latestMagY, latestMagZ}`
далее в `updateCompass()` — это 3 отдельных чтения volatile. Между ними
другой поток не пишет (только `onSensorChanged` пишет, а он однопоточный
от SensorManager), так что на практике это безопасно. Но если в будущем
появится второй writer — появится гонка. Стоит либо явно документировать
«single-writer», либо использовать `float[]` + synchronized.

```java
// SensorController.java:363-408
private void updateCompass() {
    if (!hasGravityAccel) return;

    gravBuf[0] = gravityAccelX;
    gravBuf[1] = gravityAccelY;
    gravBuf[2] = gravityAccelZ;
    geomBuf[0] = latestMagX;
    geomBuf[1] = latestMagY;
    geomBuf[2] = latestMagZ;

    if (SensorManager.getRotationMatrix(rotationMatrix, null, gravBuf, geomBuf)) {
        SensorManager.getOrientation(rotationMatrix, orientationVals);
        float newHeading = (float) Math.toDegrees(orientationVals[0]);
        rawHeading = newHeading;
        pitch = orientationVals[1];
        roll = orientationVals[2];
        if (newHeading < 0.0f) newHeading += 360.0f;

        if (!compassReady) {
            heading = newHeading;
            headingFilter.reset();
            compassReady = true;
        } else {
            if (isHeadingOnHold()) {
                heading = headingHoldValue;
            } else {
                double filtered = headingFilter.update(newHeading, SystemClock.elapsedRealtime());
                if (!Double.isNaN(filtered)) {
                    heading = (float) filtered;
                }
            }
        }
    } else if (magnetometer == null) {
        // Нет магнитометра — pitch/roll из гравитации
        pitch = (float) Math.atan2(-gravityAccelX,
                Math.sqrt(gravityAccelY * gravityAccelY + gravityAccelZ * gravityAccelZ));
        roll = (float) Math.atan2(gravityAccelY, gravityAccelZ);
    } else {
        compassReady = false;                                // ← (3) КРИТИЧНО
        pitch = (float) Math.atan2(-gravityAccelX,
                Math.sqrt(gravityAccelY * gravityAccelY + gravityAccelZ * gravityAccelZ));
        roll = (float) Math.atan2(gravityAccelY, gravityAccelZ);
    }
}
```

**Баг №10 (КРИТИЧЕСКИЙ) — `compassReady = false` при единичном отказе `getRotationMatrix`.**
`getRotationMatrix()` возвращает `false`, если определитель матрицы
ускорения слишком мал — то есть при сильном ускорении (турбулентность,
старт, клевок). В парапланерном полёте это случается регулярно.

Что происходит при `compassReady = false`:
1. На следующем сэмпле магнетометра, если `getRotationMatrix` снова успешен,
   попадаем в ветку `if (!compassReady)` (строка 381), которая вызывает
   `headingFilter.reset()` — фильтр сносится, и `heading` прыгает на raw
   значение с магнетометра.
2. В `MainActivity.getCompassHeading()` (строка 1683) `isCompassReady()`
   вернёт `false`, и приложение уйдёт в GPS-fallback. Если скорость <2 м/с

**Баг №12 (КРИТИЧЕСКИЙ) — `calibrateHeading()` убивает компас на 2 секунды.**
Эта функция вызывается при старте лога (MainActivity:709, 837) и при
обнаружении старта полёта. На 2 секунды:
- `compassReady = false` → `getCompassHeading()` уходит в GPS-fallback.
- `headingFilter.reset()` → фильтр теряет историю.
- `isHeadingOnHold() == true` → `heading = headingHoldValue` (замороженное значение).

Если в этот момент GPS-скорость <2 м/с (старт с земли), то `getCompassHeading()`

курса. Если пользователь в этот момент смотрит на радар, он видит, что
«компас показывает 0» — то есть не работает.

калибровки девиации (compass swing). Это НЕ значит «заморозить показания

манёвра, используйте «trust period», в течение которого фильтр работает,

**Баг №13 — Fallback на GPS срабатывает только при полном отсутствии магнитометра.**
Условие `magnetometer == null` истинно только если `getDefaultSensor(TYPE_MAGNETIC_FIELD)`

видит «компас не работает».

```java
public void updateGpsHeading(float gpsHeading) {

    boolean compassUnreliable = !compassReady
            || magnetometer == null
            || magAccuracy < SensorManager.SENSOR_STATUS_ACCURACY_MEDIUM
            || compassFailureStreak > COMPASS_FAILURE_THRESHOLD;
    
    if (compassUnreliable) {
        double filtered = headingFilter.update(gpsHeading, SystemClock.elapsedRealtime());
        if (!Double.isNaN(filtered)) {
            heading = (float) filtered;
        }
    }
```

### 2.3. `MainActivity.getCompassHeading()` — построчно

```java

        return sensorController.getHeading();
```

**Баг №14 (КРИТИЧЕСКИЙ) — Нет проверки `magAccuracy`.** `isCompassReady()`
возвращает `compassReady`, который становится `true` при **первом
успешном** `getRotationMatrix`. После этого магнетометр может дать
худшую калибровку (пользователь положил телефон рядом с магнитом,
или софт-железная дисторсия изменилась из-за чехла), `magAccuracy`
упадёт до `SENSOR_STATUS_UNRELIABLE`, но `compassReady` останется `true`.
Компас продолжит возвращать мусорные значения, и UI будет их отображать.

**Исправление:**
```java
private float getCompassHeading() {
    boolean magReliable = sensorController.isCompassReady()
            && sensorController.isMagAccurate();      // ← добавить
    
    if (magReliable) {
        return sensorController.getHeading();
    }
    // Fallback на GPS — но с проверкой скорости
    if (gpsManager.isReady() && gpsManager.getSpeed() > 2f
            && gpsManager.getFixAgeMs() < 5000) {
        sensorController.updateGpsHeading(gpsManager.getHeading());
        return sensorController.getHeading();
    }
    // Совсем нет источника — держим последнее (не возвращаем 0!)
    return sensorController.getHeading();
}
```

Заметьте ещё: `return 0.0f` в конце — это катастрофа. Если ни компас,
ни GPS не работают, радар разворачивается на север. В авиации в такой
ситуации показывают «HDG FAIL» красным, а не 0°.

**Баг №17 — Resource leak в `BitmapFactory.decodeStream`.** Если
`decodeStream` бросает `OutOfMemoryError` (которая не ловится
`catch (Exception e)` — это `Error`), `is` и `conn` не закрываются.
Используйте try-with-resources:

```java

try {
    URL url = new URL(urlStr);
    conn = (HttpURLConnection) url.openConnection();
    conn.setConnectTimeout(5000);
    conn.setReadTimeout(10000);
    conn.setRequestProperty("User-Agent", "ThermalRadar/0.2.5");
    conn.connect();

    if (conn.getResponseCode() != 200) {
        Log.e(TAG, "HTTP " + conn.getResponseCode() + " for " + urlStr);
        return null;
    }

    Bitmap bitmap;
    try (InputStream is = conn.getInputStream()) {
        BitmapFactory.Options opts = new BitmapFactory.Options();
        opts.inPreferredConfig = Bitmap.Config.RGB_565;  // 50% памяти
        bitmap = BitmapFactory.decodeStream(is, null, opts);
    }
    if (bitmap == null) return null;
```

**Баг №19 (КРИТИЧЕСКИЙ) — Округление до 0.01° (~1.1 км) ломает порог 500 м.**
`Math.round(lat * 100.0) / 100.0` — это округление до сотых долей градуса.
На широте 60° одна сотая градуса ≈ 1.1 км по меридиану и ≈ 0.55 км по
параллели.

Порог обновления `UPDATE_THRESHOLD_M = 500`. Что происходит:
1. Пилот на старте: `lat=60.004, lon=30.301`. `roundedLat = 60.00, roundedLon = 30.30`. Загружаем карту для центра `(60.00, 30.30)`.
2. Пилот пролетел 400 м на север: `lat=60.0076, lon=30.301`. Расстояние до cached center `(60.00, 30.30)` = 840 м > 500 м → триггер обновления.
3. Новый `roundedLat = Math.round(60.0076 * 100) / 100 = 60.01`. `roundedLon = 30.30`. Загружаем новую карту для `(60.01, 30.30)`.
4. Пилот пролетел ещё 300 м на север: `lat=60.0103, lon=30.301`. Расстояние до `(60.01, 30.30)` = 33 м < 500 м → обновления нет.
5. Пилот пролетел ещё 300 м на север: `lat=60.013, lon=30.301`. Расстояние до `(60.01, 30.30)` = 333 м < 500 м → обновления нет.
6. Пилот пролетел ещё 300 м: `lat=60.0157, lon=30.301`. Расстояние до `(60.01, 30.30)` = 633 м > 500 м → триггер. `roundedLat = 60.02`. Загружаем для `(60.02, 30.30)`.

Видите проблему? Реальный порог обновления получился **≈1000-1100 м**
вместо 500 м, потому что округление всегда «дотягивает» позицию до
центра ячейки. Карты подгружаются в 2 раза реже, чем задумано.

Кроме того, в точке `(60.005, lon)` пилот находится ровно на границе
ячеек. Малейший дрейф ±5 м переключает cached center между `60.00` и
`60.01`, и при каждом переключении делается HTTP-запрос. Это уже
flapping-эффект.

**Исправление:** не округляйте координаты для триггера обновления.

**Баг №21 — Race condition: recycle bitmap, который ещё рисуется.**
`entryRemoved` вызывает `bitmap.recycle()`, когда LRU вытесняет запись.
Но в этот момент `RadarRenderer.draw()` может держать ссылку на тот же
bitmap (через `backgroundMap`) и пытаться его отрисовать. Результат:
`Canvas.drawBitmap()` упадёт с `IllegalStateException: Cannot draw a recycled Bitmap`.

Сценарий:
1. `loadMap()` на cache hit достаёт bitmap из `memCache.get()` и
   вызывает `callback.onMapLoaded(bitmap, ...)`.
2. Callback (в `MainActivity.setCallback`) зовёт
   `radarRenderer.setBackgroundMap(bitmap, ...)` — renderer сохраняет
   ссылку в `backgroundMap`.
3. Через 30 секунд `loadMap()` для нового региона добавляет в memCache
   новую bitmap. Старая вытесняется, `entryRemoved` её recycle-ит.
4. Следующий кадр `RadarRenderer.draw()` пытается рисовать
   `backgroundMap` (старую bitmap) → crash.

**Исправление:** не recycle-ите в `entryRemoved`. Renderer — единственный
владелец ссылки после `setBackgroundMap`. LruCache должен просто
забывать ссылку. Если очень хочется экономить память —
используйте `BitmapPool` и явно возвращайте битмапы из renderer
при `setBackgroundMap(null)`:

```java
// В RadarRenderer:
public void setBackgroundMap(Bitmap bitmap, ...) {
    Bitmap old = this.backgroundMap;
    this.backgroundMap = bitmap;
    // старый bitmap просто отпускается — GC заберёт
    // НЕ делаем old.recycle() — он мог быть в memCache.
}
// И в LruCache:
protected void entryRemoved(boolean evicted, String key,
                             Bitmap oldValue, Bitmap newValue) {
    // Ничего не делаем. GC сам разберётся.
}
```

```java
// StaticMapLoader.java:109-111

    executor.shutdown();
}
```

**Баг №22 — `shutdown()` без ожидания.** Если `destroy()` вызывается

### 3.2. `RadarRenderer.computeMapOffsetPx()` — построчно

```java
// RadarRenderer.java:103-104
// Пикселей на метр при zoom 14, ~60°N (Санкт-Петербург)
private static final double METERS_PER_PIXEL = 4.78;
```

**Баг №23 — Хардкод `METERS_PER_PIXEL` для конкретной широты.**
Реальное значение зависит от:
- широты (`m/px = 156543.03 * cos(lat) / 2^zoom`),
- zoom level.

На широте 45° при zoom 14 это будет 6.56 м/px (вместо 4.78).
Если пользователь полетит в Альпах (45°N), карта будет сдвигаться
на ~30% сильнее, чем реальное смещение пилота → радар и карта
рассинхронизируются.

**Исправление:**
```java
private double metersPerPixel(int zoom, double lat) {
    return 156543.03 * Math.cos(Math.toRadians(lat)) / Math.pow(2, zoom);
}
// в computeMapOffsetPx:
double mpp = metersPerPixel(mapZoom, mapCenterLat);
float offsetX = (float) (-eastM / mpp);
float offsetY = (float) (northM / mpp);
```

```java
// RadarRenderer.java:693-716
private float[] computeMapOffsetPx() {
    if (!mapValid || !pilotPositionValid) {
        return new float[]{0, 0};                            // ← (6) Аллокация
    }
    android.location.Location.distanceBetween(
            mapCenterLat, mapCenterLon,
            pilotLat, pilotLon,
            mapDistRes);
    float distM = mapDistRes[0];
    float bearing = mapDistRes[1];

    double rad = Math.toRadians(bearing);
    double eastM = distM * Math.sin(rad);
    double northM = distM * Math.cos(rad);

    float offsetX = (float) (-eastM / METERS_PER_PIXEL);
    float offsetY = (float) (northM / METERS_PER_PIXEL);

    return new float[]{offsetX, offsetY};                    // ← (6) Аллокация
}
```

вызывается из `draw()`, который идёт ~30-60 раз в секунду. На каждом
кадре — `new float[2]` для возвращаемого значения + `new float[3]`
для `mapDistRes` (если это поле, а не локальный). Это сотни аллокаций
в секунду. GC-паузы видны как микрофризы.

**Исправление:**

```java
private final float[] mapDistRes = new float[3];    // уже поле? проверьте

// в draw():

// использовать mapOffsetOut[0], mapOffsetOut[1]
```

### 4.3. `VarioManager.java`


**Баг №32 — Барометрическая формула без учёта температуры.** Это
стандартная ISA-формула (International Standard Atmosphere), дающая
pressure altitude. Реальная высота отличается:
- в холодном воздухе (зима, -20°C) — pressure altitude ВЫШЕ реальной,
- в тёплом (лето, +30°C) — НИЖЕ.

Для параплана в горах ошибка может достигать 10-15% (на 1000 м высоты
это 100-150 м). Это критично для `altAgl` (высота над рельефом) —
пилот может думать, что у него 200 м до земли, а реально 50 м.

В авиации используют QNH (давление на уровне моря, приведённое) и
OAT (температуру наружного воздуха). QNH можно получить:
- из METAR ближайшего аэропорта (но нужен интернет),
- от пользователя (ручной ввод при старте).

Минимальное улучшение — добавить температурную коррекцию:
```java
// Laplace formula с температурой
double altRaw = (Math.pow(baselinePressure / pressure, 0.190263) - 1)
    * (287.05 * (273.15 + oat)) / 9.80665;
// где oat — outside air temperature, °C
```

## 5. Авиационные риски (avionics view)

С точки зрения парапланерной авиации, помимо описанных багов, есть
системные риски, которые нужно учитывать при доработке:

### 5.1. Single-point heading failure

В авиации правило redundancy: ни один источник heading не должен
быть single point of failure. Сейчас в приложении:
- Primary: магнетометр (1 шт.).
- Fallback: GPS bearing (только при скорости >2 м/с, fix age <5 сек).
- Tertiary: нет.

На параплане:
- Магнетометр сбоит в горах (магнитные породы), рядом с ЛЭП, в кокпите
  с металлическими элементами.
- GPS bearing в спирали (медленное движение по кругу) ненадёжен —
  track ≠ heading при ветре.
- Гироскоп мог бы дать relative heading (rate integration), но он
  не используется для компаса.

**Рекомендация:** добавить gyro-based heading integration как
третий источник. При потере магнетометра на N секунд,
переключаться на gyro integration (zero-rate-update при_DETECTed
straight flight), корректируя от GPS при наличии скорости >5 м/с.

### 5.2. Нет индикации деградации режима

В реальном EFIS пилот всегда видит:
- `MAG FAIL` красным, если магнетометр неисправен,
- `GPS FAIL` красным, если GPS потерян,
- `BARO FAIL` красным, если барометр сбоит,
- `HDG DEGRADED` жёлтым, если использует fallback source.

В ThermalRadar UI показывает только статус полёта (`ПОИСК`, `ТЕРМИК`,
`НАБОР`, `СНИЖЕНИЕ`, `СПИРАЛЬ`). Пилот не знает, что компас работает
через GPS-fallback или что вариометр использует кэшированный baseline.

**Рекомендация:** добавить в `UiManager` строку статуса сенсоров:

```java
public void drawSensorStatus(Canvas c, float left, float top,
                             boolean magOk, boolean gpsOk, boolean baroOk,
                             int magAccuracy) {
    // MAG: green/yellow/red
    // GPS: green/yellow/red (с количеством спутников?)
    // BARO: green/red
}
```

### 5.3. Логгирование не синхронизировано по времени

`System.currentTimeMillis()` (wall clock, может прыгать при NTP-синхронизации)
используется в `GpsManager` для `lastFixMs`, `LogManager`, `IgcLogger`.
`SystemClock.elapsedRealtime()` (monotonic, не прыгает) — в `SensorController`
для `lastBaroNanos`, `headingFilter`.

В IGC-логе временная шкала должна быть monotonic UTC (B-record). Если
NTP синхронизация поменяет время во время полёта, в логе будут
непоследовательные метки. Используйте `elapsedRealtime()` для
всех временных меток внутри сессии и конвертируйте в UTC только
при записи B-record (зафиксировав оффсет при старте лога).

### 5.4. Нет предпосадочного чек-листа

В авиации перед взлётом пилот проверяет инструменты. В ThermalRadar
при `onCreate` сенсоры стартуют немедленно, без проверки:
- есть ли магнетометр,
- откалиброван ли он,
- есть ли GPS fix,
- есть ли барометр.

Если хотя бы один критический сенсор отсутствует, приложение должно
показать красный экран «CHECK SENSORS» с инструкцией, а не молча
продолжать работать с мусорными данными.

### 5.5. Звуковой variometer

`VarioSoundManager` не разбирался подробно, но если он использует
TTS для голосовых подсказок направления на термик — это критический
avionics риск: TTS latency 200-500 мс означает, что к моменту
озвучки «Термик справа» термик может быть уже сзади. Для критических
предупреждений нужен короткий тоновой сигнал (<50 мс latency).

### 5.6. Режим «слепого полёта» (BlindFlightMode)

Из названия и использования видно, что это режим полёта «на ощупь»
по голосовым подсказкам. Это крайне опасный режим для параплана:
- Если TTS зависнет (что бывает на Android при низком приоритете
  процесса), пилот останется без информации.
- Если наушник отсоединится — пилот останется без информации.
- Если батарея сядется — пилот останется без информации.

В авиации «blind flight» = полёт по приборам в IMC, и для этого
нужны дублирующие системы. Для параплана такой режим допустим
только как вспомогательный, с явным предупреждением пользователю
при активации.

```java
        int centerTx = tileXY[0], centerTy = tileXY[1];
        // Загружаем 9 тайлов параллельно
        for (int dx = -1; dx <= 1; dx++) {
            for (int dy = -1; dy <= 1; dy++) {
                int tx = centerTx + dx, ty = centerTy + dy;
                String url = "https://a.tile.openstreetmap.org/" + zoom + "/" + tx + "/" + ty + ".png";
                // ... загрузка через OkHttp с дисковым кэшем ...
            }
        }
        // Склеиваем 9 тайлов в один 768×768 Bitmap
    }
    
    private static int[] latLonToTile(double lat, double lon, int zoom) {
        int n = 1 << zoom;
        int tx = (int) Math.floor((lon + 180.0) / 360.0 * n);
        double latRad = Math.toRadians(lat);
        int ty = (int) Math.floor((1.0 - Math.log(Math.tan(latRad) + 1/Math.cos(latRad)) / Math.PI) / 2.0 * n);
        return new int[]{tx, ty};
    }
}
```

Преимущества tile-based подхода:
- Не зависит от сторонних static map сервисов (которые умирают).
- Параллельная загрузка (9 тайлов быстрее, чем одна static map).
- Дисковый кэш на уровне отдельных тайлов (переиспользование при
  смещении на 1 тайл — 6 из 9 тайлов уже в кэше).
- Соответствие OSM TOS (тайлы — основной способ использования OSM).

## 8. Заключение

ThermalRadar v0.2.5 — это **хорошо концептуально проработанный, но
недоотлаженный проект**. Автор понимает предметную область
(парапланеризм, термики, спиральный набор), знает алгоритмы
(alpha-beta, EKF, медианная фильтрация), но допустил классические
ошибки:

1. **Trust external services without fallback** — `staticmap.openstreetmap.de`
   умер, и приложение потеряло всю функциональность карт. Никакого
   health check, никакой альтернативы.
2. **Mixing display state with filter state** — DEADBAND в `HeadingFilter`
   привязал плавность индикации к алгоритму фильтрации, что дало
   «залипающий компас».
3. **Brittle sensor fallback** — `compassReady = false` на любой чих
   `getRotationMatrix`, без счётчика отказов и без удержания последнего
   хорошего значения.
4. **God Object в MainActivity** — 2709 строк в одном классе означают,
   что любое изменение рискует сломать что-то в соседнем функционале.
5. **Отсутствие тестов на критические пути** — компас, карты и
   вариометр не покрыты, regression неизбежен.

Минимальный набор правок (Priority 0, ~8 часов работы) вернёт
приложению работоспособность. Полный рефакторинг (Priority 2, ~130

**Главный avionics-урок:** в авиации «работает» означает «работает
предсказуемо и отказоустойчиво». Текущая реализация работает
непредсказуемо (компас то залипает, то прыгает, то сбрасывается) и
не отказоустойчиво (умерла одна HTTP-сервис — умерли все карты).
Для парапланерного прибора это неприемлемо.
