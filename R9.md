## 0. TL;DR

Причина «плохо работает автостарт/автостоп IGC» — не одна баг, а **композиция из семи системных дефектов**. Главные из них:

| # | Баг | Влияние на автостарт/автостоп |
|---|-----|-------------------------------|
| **C-01** | `FlightStateMachine.updateSpeedBased()` **никогда не вызывается** из `MainActivity`. Весь XCSoar-style детектор (единственный адекватный механизм для параплана) — мёртвый код. | Автостарт/автостоп работают только по высоте, что физически некорректно для параплана. |
| **C-02** | В `onPause()` сенсоры, GPS и WakeLock **отключаются**, если `!logManager.isLogging()`. Если пилот выключил экран до взлёта (стандартная предполётная практика) — FSM не получает данных вообще. | Автостарт не сработает в самом частом сценарии. |
| **C-03** | В `confirmStopLogging()` (ручной стоп) **забыт вызов `igcLogger.stopLogging()`**. IGC-файл остаётся без G-record, не flush'ится. | Ручной стоп даёт повреждённый файл. |
| **C-04** | В `onFlightStarted()` файловый I/O (`new FileOutputStream(...)`) выполняется **в UI-треде** (bgHandler использует `Looper.getMainLooper()`). | ANR на медленных устройствах / при забитом внутреннем хранилище. |
| **C-05** | `IgcLogger.recordSample()` **молча пропускает B-records**, если GPS-фикс старый (>5 с) или координаты (0,0). В IGC-файле образуются дыры во времени. | Валидаторы IGC (XCSoar, SeeYou) помечают файл как некорректный. |
| **C-06** | `IgcLogger` **не делает периодический flush/fsync**. `BufferedOutputStream` с буфером 8 КБ теряет последние 4–8 с записи при краше приложения или OOM-килле системы. | Потеря хвоста полёта. |
| **C-07** | История высот в `FlightStateMachine` (кольцевой буфер на 300 сэмплов) **не сбрасывается** при переходе `FINISHED → ON_GROUND`. При новом полёте в буфере остаются старые значения. | Ложные срабатывания автостарта (например, при поездке на машине после приземления). |

Дополнительно в коде IGC-логгера есть **несоответствия формату IGC** (G-record, A-record, заголовки) и захардкоженные имена пилота/дельтаплана.

Все эти проблемы — ниже, с построчным разбором и патчами.

## 1. Архитектурный обзор подсистемы автостарта/автостопа

Поток данных для детекции полёта выглядит так:

```
  GpsManager (FusedLocation, 1 Гц)
        │
        ▼
  MainActivity.bgTask (10 Гц, главный looper!)
        │
        ├──► processSample()
        │       │
        │       └──► FlightStateMachine.update(alt, now)   ← ВЫЗЫВАЕТСЯ
        │              │
        │              ├──► checkStart() / checkStop()      (altitude-based)
        │              └──► listener.onFlightStarted/Finished
        │                          │
        │                          ▼
        │                  IgcLogger.startLogging / stopLogging
        │                  LogManager.startLogging / stopLogging
        │
        └──► (НИКТО НЕ ВЫЗЫВАЕТ) ──► FlightStateMachine.updateSpeedBased(...)
                                       (XCSoar-style детектор — мёртвый код)
```

Подсистема логирования:
```
  SensorController.onLinearAccelSample() (50 Гц, sensor thread)
        │
        └──► LogManager.recordSample()           ← 50 Гц, накапливает в StringBuilder
                                                 (sync на bufferLock)

  MainActivity render loop (30 Гц, UI thread)
        │
        ├──► igcLogger.updateGps(...)             ← 30 Гц копирует GPS в snapshot
        ├──► igcLogger.recordSample()             ← децимация до 1 Гц внутри
        └──► logManager.updateGpsCache(...)       ← 30 Гц, 7 volatile-полей
```

**Уже на этом уровне видны три архитектурные проблемы:**

1. **`bgHandler` использует `Looper.getMainLooper()`** (`MainActivity.java:243`). Все «фоновые» операции (FSM, ThermalLocator, WindDrift, circlingManager) бегут на UI-треде. На слабых устройствах это ANR.
2. **Рендер-луп на 30 Гц делает GPS-копирование и вызовы в логгеры** — снова на UI-треде.
3. **Нет единой точки управления состоянием полёта.** Состояние размазано по `FlightStateMachine`, `logManager.isLogging()`, `igcLogger.isLogging()`, и они могут рассинхронизироваться (см. C-03).

## 2. КРИТИЧНО — Построчный разбор автостарта/автостопа

### 2.1. C-01: `updateSpeedBased()` — мёртвый код

**Файл:** `app/src/main/java/com/termo1/radar/flight/FlightStateMachine.java:138-185`

Сам детектор написан грамотно (гистерезис 5.0 / 2.5 м/с, таймеры 10 с / 30 с, синхронизация на `stateLock`). Вот его сигнатура:

```java
public void updateSpeedBased(float gpsSpeed, double gpsLat, double gpsLon,
                             float altitude, long nowMs) {
    synchronized (stateLock) {
        if (state.get() == STATE_FINISHED) return;

        if (gpsSpeed >= TAKEOFF_SPEED_MS) {
            // Moving — увеличиваем moving_clock, сбрасываем stationary
            if (!movingClockActive) {
                movingSinceMs = nowMs;
                movingClockActive = true;
            }
            stationaryClockActive = false;
            stationarySinceMs = 0;

            // Если не летим, но движемся 10с → взлёт
            if (state.get() == STATE_ON_GROUND && movingClockActive
                    && (nowMs - movingSinceMs) >= TAKEOFF_CONFIRM_MS) {
                Log.i(TAG, "Flight START (speed-based): speed=" + gpsSpeed
                        + " m/s for " + TAKEOFF_CONFIRM_MS / 1000 + "s");
                state.set(STATE_FLYING);
                if (listener != null) listener.onFlightStarted();
            }
        } else if (gpsSpeed < LANDING_SPEED_MS && gpsSpeed >= 0) {
            // ... симметрично для посадки
        } else {
            // ГИСТЕРЕЗИС (BUG-09): 2.5–5.0 м/с — неопределённо,
            // сохраняем текущее состояние, таймеры НЕ сбрасываем
        }
    }
}
```

**Проблема.** Поиск по всей кодовой базе показывает, что этот метод **никогда не вызывается**:

```
$ rg "updateSpeedBased" app/src/main/java/
flight/FlightStateMachine.java:19:  * чтобы update() и updateSpeedBased() не конфликтовали.
flight/FlightStateMachine.java:26:  // Lock для синхронизации update() и updateSpeedBased()
flight/FlightStateMachine.java:138: public void updateSpeedBased(float gpsSpeed, ...
```

Только упоминание в Javadoc и собственно декларация. В `MainActivity` он не подключён ни в `processSample()`, ни в `bgTask`, ни в render loop. Автор написал детектор «как в XCSoar», но забыл его включить.

**Влияние.** Для параплана скорость — единственный надёжный признак взлёта/посадки, потому что:
- взлёт часто начинается с бега по склону при нулевом наборе высоты (alt-delta < 10 м за 10 с — не сработает);
- посадка часто происходит на той же высоте, что и старт (особенно при тренировочных полётах на учебной горе);
- в слабую термичку высота может колебаться ±5 м на старте, давая ложные срабатывания `checkStart()`.

Без speed-based детектора **автостарт либо опаздывает на 30–60 с, либо не наступает вообще**, а автостоп в типовом сценарии «приземлился там же, откуда взлетел» не сработает никогда.

**Фикс.** Добавить вызов в `bgTask` (который ходит с частотой 10 Гц — даже избыточно, можно децимировать до 1 Гц):

```java
// MainActivity.java, внутри bgTask.run(), сразу после processSample()
long bgNow = SystemClock.elapsedRealtime();
if (!testMode && !simMode && !scenarioMode && !trackMode
        && gpsManager.isReady()
        && gpsManager.getFixAgeMs() < 5000) {
    flightStateMachine.updateSpeedBased(
            gpsManager.getSpeed(),
            gpsManager.getLat(),
            gpsManager.getLon(),
            gpsManager.isAltitudeInitialized()
                ? gpsManager.getAltitude()
                : sensorController.getAltitudeRaw(),
            bgNow);
}
```

**Дополнительно — несимметричное поведение состояний.** В `update()`:

```java
// FlightStateMachine.java:95-121
public void update(float altitudeMsl, long nowMs) {
    synchronized (stateLock) {
        if (state.get() == STATE_FINISHED) {
            state.set(STATE_ON_GROUND);   // ← авто-сброс
            lastCheckMs = 0;
        }
        ...
    }
}
```

А в `updateSpeedBased()`:

```java
// FlightStateMachine.java:138-141
public void updateSpeedBased(float gpsSpeed, ...) {
    synchronized (stateLock) {
        if (state.get() == STATE_FINISHED) return;   // ← НЕ сбрасывает!
        ...
    }
}
```

Если `updateSpeedBased()` всё-таки подключат, асимметрия приведёт к тому, что после посадки FSM останется в `STATE_FINISHED` для speed-based, но `STATE_ON_GROUND` для altitude-based. Это создаст гонку состояний. **Нужно сделать симметрично:**

```java
public void updateSpeedBased(float gpsSpeed, double gpsLat, double gpsLon,
                             float altitude, long nowMs) {
    synchronized (stateLock) {
        if (state.get() == STATE_FINISHED) {
            state.set(STATE_ON_GROUND);
            // History не трогаем — это ответственность update()
        }
        ...
    }
}
```

Ещё лучше — вынести авто-сброс `FINISHED → ON_GROUND` в отдельный метод `tickStateTransition(nowMs)` и вызывать его из обоих апдейторов первым делом.

### 2.2. C-02: `onPause()` убивает сенсоры и GPS до того, как сработает автостарт

**Файл:** `app/src/main/java/com/termo1/radar/MainActivity.java:527-541`

```java
@Override
protected void onPause() {
    super.onPause();
    running = false;
    stopRendering();
    // При активной записи лога — не убиваем сенсоры/GPS/ваклок
    // чтобы прибор продолжал работать при выключенном экране
    if (!logManager.isLogging()) {                  // ← BUG
        sensorController.unregisterSensors();
        releaseWakeLock();
        gpsManager.stopGps();
        if (varioSoundManager != null) varioSoundManager.stop();
    }
    stopTestMode();
}
```

**Что здесь происходит.** Условие `!logManager.isLogging()` означает: «если запись ещё не началась — выключить всё». Это **ровно тот момент, когда автостарт нужен больше всего** — пилот положил телефон в кокпит, выключил экран (батарея), разбежался и взлетел. Экран погас → `onPause()` → сенсоры и GPS выключены → `FlightStateMachine.update()` продолжает получать вызовы из `bgTask`, но:

- `gpsManager.isReady()` = `false` после `stopGps()`;
- `gpsManager.getAltitude()` возвращает последнее известное значение (замороженное);
- `sensorController.getAltitudeRaw()` возвращает последнее баро-значение (тоже замороженное, потому что сенсоры не регистрированы);
- `FlightStateMachine.update()` пишет в историю одну и ту же высоту → `checkStart()` не видит `delta >= 10m` → автостарт не наступает.

Через 30 минут пилот приземляется, включает экран, видит: «лог не начат». Это и есть наблюдаемый пользователем симптом.

**Дополнительная проблема.** Даже если автостарт не нужен — датчики и GPS выключаются в `onPause`, а потом включаются в `onResume`. Если пользователь просто переключается между активити (например, открыл Settings), происходит полный цикл `unregister → register` сенсоров. У `SensorManager.registerListener` есть существенный overhead (особенно для `TYPE_ROTATION_VECTOR`, который стартует встроенный fusion ядра). Это также сбрасывает калибровку `VarioManager` (см. §2.5).

**Фикс.** Решение зависит от того, что считать «активным режимом»:

```java
// Вариант 1 (минимальный): всегда держим сенсоры/GPS, пока приложение запущено.
// Условие полностью убрать.
@Override
protected void onPause() {
    super.onPause();
    running = false;
    stopRendering();
    // Сенсоры/GPS/WakeLock НЕ трогаем — ThermalRadarService держит foreground.
    // Они выключаются только в onDestroy() (или по явному действию пользователя).
    stopTestMode();
}

// Вариант 2 (экономнее): держим всё, пока приложение в foreground service.
// Это уже фактически сделано через ThermalRadarService, просто убираем дублирование.
```

**Важно.** Вместе с этим фиксом нужно гарантировать, что `ThermalRadarService` действительно держит WakeLock и foreground. Сейчас он это делает (`ThermalRadarService.java:50-56`), но только если сервис запущен. `startForegroundService(...)` вызывается в `onCreate()` (строка 450), так что сервис живёт всё время жизни Activity. Если Activity убита системой — сервис остаётся, но теряет связь с датчиками (потому что `SensorManager` привязан к context Activity, а не к Service). Это отдельная архитектурная проблема — см. §4.

### 2.3. C-03: В `confirmStopLogging()` забыт `igcLogger.stopLogging()`

**Файл:** `app/src/main/java/com/termo1/radar/MainActivity.java:949-962`

```java
public void confirmStopLogging() {
    if (!logManager.isLogging()) return;
    new AlertDialog.Builder(this)
            .setTitle("Остановить запись?")
            .setMessage("Лог будет сохранён. Вы уверены?")
            .setPositiveButton("Да", (d, w) -> {
                flightStateMachine.reset();
                logManager.stopLogging();
                // ← ЗДЕСЬ НЕТ igcLogger.stopLogging() !!!
                android.widget.Toast.makeText(MainActivity.this,
                        "Лог сохранён", android.widget.Toast.LENGTH_SHORT).show();
            })
            .setNegativeButton("Нет", null)
            .show();
}
```

**Что происходит.** При ручном стопе:
- `flightStateMachine.reset()` сбрасывает FSM в `ON_GROUND` (нормально).
- `logManager.stopLogging()` флушит sensor ZIP (нормально).
- `igcLogger` остаётся в состоянии `logging = true`, `currentOut` открыт.

Файл IGC остаётся открытым до `onDestroy()` (строка 552), где вызывается `igcLogger.destroy()` → `stopLogging()` → `closeCurrentFile()` с G-record. Но:

1. Если пользователь не закрывает приложение, а просто начинает новый полёт — `onFlightStarted()` вызовет `igcLogger.startLogging()`, который увидит `logging == true` и **молча вернётся** (`if (logging) return;`, IgcLogger.java:161). Новый полёт пишется в **старый** файл.
2. Если система убьёт процесс (low memory) — `onDestroy` не вызовется, IGC-файл останется без G-record и без flush. XCSoar выдаст «File too short» или «G-record invalid».
3. Имя файла (`baseFileName`) обновится через `setBaseFileName()`, но текущий открытый файл это не переименует.

Сравним с автоматическим стопом (`flightListener.onFlightFinished()`, строки 807-813):

```java
@Override
public void onFlightFinished() {
    igcLogger.stopLogging();        // ← корректно
    logManager.stopLogging();
    android.widget.Toast.makeText(MainActivity.this,
            "Полёт завершён, лог сохранён", android.widget.Toast.LENGTH_SHORT).show();
}
```

Тут всё правильно. Просто при ручном стопе про IGC забыли.

**Фикс:**

```java
.setPositiveButton("Да", (d, w) -> {
    flightStateMachine.reset();
    igcLogger.stopLogging();        // ← ДОБАВИТЬ
    logManager.stopLogging();
    android.widget.Toast.makeText(MainActivity.this,
            "Лог сохранён", android.widget.Toast.LENGTH_SHORT).show();
})
```

**Дополнительно — defensive programming.** В `IgcLogger.startLogging()` стоит сделать проверку, что предыдущий файл закрыт, и если нет — закрыть его:

```java
public void startLogging() {
    if (logging) {
        // Защита от рассинхронизации: закрываем предыдущий файл
        Log.w(TAG, "startLogging() called while already logging — closing previous");
        stopLogging();
    }
    ...
}
```

А лучше — вынести управление старт/стоп в единый `FlightRecorder` фасад, который атомарно управляет обоими логгерами (см. §4).

### 2.4. C-04: Файловый I/O в UI-треде при `onFlightStarted()`

**Файлы:** `MainActivity.java:243`, `MainActivity.java:788-805`, `IgcLogger.java:160-196`

```java
// MainActivity.java:243
private Handler bgHandler = new Handler(Looper.getMainLooper());  // ← UI thread!
```

```java
// MainActivity.java:788-805
private final FlightStateMachine.FlightStateListener flightListener =
        new FlightStateMachine.FlightStateListener() {
    @Override
    public void onFlightStarted() {
        java.text.SimpleDateFormat sdf = new java.text.SimpleDateFormat("yyyyMMdd_HHmmss", java.util.Locale.US);
        String baseName = "Flight_" + sdf.format(new java.util.Date());
        igcLogger.setBaseFileName(baseName);
        logManager.setBaseFileName(baseName);
        logManager.startLogging();
        igcLogger.startLogging();          // ← ВЫЗЫВАЕТСЯ НА UI-ТРЕДЕ
        resetCalibration();
        sensorController.calibrateHeading();
        ...
    }
    ...
};
```

```java
// IgcLogger.java:160-196
public void startLogging() {
    if (logging) return;
    if (logDir == null) { ... return; }

    logging = true;
    startElapsedMs = SystemClock.elapsedRealtime();
    ...

    File dir = new File(logDir, "igc");
    if (!dir.exists()) dir.mkdirs();        // ← disk I/O

    ...
    File file = new File(dir, currentFileName);

    try {
        currentOut = new BufferedOutputStream(new FileOutputStream(file));  // ← disk I/O
        writeHeader();                       // ← disk I/O (8 строк)
        headerWritten = true;
        ...
    } catch (IOException e) {
        ...
    }
}
```

**Проблема.** `bgHandler` привязан к `Looper.getMainLooper()`. Все `bgTask` runs, включая `processSample()` → `flightStateMachine.update()` → `listener.onFlightStarted()`, выполняются синхронно в UI-треде. Внутри `onFlightStarted()`:

1. `new FileOutputStream(file)` — системный вызов `open()`, на зашифрованном внутреннем хранилище может занимать 20–80 мс.
2. `dir.mkdirs()` — если директория не существует, `stat()` + `mkdir()` подряд.
3. `writeHeader()` — 8 строк по `currentOut.write(...)` через `BufferedOutputStream`. Запись в буфер, не на диск, но если буфер с флашем — то на диск.
4. `logManager.startLogging()` — инициализация `StringBuilder(65536)`. Это аллокация ~65 КБ, что для GC pause в UI-треде заметно.
5. `resetCalibration()` → `varioManager.reset()` — быстро.
6. `sensorController.calibrateHeading()` — запускает накопление сэмплов, не блокирует.

Суммарно — 50–200 мс блокировки UI на медленном устройстве. На 30 Гц рендере это 5–6 пропущенных кадров. На廉价 китайских планшетах, которыми пользуются парапланеристы (типичный кейс) — может быть и ANR.

**Фикс.** Перенести файловый I/O на фоновый HandlerThread. У `LogManager` уже есть свой `HandlerThread` (`ioThread`), нужно сделать то же для `IgcLogger`:

```java
// IgcLogger.java
private final HandlerThread ioThread;
private final Handler ioHandler;

public IgcLogger() {
    ioThread = new HandlerThread("termo1-igc-io");
    ioThread.start();
    ioHandler = new Handler(ioThread.getLooper());
}

public synchronized void startLogging() {
    if (logging) return;
    logging = true;
    // Снимок параметров для фонового треда
    final String dir = logDir;
    final String name = (baseFileName != null) ? baseFileName + ".igc"
            : "Flight_" + FILE_DATE_FMT_TL.get().format(new Date()) + ".igc";
    final long startMs = SystemClock.elapsedRealtime();

    ioHandler.post(() -> {
        startElapsedMs = startMs;
        seqNum = 0;
        lastLogElapsedMs = 0;
        headerWritten = false;
        crc32 = new CRC32();
        File dirFile = new File(dir, "igc");
        if (!dirFile.exists()) dirFile.mkdirs();
        currentFileName = name;
        File file = new File(dirFile, name);
        try {
            currentOut = new BufferedOutputStream(new FileOutputStream(file));
            writeHeader();
            headerWritten = true;
            Log.i(TAG, "IGC logging STARTED: " + name);
        } catch (IOException e) {
            Log.e(TAG, "Failed to create IGC file: " + name, e);
            currentOut = null;
            logging = false;   // откатываем флаг
        }
    });
}
```

### 2.5. C-05: `recordSample()` молча пропускает B-records при потере GPS

**Файл:** `app/src/main/java/com/termo1/radar/logging/IgcLogger.java:216-240`

```java

    if (!logging) return;

    long elapsedNow = SystemClock.elapsedRealtime();
    long elapsed = elapsedNow - startElapsedMs;

    // 1 Гц децимация
    if (elapsed - lastLogElapsedMs < LOG_INTERVAL_MS) return;
    lastLogElapsedMs = elapsed;

    // Проверка валидности GPS — атомарное чтение снимка
    GpsSnapshot gps = gpsSnapshot;
    if (gps == null) return;                              // ← silent skip
    if (gps.lat == 0.0 && gps.lon == 0.0) return;         // ← silent skip
    if (gps.fixAgeMs > 5000) return;                      // ← silent skip

    seqNum++;

    // Формируем B-record
    String bRecord = formatBRecord(
            elapsed, gps.lat, gps.lon,
            gps.altBaro, gps.altGps, gps.fixAgeMs, gps.accuracy);

}
```

**Проблема.** IGC-стандарт требует, чтобы B-records шли с равномерным шагом 1 Гц (или иным, заявленным в H-записи). Любой пропуск времени воспринимается валидаторами (XCSoar, SeeYou, flight analysis tools) как разрыв лога. Что произойдёт:

1. Пилот летит, GPS ловит в ущелье и фикс теряется на 20 секунд.
2. В эти 20 секунд `recordSample()` пропускает 20 B-records.
3. В файле образуется дыра: `B110215...`, `B110216...`, [пропуск 20 секунд], `B110236...`.
4. XCSoar при открытии файла выдаст предупреждение и в худшем случае откажется считать трек.
5. FAI-валидация (для спортивных доказательств) точно забракует файл.

**Дополнительно.** Проверка `gps.lat == 0.0 && gps.lon == 0.0` — это «magic value» check. На старте GPS модуль действительно может прислать (0, 0) до первого фикса (это координаты пересечения экватора и гринвичского меридиана, в Гвинейском заливе). Но после первого фикса `gpsSnapshot` уже никогда не будет (0, 0). Лучше использовать отдельный boolean-флаг `hasValidFix` в самом снапшоте.

**Фикс.** Вместо пропуска — писать B-record с признаком `V` (no fix) и последними известными координатами:


### 2.6. C-06: Нет периодического flush/fsync — потеря хвоста при краше

**Файл:** `app/src/main/java/com/termo1/radar/logging/IgcLogger.java:74-87, 369-381`

```java
// Состояние
private BufferedOutputStream currentOut;
private CRC32 crc32;
```

```java
// writeLine — пишет только в буфер
private void writeLine(String line) {
    if (currentOut == null) return;
    try {
        byte[] bytes = (line + "\r\n").getBytes("ISO-8859-1");
        currentOut.write(bytes);
        if (crc32 != null) {
            crc32.update(bytes);
        }
    } catch (IOException e) {
        Log.e(TAG, "IGC write error", e);
    }
}
```

`BufferedOutputStream` по умолчанию имеет буфер 8192 байт. Один B-record — ~37 байт. Буфер заполняется за ~220 записей = ~3,7 минуты полёта. До этого момента данные физически не на диске.

**Сценарий потери данных:**
1. Полёт 2 часа. Пилот разбился (или приложение убито OOM-киллером).
2. `onDestroy()` не вызван.
3. В буфере сидят последние 3,7 минуты полёта.
4. ОС закрывает файловые дескрипторы приkill процесса, но **буфер `BufferedOutputStream` остаётся в heap-памяти и теряется**.
5. На диске — файл с обрезанным хвостом, без G-record.

Для парапланерного логгера это **критично**: если пилот попал в аварийную ситуацию и лог нужен для разбора инцидента, последние минуты (самые важные) потеряны.

**Фикс.** Добавить периодический flush + fsync:

```java
// IgcLogger.java — добавить поле
private long lastFlushMs;
private static final long FLUSH_INTERVAL_MS = 5000;  // 5 секунд

private void writeLine(String line) {
    if (currentOut == null) return;
    try {
        byte[] bytes = (line + "\r\n").getBytes("ISO-8859-1");
        currentOut.write(bytes);
        if (crc32 != null) crc32.update(bytes);

        // Периодический flush
        long now = SystemClock.elapsedRealtime();
        if (now - lastFlushMs > FLUSH_INTERVAL_MS) {
            currentOut.flush();
            lastFlushMs = now;
        }
    } catch (IOException e) {
        Log.e(TAG, "IGC write error", e);
    }
}
```

Лучше — добавить ещё и `FileDescriptor.sync()` через `FileOutputStream.getFD().sync()`:

```java
private void writeLine(String line) {
    if (currentOut == null) return;
    try {
        byte[] bytes = (line + "\r\n").getBytes("ISO-8859-1");
        currentOut.write(bytes);
        if (crc32 != null) crc32.update(bytes);

        long now = SystemClock.elapsedRealtime();
        if (now - lastFlushMs > FLUSH_INTERVAL_MS) {
            currentOut.flush();
            // fsync — гарантирует, что данные ушли на накопитель
            if (currentOut.getClass().isAssignableFrom(BufferedOutputStream.class)) {
                try {
                    FileOutputStream fos = (FileOutputStream)
                            ((BufferedOutputStream) currentOut).getClass()
                                    .getDeclaredField("out").get(currentOut);
                    if (fos != null) fos.getFD().sync();
                } catch (Exception ignored) { /* best effort */ }
            }
            lastFlushMs = now;
        }
    } catch (IOException e) {
        Log.e(TAG, "IGC write error", e);
    }
}
```

Проще — сохранить ссылку на `FileOutputStream` отдельно:

```java
private FileOutputStream currentFos;
private BufferedOutputStream currentOut;

// в startLogging:
currentFos = new FileOutputStream(file);
currentOut = new BufferedOutputStream(currentFos);

// в writeLine, при flush:
currentOut.flush();
currentFos.getFD().sync();
```

Цена fsync — 5–50 мс (зависит от накопителя). Раз в 5 секунд — незаметно.

7. Включается запись лога для «полёта», который на самом деле — поездка на машине.

```java
            stationaryClockActive = false;
            movingSinceMs = 0;
            stationarySinceMs = 0;
```

## 3. ВАЖНО — Построчный разбор `FlightStateMachine`

### 3.1. Алгоритм altitude-based: корректный, но с численными артефактами

**Файл:** `flight/FlightStateMachine.java:194-213`

```java
private void checkStart(long nowMs) {
    if (histFill < 2) return;

    float altNow = altitudeHistory[(histHead - 1 + ALT_HISTORY_SIZE) % ALT_HISTORY_SIZE];
    long cutoff = nowMs - START_WINDOW_MS;

    for (int i = 0; i < histFill; i++) {
        int idx = (histHead - 1 - i + ALT_HISTORY_SIZE) % ALT_HISTORY_SIZE;
        if (altitudeTimes[idx] <= cutoff) {
            float delta = altNow - altitudeHistory[idx];
            if (delta >= START_ALTITUDE_DELTA) {
                Log.i(TAG, "Flight START (altitude): +" + delta
                        + "m in " + (nowMs - altitudeTimes[idx]) / 1000 + "s");
                state.set(STATE_FLYING);
                if (listener != null) listener.onFlightStarted();
            }
            return;
        }
    }
}
```

**Логика корректная:** ищем первую точку старше 10 секунд, сравниваем с текущей. Если дельта ≥ 10 м — взлёт.

**Проблема 1 — `return` пропускает остальные точки.** Если первая точка старше 10 с не даёт `delta >= 10m`, мы выходим, не проверяя более старые точки. В типовом сценарии это нормально (мы хотим дельту за последние 10 с, не за 30), но если в истории есть провал (например, сэмплер пропустил 20 с, и точка старше 10 с — это точка 25 с назад), то проверка делается по неоптимальной точке.

**Проблема 2 — поиск первой точки ≤ cutoff.** Линейный `O(N)` проход по 300 элементам на каждом тике. Для 1 Гц обновления это 300 операций в секунду — копейки. Но если перейдут на более частый update (например, 10 Гц), будет 3000 оп/с. Лучше — бинарный поиск по отсортированному кольцевому буферу (или просто хранить индекс первой точки старше cutoff с прошлого тика — он монотонно движется вперёд).

**Проблема 3 — `altNow` берётся из `histHead - 1`.** Это последняя записанная точка. Но что если `update()` вызывается с тем же `nowMs` дважды (например, при коллизии тайминга)? Тогда `altNow` может быть устаревшим на 1 с. Лучше — брать `altitudeMsl` напрямую из аргумента метода.

**Фикс:**

```java
private void checkStart(float altitudeMsl, long nowMs) {
    if (histFill < 2) return;

    long cutoff = nowMs - START_WINDOW_MS;

    for (int i = 0; i < histFill; i++) {
        int idx = (histHead - 1 - i + ALT_HISTORY_SIZE) % ALT_HISTORY_SIZE;
        if (altitudeTimes[idx] <= cutoff) {
            float delta = altitudeMsl - altitudeHistory[idx];
            if (delta >= START_ALTITUDE_DELTA) {
                Log.i(TAG, "Flight START (altitude): +" + delta
                        + "m in " + (nowMs - altitudeTimes[idx]) / 1000 + "s");
                state.set(STATE_FLYING);
                if (listener != null) listener.onFlightStarted();
            }
            return;
        }
    }
}
```

### 3.2. `checkStop` — логика почти корректная, но `STOP_ALTITUDE_DELTA * 2` — магия

**Файл:** `flight/FlightStateMachine.java:219-253`

```java
private void checkStop(long nowMs) {
    if (histFill < 2) return;

    float altNow = altitudeHistory[(histHead - 1 + ALT_HISTORY_SIZE) % ALT_HISTORY_SIZE];
    long cutoff = nowMs - STOP_WINDOW_MS;

    float minAlt = altNow;
    float maxAlt = altNow;

    for (int i = 0; i < histFill; i++) {
        int idx = (histHead - 1 - i + ALT_HISTORY_SIZE) % ALT_HISTORY_SIZE;
        if (altitudeTimes[idx] < cutoff) break;
        float a = altitudeHistory[idx];
        if (a < minAlt) minAlt = a;
        if (a > maxAlt) maxAlt = a;
    }

    float delta = maxAlt - minAlt;
    if (delta <= STOP_ALTITUDE_DELTA * 2) {   // ← ПОЧЕМУ * 2 ?
        // BUG-08: полное окно = есть точка старше cutoff
        boolean fullWindow = false;
        for (int i = 0; i < histFill; i++) {
            int idx = (histHead - 1 - i + ALT_HISTORY_SIZE) % ALT_HISTORY_SIZE;
            if (altitudeTimes[idx] <= cutoff && altitudeTimes[idx] > 0) {
                fullWindow = true;
                break;
            }
        }
        if (fullWindow) {
            Log.i(TAG, "Flight FINISH (altitude): delta=" + delta + "m over 5min");
            state.set(STATE_FINISHED);
            if (listener != null) listener.onFlightFinished();
        }
    }
}
```

**Магия `STOP_ALTITUDE_DELTA * 2`.** `STOP_ALTITUDE_DELTA = 2.0f`, а в условии — `delta <= 4.0f`. Это, видимо, означает «амплитуда колебаний ≤ ±2 м вокруг среднего», но в коде это не задокументировано. Нужно либо переименовать константу в `STOP_ALTITUDE_AMPLITUDE = 4.0f`, либо убрать `*2` и использовать `delta <= STOP_ALTITUDE_DELTA` с `STOP_ALTITUDE_DELTA = 4.0f`.

**Повторный проход по истории.** Первый цикл считает `minAlt`/`maxAlt`. Второй цикл проверяет `fullWindow`. Это можно объединить:

```java
private void checkStop(float altitudeMsl, long nowMs) {
    if (histFill < 2) return;

    long cutoff = nowMs - STOP_WINDOW_MS;

    float minAlt = altitudeMsl;
    float maxAlt = altitudeMsl;
    boolean fullWindow = false;

    for (int i = 0; i < histFill; i++) {
        int idx = (histHead - 1 - i + ALT_HISTORY_SIZE) % ALT_HISTORY_SIZE;
        long t = altitudeTimes[idx];
        if (t < cutoff) {
            // Точка за пределами окна — но если t > 0, значит, окно полное
            if (t > 0) fullWindow = true;
            break;
        }
        // Точка внутри окна
        float a = altitudeHistory[idx];
        if (a < minAlt) minAlt = a;
        if (a > maxAlt) maxAlt = a;
    }

    float delta = maxAlt - minAlt;
    if (fullWindow && delta <= STOP_ALTITUDE_AMPLITUDE) {
        Log.i(TAG, "Flight FINISH (altitude): delta=" + delta + "m over 5min");
        state.set(STATE_FINISHED);
        if (listener != null) listener.onFlightFinished();
    }
}
```

**Фундаментальная проблема алгоритма stop-детекции.** Требование «5 минут в коридоре ±2 м» работает только если пилот приземлился на равнине рядом с точкой старта. Для параплана типичны сценарии:

1. **Приземление на другой высоте** (горный полёт): пилот стартовал с 1500 м, приземлился на 800 м. Высота «устаканилась» на 800 м, но `altNow` = 800, история содержит значения от 1500 до 800. `maxAlt - minAlt` будет 700+ м. `delta <= 4` никогда не выполнится → **автостоп не сработает**.
2. **Приземление в турбулентном потоке** (у склона): даже на земле баро-высота колеблется ±5 м от порывов ветра. `delta` > 4 → **автостоп не сработает**.
3. **Тренировочный полёт на учебной горе**: 5 минут недостаточно — пилот может стоять на старте после приземления, потом снова взлетать.

Для параплана единственный надёжный признак посадки — **скорость < 2,5 м/с в течение 30 секунд** (что и реализовано в мёртвом `updateSpeedBased()`). Алгоритм altitude-based стоит рассматривать только как резервный.

### 3.3. Состояние `STATE_FINISHED` — избыточно

```java
public static final int STATE_ON_GROUND = 0;
public static final int STATE_FLYING = 1;
public static final int STATE_FINISHED = 2;
```

`STATE_FINISHED` живёт только до следующего `update()`, где сразу сбрасывается в `ON_GROUND`. Зачем он нужен? Только чтобы `updateSpeedBased()` мог `return` (см. строку 141). Если убрать `STATE_FINISHED` и сразу после `onFlightFinished()` переходить в `ON_GROUND`, поведение не изменится, а логика станет проще.

Альтернативно — `STATE_FINISHED` можно держать как hysteresis: «приземлились, но 30 минут не стартуем новый полёт, чтобы избежать ложных срабатываний от езды на машине». Тогда нужно добавить таймер и явный переход. Сейчас этого нет.

### 4.2. G-record — некорректный формат

**Файл:** `IgcLogger.java:348-367`

```java
private void closeCurrentFile() {
    if (currentOut == null) return;

    try {
        long crcValue = crc32.getValue();
        writeLine("G" + String.format("%08X", crcValue & 0xFFFFFFFFL));   // ← НЕПРАВИЛЬНО

        currentOut.flush();
        currentOut.close();
        ...
    } catch (IOException e) {
        Log.e(TAG, "Failed to close IGC file", e);
    } finally {
        currentOut = null;
        currentFileName = null;
    }
}
```

Стандарт IGC требует:
- G-record состоит из **4 строк по 16 hex-символов** (всего 64 hex = 256 бит).
- Алгоритм — не простой CRC32, а специфический IGC-CRC (полином 0x1021, инициализация 0xFFFF, no reflection).
- Это позволяет валидаторам проверить целостность всего файла.

Текущий код пишет одну строку из 8 hex-символов с CRC32. Большинство валидаторов это отклонит или пометит файл как «G-record invalid».

**Фикс.** Реализовать IGC-CRC и 4-строчный формат:

```java
// IGC-CRC (полином 0x1021, init 0xFFFF, no reflect)
private int igcCrc = 0xFFFF;

private void updateIgcCrc(byte[] bytes) {
    for (byte b : bytes) {
        int v = b & 0xFF;
        igcCrc ^= (v << 8);
        for (int i = 0; i < 8; i++) {
            if ((igcCrc & 0x8000) != 0) {
                igcCrc = (igcCrc << 1) ^ 0x1021;
            } else {
                igcCrc <<= 1;
            }
            igcCrc &= 0xFFFF;
        }
    }
}

// 4 строки по 16 hex
private void writeGRecord() throws IOException {
    // Дополняем до 256 бит — padding нулями (или специальной схемой IGC)
    String crc16Hex = String.format("%04X", igcCrc);
    // IGC использует многократное дублирование CRC + padding для 4 строк
    String line1 = "G" + repeat(crc16Hex, 4);   // 16 символов
    String line2 = "G" + repeat(crc16Hex, 4);
    String line3 = "G" + repeat(crc16Hex, 4);
    String line4 = "G" + repeat(crc16Hex, 4);
    // Не включаем G-записи в самоконтроль CRC!
    writeLine(line1);
    writeLine(line2);
    writeLine(line3);
    writeLine(line4);
}
```

(Точная схема padding'а для G-record описана в IGC GNSS spec, приложение B — там могут быть нюансы для конкретного валидатора.)

Если цель — просто «читается в XCSoar», текущая реализация проходит. Но для официальных полётов — нет.

### 4.3. `formatIGCTime` — смешение wall-clock и elapsedRealtime

**Файл:** `IgcLogger.java:300-305`

```java
private String formatIGCTime(long elapsedMs) {
    long wallMs = System.currentTimeMillis()
            - (SystemClock.elapsedRealtime() - startElapsedMs) + elapsedMs;
    Date d = new Date(wallMs);
    return IGC_TIME_FMT_TL.get().format(d);
}
```

Идея: запомнить `wallClockAtStart = System.currentTimeMillis() - SystemClock.elapsedRealtime() + startElapsedMs` и потом прибавлять `elapsedMs`. Это правильно — `elapsedRealtime` не зависит от NTP-скачков. Но формула вычисляется на каждом вызове заново, что означает:

1. Если `SystemClock.elapsedRealtime()` ушёл вперёд (он монотонный, но при засыпании устройства может стоять на месте, а потом скакнуть), формула ломается.
2. Если `System.currentTimeMillis()` шагнул из-за NTP-синхронизации, IGC-время тоже шагнёт.

**Фикс.** Запомнить `wallStartMs` один раз при старте:

```java
// Поля
private long wallStartMs;

// В startLogging
startElapsedMs = SystemClock.elapsedRealtime();
wallStartMs = System.currentTimeMillis();

// В formatIGCTime
private String formatIGCTime(long elapsedMs) {
    long wallMs = wallStartMs + elapsedMs;
    return IGC_TIME_FMT_TL.get().format(new Date(wallMs));
}
```

Это даёт монотонное IGC-время, привязанное к моменту старта, не зависящее от сбоев NTP.

### 4.4. `ThreadLocal<SimpleDateFormat>` — over-engineering

**Файл:** `IgcLogger.java:41-61`

```java
private static final ThreadLocal<SimpleDateFormat> IGC_TIME_FMT_TL =
        ThreadLocal.withInitial(() -> {
            SimpleDateFormat f = new SimpleDateFormat("HHmmss", Locale.US);
            f.setTimeZone(java.util.TimeZone.getTimeZone("UTC"));
            return f;
        });
```

`SimpleDateFormat` действительно не thread-safe, и `ThreadLocal` — стандартное решение. Но:

1. Все вызовы идут из одного треда (UI или одного IO-треда после §2.4 фикса).
2. `SimpleDateFormat.format()` для строки `"HHmmss"` настолько быстр, что создание нового объекта на каждый вызов дешевле, чем поддержка ThreadLocal.

Лучше — `DateTimeFormatter` (Java 8+ thread-safe, immutable):

```java

private String formatIGCTime(long elapsedMs) {
    long wallMs = wallStartMs + elapsedMs;
    return IGC_TIME_FMT.format(Instant.ofEpochMilli(wallMs));
}
```

Не критично, но чище.

### 5.1. Неограниченный `StringBuilder` — OOM на длинных полётах

**Файл:** `LogManager.java:84, 145-181`

```java

    logBuffer = new StringBuilder(65536);
    ...
}

public void recordSample() {
    if (!isLogging || dataProvider == null) return;
    sampleCounter++;
    if (sampleCounter % SAMPLE_DECIMATION != 0) return;

    long elapsedNow = SystemClock.elapsedRealtime();
    long dtMs = elapsedNow - elapsedStartMs;

    synchronized (bufferLock) {
        logBuffer.append(dtMs).append(',');
        logBuffer.append(cachedGpsSpeed).append(',');
        ...  // ~25 полей
        logBuffer.append('\n');
    }
}
```

`SAMPLE_DECIMATION = 1`, частота вызовов — 50 Гц (на каждый сэмпл `TYPE_LINEAR_ACCELERATION`). Одна строка CSV — ~150 байт. За 3 часа полёта:

```
3 * 3600 * 50 * 150 = 81 000 000 байт = 81 МБ
```

`StringBuilder` с начальным размером 64 КБ будет реаллоцироваться ~20 раз, дойдя до 128 МБ heap allocation. На устройстве с 2 ГБ ОЗУ это OOM-килл.

**Фикс.** Стримить данные на диск порциями (например, каждую минуту), а не копить всё в памяти:

```java
private static final long FLUSH_INTERVAL_MS = 60_000;  // 1 минута
private long lastFlushMs;
private File currentCsvFile;
private BufferedOutputStream currentOut;

public void startLogging() {
    ...
    // Открыть CSV-файл сразу
    File dir = new File(externalLogDir, "logs");
    if (!dir.exists()) dir.mkdirs();
    currentCsvFile = new File(dir, baseFileName + ".csv");
    try {
        currentOut = new BufferedOutputStream(new FileOutputStream(currentCsvFile));
        // Заголовок
        writeCsvHeader(currentOut);
    } catch (IOException e) {
        Log.e(TAG, "Cannot open CSV", e);
        isLogging = false;
    }
}

public void recordSample() {
    if (!isLogging || dataProvider == null) return;
    ...
    synchronized (bufferLock) {
        // Сразу пишем в поток
        try {
            currentOut.write(buildCsvLine().getBytes("UTF-8"));
        } catch (IOException e) {
            Log.e(TAG, "CSV write error", e);
        }
    }

    // Периодический flush
    long now = SystemClock.elapsedRealtime();
    if (now - lastFlushMs > FLUSH_INTERVAL_MS) {
        try { currentOut.flush(); } catch (IOException ignored) {}
        lastFlushMs = now;
    }
}

public void stopLogging() {
    ...
    synchronized (bufferLock) {
        try {
            currentOut.flush();
            currentOut.close();
        } catch (IOException e) {
            Log.e(TAG, "Cannot close CSV", e);
        }
    }
    // Затем — упаковка в ZIP на IO-треде
    ioHandler.post(() -> zipCurrentCsv());
}
```

Это убирает риск OOM и упрощает логику: ZIP создаётся из уже готового CSV-файла при стопе.

### 6.1. `lastFixMs` — wall-clock, ломается при NTP-синхронизации

**Файл:** `GpsManager.java:33, 58, 109`

```java

...
```

`System.currentTimeMillis()` может скачкообразно измениться при NTP-синхронизации. Если телефон в горах поймал сеть и обновил время на 30 секунд назад, `getFixAgeMs()` вернёт `-30000`, что приведёт к:

- В `IgcLogger.recordSample()`: `gps.fixAgeMs > 5000` — false (т.к. -30000 < 5000), и сэмпл записывается с «мёртвым» фиксом.
- В `FlightStateMachine.updateSpeedBased()`: аналогично.

**Фикс.** Использовать `SystemClock.elapsedRealtime()` (монотонное время, не зависит от NTP):

Это же правило применяется в `IgcLogger` (через `SystemClock.elapsedRealtime()`), но в `IgcLogger.formatIGCTime()` для конвертации в wall-clock снова используется `System.currentTimeMillis()` — там это OK, потому что нужно показать UTC-время.

### 6.2. `gpsAltitude` обновляется только если `gpsAccuracy < 25f`

Лучше — вообще не использовать `gpsAltitude` для FSM, а использовать `sensorController.getAltitudeRaw()` (барометр), который в 10 раз точнее по вертикали. Сейчас `processSample()` (строка 827-828) предпочитает GPS-altitude, что неверно:

Барометр даёт точность ±0.5 м, GPS — ±10 м. Для детекции взлёта/посадки барометр гораздо надёжнее. Должно быть:

```java
// Правильно:
float alt = sensorController.hasPressureReading()
        ? sensorController.getAltitudeRaw()
```

И **автокалибровка баро на старте** — `VarioManager.reset()` ставит `altRaw = 0`, что для FSM означает «высота не растёт». Нужно либо калибровать баро по GPS-высоте при первом фиксе, либо не сбрасывать баро между полётами.

```java
    android.util.Log.i("TERMO1", "Flight START: " + baseName);
    android.widget.Toast.makeText(MainActivity.this,
            "Полёт обнаружен, запись", android.widget.Toast.LENGTH_SHORT).show();
}
```

**Проблема 1.** `resetCalibration()` вызывает `varioManager.reset()`, который:
- обнуляет `altRaw = 0`, `altFiltered = 0`, `prevAltRaw = 0`;
- сбрасывает `baroCalibCount = 0`;
- далее `VarioManager` будет калиброваться заново 50 сэмплов (≈5 секунд при 10 Гц баро).

В эти 5 секунд `sensorController.getAltitudeRaw()` вернёт `0`. Если `processSample()` в эти 5 секунд выбирает источник высоты как баро (т.к. GPS может быть ещё не инициализирован на старте), FSM получает `alt = 0` в историю. Это мусор, который потом мешает `checkStart()` для следующего полёта.

**Фикс.** Не сбрасывать баро-калибровку при старте полёта. Баро должен быть откалиброван ещё на земле, до взлёта. `resetCalibration()` должен вызываться только вручную (из настроек) или при первом старте приложения.

```java
@Override
public void onFlightStarted() {
    ...
    logManager.startLogging();
    igcLogger.startLogging();
    // НЕ вызываем resetCalibration() — баро уже откалиброван с onResume()
    // Калибровка компаса — OK, она не сбрасывает баро
    sensorController.calibrateHeading();
    ...
}
```

**Проблема 2.** `Toast.makeText(...)` в callback'е, который может быть вызван из фонового треда (если когда-нибудь `bgHandler` переведут на отдельный HandlerThread, см. §2.4). `Toast` требует UI-треда.

**Фикс.**

```java
runOnUiThread(() -> android.widget.Toast.makeText(MainActivity.this,
        "Полёт обнаружен, запись", android.widget.Toast.LENGTH_SHORT).show());
```

**Проблема 3.** `SimpleDateFormat` создаётся на каждый вызов. Лучше — static final, как в `IgcLogger`.

### 7.2. `processSample()` — выбор источника высоты неправильный

**Файл:** `MainActivity.java:820-831`

```java
private void processSample() {
    if (varioSoundManager != null) {
        varioSoundManager.update(sensorController.getVario());
    }

    // Flight state machine (на основе высоты MSL)
    if (!testMode && !simMode && !scenarioMode && !trackMode) {
        float alt = gpsManager.isAltitudeInitialized()
                ? gpsManager.getAltitude() : sensorController.getAltitudeRaw();
        flightStateMachine.update(alt, SystemClock.elapsedRealtime());
    }
}
```

Как отмечено в §6.2, GPS-высота в 10–20 раз менее точна, чем баро. Должно быть наоборот — предпочтение барометру, fallback на GPS.

**Дополнительная проблема.** Если `gpsManager.isAltitudeInitialized() = true`, но GPS-фикс потерян 30 секунд назад, `getAltitude()` возвращает замороженное значение. FSM будет видеть «неменяющуюся» высоту. Лучше проверять `getFixAgeMs()`:

```java
private void processSample() {
    ...
    if (!testMode && !simMode && !scenarioMode && !trackMode) {
        float alt;
        long fixAge = gpsManager.getFixAgeMs();

        // Предпочтение барометру
        if (sensorController.hasPressureReading()
                && sensorController.getVarioManager().isCalibrated()) {
            alt = sensorController.getAltitudeRaw();
        } else if (gpsManager.isAltitudeInitialized() && fixAge < 10_000) {
            alt = gpsManager.getAltitude();
        } else {
            alt = Float.NaN;   // нет данных — не кормим FSM
        }

        if (!Float.isNaN(alt)) {
            flightStateMachine.update(alt, SystemClock.elapsedRealtime());
        }
    }
}
```

### 7.4. `startManualLogging()` — нет установки `flightStateMachine.setStateFlying()`

**Файл:** `MainActivity.java:939-947`

```java
public void startManualLogging() {
    if (logManager.isLogging()) return;
    logManager.startLogging();
    igcLogger.startLogging();
    sensorController.calibrateHeading();
    android.widget.Toast.makeText(this,
            "Запись лога начата", android.widget.Toast.LENGTH_SHORT).show();
}
```

При ручном старте FSM остаётся в `STATE_ON_GROUND`. Это значит:
1. Через 10 секунд, если баро-высота поднимется на 10 м (пилот пошёл вверх по склону), `checkStart()` сработает и вызовет `onFlightStarted()`.
2. `onFlightStarted()` вызовет `logManager.startLogging()` — но `isLogging == true` (мы уже логируем вручную), так что просто `return`.
3. Но! `setBaseFileName(baseName)` с новым именем выполнится. Если `IgcLogger` когда-нибудь будет переоткрывать файл, он использует новое имя.

Лучше — при ручном старте явно перевести FSM в `STATE_FLYING`:

```java
public void startManualLogging() {
    if (logManager.isLogging()) return;
    logManager.startLogging();
    igcLogger.startLogging();
    flightStateMachine.setStateFlying();   // ← добавить
    sensorController.calibrateHeading();
    ...
}
```

Тогда не будет двойного срабатывания и переименования файла.

### 7.5. `onResume()` — двойная инициализация

**Файл:** `MainActivity.java:457-525`

```java

    running = true;

    uiManager.setNightMode(prefs.getBoolean("night_mode", false));
```

```java
                .setFastestInterval(500L)
                .setPriority(LocationRequest.PRIORITY_HIGH_ACCURACY);
    }
    locationCallback = new LocationCallback() { ... };
}
```

### 8.1. `ThermalRadarService` — singleton без слабой ссылки

**Файл:** `ThermalRadarService.java:35-39`


Static-ссылка на Service удерживает Service от GC, если Activity пытается его остановить. Это может быть и хорошо (keep alive), но `MainActivity` никогда не вызывает `stopService()`, так что Service живёт вечно. Это **нормально для foreground service**, но нужно явно документировать.

Также: `MainActivity` хранит `private ThermalRadarService radarService;` (строка 108), которое **никогда не присваивается**. Мёртвый код.

### 8.2. `flightStateMachine.setStateFlying()` — мёртвый метод

**Файл:** `FlightStateMachine.java:272-274`

```java

    state.set(STATE_FLYING);
}
```

### 8.3. `LogManager.recordEvent()` — no-op

**Файл:** `LogManager.java:183-186`

```java
public void recordEvent(String eventType, String details) {
    // Events are currently not included in the single CSV per flight.
    // Kept as a no-op to preserve the interface contract.
}
```

`MainActivity` вызывает `logManager.recordEvent("CIRCLING_START", ...)` и т.п. в нескольких местах. Все эти вызовы — пустые. События полёта (вход в термик, смена ветра) теряются. Если CSV предназначен для пост-обработки, это значимая потеря.

**Фикс.** Добавить event-log в отдельный файл или в специальные строки CSV:

```java
public void recordEvent(String eventType, String details) {
    if (!isLogging) return;
    synchronized (bufferLock) {
        // События — отдельные строки с префиксом EVENT
        logBuffer.append("#EVENT,").append(SystemClock.elapsedRealtime() - elapsedStartMs)
                .append(',').append(eventType).append(',')
                .append(details.replace(',', ';').replace('\n', ' '))
                .append('\n');
    }
}
```

### 8.4. Захардкоженные `Locale.US` инстансы — повторяющийся код

По всему `MainActivity` встречается:


Это работает, но:

```java
static { FILE_NAME_FMT.setTimeZone(java.util.TimeZone.getTimeZone("UTC")); }
```

## 11. Заключение

Кодовая база ThermalRadar — это типичный **студенческий проект, который вырос** из прототипа в почти-production. Видно, что автор:
- читал стандарты IGC и алгоритмы XCSoar (упоминаются в комментариях и Javadoc);
- знаком с типичными проблемами Android (WakeLock, foreground service, sensor registration);
- знает о гонках и пытается их решать (`volatile`, `synchronized`, snapshot-паттерны);
- осознанно работает над багами (видно по `BUG-08`, `BUG-09` комментариям).

Но типичные «болезни роста» на месте:
1. **God Object** `MainActivity` (2881 строка) —mixed concerns.
2. **Дублирование логики** между `LogManager` и `IgcLogger` (старт/стоп, baseFileName, GPS cache).
3. **Мёртвый код** (`updateSpeedBased`, `setStateFlying`, `radarService`, no-op `recordEvent`).
4. **UI-тред перегружен** — все «фоновые» операции бегают на main looper.
5. **Нет unit-тестов** на критичные компоненты (FSM, IGC-форматирование).

Для починки «автостарт/автостоп IGC» достаточно сделать **пункты P0 (1–5)** из §9.1 — это около 50 строк кода. Остальные пункты — для перехода от «работает у автора» к «надёжно работает у произвольного пилота».

Самая большая архитектурная рекомендация: **вынести логирование и FSM в отдельный `FlightRecorder` компонент**, который:
- управляет жизненным циклом обоих логгеров (IGC + CSV) атомарно;
- слушает GPS и сенсоры напрямую (а не через `MainActivity`);
- работает в собственном `HandlerThread`;
- предоставляет UI только статус (`isRecording()`, `getFlightTime()`).

Это убрало бы ~300 строк из `MainActivity` и устранило класс ошибок, связанных с рассинхронизацией логгеров.
