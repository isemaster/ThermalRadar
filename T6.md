# ThermalRadar v0.2.7 — delta-ревью (после правок)

**Репозиторий:** `https://github.com/isemaster/ThermalRadar` (тег `v0.2.7`)
**Коммит:** `a1e604a v0.2.7: CartoDB light tiles + User-Agent`
**Сравнение с:** v0.2.5 (предыдущее ревью)

---

## 0. Итог в одном абзаце

Автор откликнулся на ~60% Priority-0 замечаний, но **главная проблема с
картами не решена, а заменена новой**: вместо мёртвого `staticmap.openstreetmap.de`
теперь используется живой CartoDB tile-сервер, но код грузит **один тайл
256×256 px** и растягивает его на всю площадь радара (≈ 768×768 px).
Это даёт сильно размытую карту, занимающую 1/9 нужной площади —
визуально «карта подгружается», но выглядит как пиксельное месиво.
Часть компасных правок также не доведена до конца: `compassFailureStreak`
и `deadband=0.5°` применены, но критичные `calibrateHeading()` и
`getCompassHeading()` остались в исходном виде — компас всё равно будет
«отваливаться» на 2 секунды при старте лога и не переключится на
GPS-fallback при низкой калибровке магнетометра. Появился один новый
баг: `cacheKey` основан на tile-координатах, но `cacheFile` использует
его же + `.png` — это создаёт коллизию файлов между разными region-ами
на одном тайле.

## 2. Что не починено (подробно)

### 2.1. `SensorController.calibrateHeading()` — не тронут

```java
// SensorController.java:617-620 (v0.2.7 — без изменений)
public void calibrateHeading() {
    headingHoldValue = heading;
    headingHoldUntilMs = SystemClock.elapsedRealtime() + 2000; // 2 сек заморозки
    headingFilter.reset();
    // ← ОТСУТСТВУЕТ: не сбрасываем compassReady (по рекомендации ревью)
}
```

Положительная сторона: `compassReady = false` действительно убрано (в v0.2.5 было на строке 612, в v0.2.7 отсутствует). Это уже частичный фикс — компас не «сбрасывается» полностью.

**Но:** `headingFilter.reset()` остался. Это значит, что при старте лога (когда `calibrateHeading()` вызывается из `MainActivity.java:709, 837`) фильтр теряет всю накопленную историю α-β, и первые ~0.5 секунды после калибровки heading будет «плавать» от `headingHoldValue` к raw-значению. На быстром полёте это заметно как лёгкое дёргание.

**Кроме того, `isHeadingOnHold()` всё ещё работает 2 секунды:**

```java
// SensorController.java (v0.2.7) — обновление компаса
if (isHeadingOnHold()) {
    heading = headingHoldValue;   // ← 2 секунды показываем замороженное значение
} else {
    heading = (float) headingFilter.update(newHeading, SystemClock.elapsedRealtime());
}
```

То есть 2 секунды после каждого `calibrateHeading()` UI показывает ЗАМОРОЖЕННОЕ значение, а не реальное. Если пилот в этот момент делает вираж перед стартом, он видит, что компас «не двигается». Это ровно та же проблема, что в v0.2.5.

**Рекомендация (повторно):** убрать `headingFilter.reset()` и 2-секундную заморозку. Если нужна стабилизация после манёвра — использовать trust-period с увеличенным α (см. предыдущее ревью §2.2, баг №12).

### 2.2. `MainActivity.getCompassHeading()` — не тронут

```java
// MainActivity.java:1681-1691 (v0.2.7 — без изменений)
private float getCompassHeading() {
    if (sensorController.isCompassReady()) {
        return sensorController.getHeading();
    }
    if (gpsManager.isReady()) {
        sensorController.updateGpsHeading(gpsManager.getHeading());
        return sensorController.getHeading();
    }
    return 0.0f;   // ← возвращает 0 при отсутствии источников
}
```

Проблемы:
1. **Нет проверки `isMagAccurate()`.** Если `compassReady == true`, но `magAccuracy == SENSOR_STATUS_UNRELIABLE` (пользователь положил телефон рядом с магнитом, или чехол с магнитной застёжкой), `getCompassHeading()` вернёт мусор. Несмотря на то, что в `sensorDataListener` (строка 556, 603) такая проверка ЕСТЬ — но только для выбора rotation matrix. Сама heading используется без проверки.

2. **Нет проверки скорости/возраста фикса для GPS-fallback.** Условие `gpsManager.isReady()` не проверяет `getSpeed() > 2f` и `getFixAgeMs() < 5000`. Если GPS даёт последний фикс 30 секунд назад при скорости 0, GPS-курс всё равно подаётся в `updateGpsHeading()`. А там (см. ниже) — ничего не происходит, потому что условие строже.

3. **`return 0.0f` в конце.** Если ни компас, ни GPS не работают, радар разворачивается на север (0°). В авиации должно показываться «HDG FAIL» или хотя бы последнее известное значение.

### 2.3. `SensorController.updateGpsHeading()` — не тронут

```java
// SensorController.java:633-641 (v0.2.7 — без изменений)
public void updateGpsHeading(float gpsHeading) {
    if (!compassReady && magnetometer == null) {     // ← сверхжёсткое условие
        double filtered = headingFilter.update(gpsHeading, SystemClock.elapsedRealtime());
        if (!Double.isNaN(filtered)) {               // ← мёртвый код: filtered никогда не NaN
            heading = (float) filtered;
        }
    }
}
```

Условие `!compassReady && magnetometer == null` означает: GPS-fallback включается ТОЛЬКО если магнетометра физически нет на устройстве. На любом телефоне с магнетометром (99% устройств) fallback не сработает, даже если магнетометр неоткалиброван или `getRotationMatrix` стабильно отказывает.

Также `if (!Double.isNaN(filtered))` — это мёртвый код после фикса `HeadingFilter` (теперь `update()` всегда возвращает число). Можно убрать.

**Рекомендация (повторно):**

```java
public void updateGpsHeading(float gpsHeading) {
    boolean compassUnreliable = !compassReady
            || magnetometer == null
            || magAccuracy < SensorManager.SENSOR_STATUS_ACCURACY_MEDIUM
            || compassFailureStreak > COMPASS_FAILURE_THRESHOLD;
    if (compassUnreliable) {
        heading = (float) headingFilter.update(gpsHeading, SystemClock.elapsedRealtime());
    }
}
```

### 2.4. `StaticMapLoader.updateIfNeeded()` — rounding не убран

```java
// StaticMapLoader.java:119-133 (v0.2.7 — без изменений)
public void updateIfNeeded(double lat, int zoom) {
    if (hasCachedCenter) {
        double dist = haversineM(cachedCenterLat, cachedCenterLon, lat, lon);
        if (dist < UPDATE_THRESHOLD_M && zoom == cachedZoom) {
            return;
        }
    }
    double roundedLat = Math.round(lat * 100.0) / 100.0;   // ← ~1.1 км ячейка
    double roundedLon = Math.round(lon * 100.0) / 100.0;
    cachedCenterLat = roundedLat;
    cachedCenterLon = roundedLon;
    cachedZoom = zoom;
    hasCachedCenter = true;
    loadMap(roundedLat, roundedLon, zoom);
}
```

Округление осталось. Порог 500 м по-прежнему фактический ≈ 1.0–1.1 км, плюс flapping на границах ячеек. В v0.2.7 эта проблема усугубляется тем, что tile-запрос использует НЕ `roundedLat/roundedLon`, а `lonToTileX(lon, zoom)` от **исходных** координат (переданных в `loadMap`) — см. ниже.

### 2.5. `StaticMapLoader.entryRemoved` — recycle остался

```java
// StaticMapLoader.java:92-100 (v0.2.7 — без изменений)
@Override
protected void entryRemoved(boolean evicted, String key,
                             Bitmap oldValue, Bitmap newValue) {
    // BUG-14: recycle bitmap при вытеснении из кэша
    if (evicted && oldValue != null && !oldValue.isRecycled()) {
        oldValue.recycle();
        Log.d(TAG, "Bitmap recycled: " + key);

}
```

Комментарий прямо упоминает «BUG-14», но фикс не внесён. Crash на `Canvas.drawBitmap(recycledBitmap)` по-прежнему возможен, причём в v0.2.7 вероятность выше: теперь `backgroundMap` в RadarRenderer чаще меняется (т.к. tile-запросы делаются на каждый переход тайла, а не раз в 500 м).

### 2.6. `MainActivity.onDestroy()` — `staticMapLoader.destroy()` не вызывается

```java
// MainActivity.java:485-501 (v0.2.7 — без изменений)
@Override
protected void onDestroy() {
    super.onDestroy();
    releaseWakeLock();
    if (simHandler != null && simTask != null) {
        simHandler.removeCallbacks(simTask);
    }
    logManager.destroy();
    if (igcLogger != null) igcLogger.destroy();
```

`StaticMapLoader.executor` (single-thread ExecutorService) остаётся

`runOnUiThread` на мёртвой Activity просто тихо игнорируется —
некритично, но утечка executor остаётся).

## 3. НОВЫЕ проблемы, которых в v0.2.5 не было

### 3.1. КРИТИЧЕСКИЙ НОВЫЙ БАГ — загружается один тайл 256×256 вместо карты 768×768

Это главная регрессия v0.2.7. Автор заменил static map URL на tile URL, но **не изменил ни消费者a (`RadarRenderer`), ни размер ожидаемого bitmap-а**.

**Что происходит:**

```java
// StaticMapLoader.java:184-187
int tileX = lonToTileX(lon, zoom);
int tileY = latToTileY(lat, zoom);
final String url = String.format(Locale.US, OSM_TILE_URL, zoom, tileX, tileY);
// url = https://a.basemaps.cartocdn.com/light_all/14/9570/4762.png
//       (один тайл 256×256 px, содержащий район Питера)
```

`downloadBitmap()` возвращает `Bitmap` размером **256×256 px** (стандартный tile). Этот bitmap передаётся в:

```java
// RadarRenderer.java:330-340
if (mapValid && backgroundMap != null) {
    float[] offset = computeMapOffsetPx();
    float mapSize = r * 2.1f;        // ~768 px (при радиусе радара ~366 px)
    float mapLeft = cx - mapSize / 2f - offset[0];
    float mapTop  = cy - mapSize / 2f + offset[1];
    Rect dst = new Rect((int) mapLeft, (int) mapTop,
                        (int) (mapLeft + mapSize), (int) (mapTop + mapSize));
    canvas.drawBitmap(backgroundMap, null, dst, mapAlphaPaint);
    //                  ↑ 256×256    ↑ ~768×768   → растягивание 3×
}
```

`drawBitmap` растягивает 256×256 на 768×768 — это даёт **3× бикубическое размытие + видимый пиксельный «стеклянный» вид**. Один тайл zoom 14 покрывает ~9.6 км × ~6.5 км (на широте Питера) — это слишком крупный регион для радара диаметром 768 px, на котором пилот видит термик в 500 м от себя.

**Визуальный эффект:** пользователь видит серое размытое пятно вместо чёткой карты дорог и населённых пунктов. Это технически «карта подгрузилась», но практически — бесполезно.

**Дополнительная проблема: сдвиг карты несогласован с растяжением.**

```java
// RadarRenderer.java:103-104
private static final double METERS_PER_PIXEL = 4.78;  // для zoom 14, 60°N
// ...
// RadarRenderer.java:712-713
float offsetX = (float) (-eastM / METERS_PER_PIXEL);
float offsetY = (float) (northM / METERS_PER_PIXEL);
```

`computeMapOffsetPx()` считает сдвиг в пикселях при zoom 14 в предположении, что карта рисуется в нативном масштабе (1 тайл = 256 px = ~1.2 км). Но `drawBitmap` растягивает тайл до 768 px — реальный масштаб становится 3× крупнее (1 px ≈ 4 м). Сдвиг `offsetX = -eastM / 4.78` пересчитывает метры в «нативные» пиксели, а потом `drawBitmap` применяет этот сдвиг к растянутой карте. В итоге карта сдвигается в 3 раза слабее, чем нужно.

**Что нужно было сделать:** либо грузить 3×3 = 9 тайлов и склеивать в 768×768 bitmap (как было рекомендовано в предыдущем ревью §7.3), либо — как минимум — скорректировать `METERS_PER_PIXEL` на коэффициент растяжения:

```java
// Минимальный патч (костыль):
private static final double STRETCH_FACTOR = 3.0;  // 768/256
private double effectiveMetersPerPixel(int zoom, double lat) {
    return 156543.03 * Math.cos(Math.toRadians(lat)) / Math.pow(2, zoom) / STRETCH_FACTOR;
}
```

Но это полумера. Правильное решение — 9-тайловая склейка.

### 3.2. НОВЫЙ БАГ — кэш-ключ основан на tile, но центр карты сохраняется как lat/lon

```java
// StaticMapLoader.java:119-133 — updateIfNeeded()
double roundedLat = Math.round(lat * 100.0) / 100.0;
double roundedLon = Math.round(lon * 100.0) / 100.0;
cachedCenterLat = roundedLat;     // ← lat/lon
cachedCenterLon = roundedLon;
loadMap(roundedLat, roundedLon, zoom);

// StaticMapLoader.java:149-150 — loadMap()
final String cacheKey = makeTileCacheKey(lat, lon, zoom);   // ← tile-ключ

// StaticMapLoader.java:310-315 — makeTileCacheKey()
private static String makeTileCacheKey(double lat, double lon, int zoom) {
    int tx = lonToTileX(lon, zoom);
    int ty = latToTileY(lat, zoom);
    return tx + "_" + ty + "_z" + zoom;
}
```

`cachedCenterLat/Lon` хранит ОКРУГЛЁННЫЕ координаты (~1.1 км ячейка), но `cacheKey` строится от tile-координат (~6.5 км ячейка на zoom 14). Это создаёт сразу две проблемы:

**Проблема A: преждевременный cache miss.** Пилот сместился на 100 м внутри одной ячейки `roundedLat/Lon`. `updateIfNeeded` видит dist < 500 м, не обновляет. ОК. Пилот сместился ещё на 100 м — теперь `roundedLat` меняется на 0.01°, `cachedCenterLat` обновляется, вызывается `loadMap`. В `loadMap` tile-координата НЕ изменилась (тайл 6.5 км), но `makeTileCacheKey` даёт тот же ключ — cache hit. Хорошо, но 100 м движения внутри 6.5 км тайла не должны триггерить `loadMap` вообще.

**Проблема B: cachedCenterLat/Lon рассинхронизирован с реальным тайлом.** Допустим, пилот в точке `(59.950, 30.300)` — это tile `(9570, 4762)`. `cachedCenterLat = 59.95, cachedCenterLon = 30.30`. Через 600 м пилот в `(60.000, 30.300)` — `roundedLat = 60.00`, `cachedCenterLat = 60.00`. Но tile для `(60.00, 30.30)` — это `(9570, 4761)` (другой тайл!). Загружается новый bitmap, и `RadarRenderer.setBackgroundMap` его устанавливает с `centerLat=60.00, centerLon=30.30`. Но реальный центр этого тайла — `(59.978, 30.30)` (центр тайла 4761), а не `(60.00, 30.30)`. `computeMapOffsetPx()` считает расстояние от `(60.00, 30.30)` до пилота, хотя bitmap реально показывает тайл с центром `(59.978, 30.30)`. **Сдвиг карты систематически ошибается на ~1.3 км.**

**Исправление:** сохранить tile-координаты как cached state:

```java
private int cachedTileX, cachedTileY;

public void updateIfNeeded(double lat, double lon, int zoom) {
    int tx = lonToTileX(lon, zoom);
    int ty = latToTileY(lat, zoom);
    if (hasCachedCenter && tx == cachedTileX && ty == cachedTileY && zoom == cachedZoom) {
        return;
    }
    cachedTileX = tx;
    cachedTileY = ty;
    cachedZoom = zoom;
    // Сохранить ЦЕНТР тайла как cachedCenterLat/Lon (для computeMapOffsetPx):
    cachedCenterLat = tileYToLat(ty + 0.5, zoom);
    cachedCenterLon = tileXToLon(tx + 0.5, zoom);
    hasCachedCenter = true;
    loadMap(lat, lon, zoom);
}

// И сделать ThresholdDist по tile-границам, а не по haversine от roundedLat/Lon.
```

### 3.3. НОВЫЙ БАГ — `lonToTileX` для отрицательных долгот даёт сдвиг

```java
// StaticMapLoader.java:300-302
private static int lonToTileX(double lon, int zoom) {
    return (int) Math.floor((lon + 180.0) / 360.0 * (1 << zoom));
}
```

Сама функция корректна. Но при lon = -180.0 (линия перемены даты) она даёт `tileX = 0`, при lon = +180.0 — `tileX = 1 << zoom` (вне диапазона!). Стандартный tile-индекс должен быть в `[0, 2^zoom - 1]`. Для параплана это вряд ли критично (где парапланы летают рядом с линией перемены даты?), но это показывает, что код не покрыт edge-case тестами.

### 3.4. НОВЫЙ БАГ — User-Agent всё ещё не OSM-compliant

```java
// StaticMapLoader.java:215
conn.setRequestProperty("User-Agent", "ThermalRadar/0.2.6 (paragliding app)");
```

Версия в UA — `0.2.6`, а релиз — `v0.2.7`. Кроме того, OSM TOS требует контактную информацию (email или URL), чтобы владелец тайл-сервера мог связаться при злоупотреблении. CartoDB менее строг, но всё равно рекомендуется:

```java
conn.setRequestProperty("User-Agent",
    "ThermalRadar/0.2.7 (+https://github.com/isemaster/ThermalRadar)");
```

### 3.5. НОВЫЙ БАГ — `destroy()` без ожидания завершения

```java
// StaticMapLoader.java:110-113 (без изменений)
public void destroy() {
    executor.shutdown();
}
```

`shutdown()` не прерывает текущую задачу. Если `onDestroy` вызывается во время HTTP-запроса (10-секундный таймаут), executor будет жить до завершения. Не критично, но если `onDestroy` вызывается при перезапуске Activity (поворот, etc.), новый `StaticMapLoader` создаст новый executor, а старый продолжит работу — гонка за cache-файлы.

Рекомендация из предыдущего ревью остаётся: `shutdownNow()` + `awaitTermination(2s)`.

## 4. Карты: реальный статус «работают / не работают»

Чтобы понять, увидит ли пользователь карту, я протестировал CartoDB-URL напрямую:

```bash
$ curl -A "ThermalRadar/0.2.6 (paragliding app)" \
  "https://a.basemaps.cartocdn.com/light_all/14/9570/4762.png"
```

### 8.1. Починка растяжения тайла (минимум — 3×3 склейка)

```java
// StaticMapLoader.java — заменить loadMap() и downloadBitmap() на 9-тайловую склейку:

private static final int TILE_GRID = 3;       // 3×3 тайлов
private static final int TILE_SIZE  = 256;     // px

private void loadMap(double lat, double lon, int zoom) {
    final String cacheKey = makeTileCacheKey(lat, lon, zoom);

    // 1. memory cache
    Bitmap cached = memCache.get(cacheKey);
    if (cached != null && !cached.isRecycled()) {
        if (callback != null) callback.onMapLoaded(cached, lat, lon, zoom);
        return;
    }
    // 2. disk cache (без изменений)
    File cacheFile = new File(cacheDir, cacheKey + ".webp");
    if (cacheFile.exists()) { /* decode ... */ }

    // 3. HTTP: 9 тайлов параллельно
    if (!loading.compareAndSet(false, true)) return;

    int centerTx = lonToTileX(lon, zoom);
    int centerTy = latToTileY(lat, zoom);
    // центр тайла в координатах:
    double centerTileLat = tileYToLat(centerTy + 0.5, zoom);
    double centerTileLon = tileXToLon(centerTx + 0.5, zoom);

    executor.submit(() -> {
        Bitmap composite = Bitmap.createBitmap(
            TILE_GRID * TILE_SIZE, TILE_GRID * TILE_SIZE, Bitmap.Config.RGB_565);
        Canvas c = new Canvas(composite);
        Paint p = new Paint(Paint.FILTER_BITMAP_FLAG);

        // 9 тайлов: от centerTx-1..centerTx+1, centerTy-1..centerTy+1
        for (int dx = 0; dx < TILE_GRID; dx++) {
            for (int dy = 0; dy < TILE_GRID; dy++) {
                int tx = centerTx + dx - 1;
                int ty = centerTy + dy - 1;
                Bitmap tile = downloadSingleTile(tx, ty, zoom);
                if (tile != null) {
                    c.drawBitmap(tile, dx * TILE_SIZE, dy * TILE_SIZE, p);
                    tile.recycle();  // безопасно — мы скопировали пиксели в composite
                }
            }
        }
        loading.set(false);
        // Сохранить composite на диск
        // ...
        final Bitmap result = composite;
        mainHandler.post(() -> {
            memCache.put(cacheKey, result);
            if (callback != null) {
                callback.onMapLoaded(result, centerTileLat, centerTileLon, zoom);
            }
        });
    });
}

// tileYToLat / tileXToLon — обратные преобразования:
private static double tileYToLat(double ty, int zoom) {
    double n = Math.PI - 2.0 * Math.PI * ty / (1 << zoom);
    return Math.toDegrees(Math.atan(Math.sinh(n)));
}
private static double tileXToLon(double tx, int zoom) {
    return tx / (1 << zoom) * 360.0 - 180.0;
}
```

В `updateIfNeeded()` и `forceUpdate()` убрать `roundedLat/Lon`, сохранять в `cachedCenterLat/Lon` **центр тайла**, а не округлённую позицию пилота:

```java
public void updateIfNeeded(double lat, double lon, int zoom) {
    int tx = lonToTileX(lon, zoom);
    int ty = latToTileY(lat, zoom);
    if (hasCachedCenter && tx == cachedTileX && ty == cachedTileY && zoom == cachedZoom) {
        return;
    }
    cachedTileX = tx;
    cachedTileY = ty;
    cachedZoom = zoom;
    cachedCenterLat = tileYToLat(ty + 0.5, zoom);
    cachedCenterLon = tileXToLon(tx + 0.5, zoom);
    hasCachedCenter = true;
    loadMap(lat, lon, zoom);  // внутри loadMap cache key строится от tx/ty
}
```
