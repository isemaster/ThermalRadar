## 0. Executive Summary

Проект представляет собой Android-приложение «теплового радара» для парапланеристов, реализующее вариометр, детектор термиков, расчёт ветра, IGC-логгирование и проигрывание чужих треков. Архитектура гибридная: «живая» полётная логика в `FlightController` + «пайплайн IGC» в `IGCAnalyzer` для реплеев.

**Из трёх пользовательских багов все три имеют конкретные кодовые причины, ни один не является «косметическим»**:

## 1. Разбор трёх критических багов пользователя

### 1.1 Баг №1: «Показания скачут, плохо усредняются» при проигрывании чужих IGC-треков

#### 1.1.1 Корневая причина №1: неправильный приоритет источников высоты

**Файл:** `app/src/main/java/com/termo1/radar/igc/TrackPoint.java:29`

```java
// ТЕКУЩИЙ КОД (БАГ)
public TrackPoint(double lat, double lon,
                  float pressAltM, float gpsAltM,
                  float timeSec, boolean fixValid) {
    // ...
    this.displayAltM = (gpsAltM > 0f) ? gpsAltM : pressAltM;  // ← БАГ
}

public float getVarioAltM() {
    return (pressAltM > 0f) ? pressAltM : gpsAltM;  // ← правильно для vario
}
```

**Что происходит:**

В мире парапланерных приборов (XCSoar, LK8000, Flytec, LXNAV, Naviter Oudie, Vector Vario) существует жёсткое правило: **для отображения высоты используется барометрическая высота (P_ALT), а не GPS (G_ALT)**. Причины:

1. Барометр имеет шум **±0.1–0.3 м** против **±5–15 м** у GNSS (WGS84 эллипсоид + ионосферныеRefraction + multipath).
2. Барометр даёт непрерывный сигнал даже в облачности/ущелье, где GPS теряет спутники.
3. Вертикальная скорость (`vario = dh/dt`) из барометра в 30–50 раз стабильнее.

В текущей реализации `displayAltM` **предпочитает GPS** (`gpsAltM > 0`), а барометр используется только как fallback. Это означает:

- Чужой IGC-трек с барометром (например, LXNAV) → `pressAltM = 1250.0`, `gpsAltM = 1257.3` → `displayAltM = 1257.3` (GPS!). На радаре пилот видит «скачущую» высоту с размахом ±15 м, что при α=0.25 EMA даёт видимые колебания ±3–5 м в секунду.
- Собственный IGC без барометра (`pressAltM = 0`) → `displayAltM = gpsAltM` — это корректно (выбора нет).

**Что говорят мировые стандарты (из algo.md и XCSoar):**

```java
// XCSoar: src/Computer/State.cpp
// Приоритет для отображаемой высоты:
//   1. Baro altitude (если есть и валидна)
//   2. GPS altitude (fallback)
//   3. 0 (нет данных)
```

**Исправление:**

```java
// ФИКС: давление приоритет, GPS только fallback
public TrackPoint(double lat, double lon,
                  float pressAltM, float gpsAltM,
                  float timeSec, boolean fixValid) {
    this.lat = lat;
    this.lon = lon;
    this.pressAltM = pressAltM;
    this.gpsAltM = gpsAltM;
    this.timeSec = timeSec;
    this.fixValid = fixValid;

    // ПРИОРИТЕТ: баро (если > 0), иначе GPS
    // Примечание: pressAltM = 0 трактуется как «нет бародатчика»
    // (см. IGCParser — там "00000" даёт pressAlt=0)
    this.displayAltM = (pressAltM > 0f) ? pressAltM : gpsAltM;
}

// Высота для vario — та же логика (было правильно, оставляем)
public float getVarioAltM() {
    return (pressAltM > 0f) ? pressAltM : gpsAltM;
}
```

**Дополнительно — вычислить QNH-поправку для отображения:**

Стандартная барометрическая высота даётся относительно QNE=1013.25 hPa. Для отображения высоты над уровнем моря (MSL) нужна поправка:

```java
// Поправка: 1 hPa ≈ 8.43 м на уровне моря
float qnhCorrectionM = (qnhHpa - 1013.25f) * 8.43f;
this.displayAltM = (pressAltM > 0f) ? pressAltM + qnhCorrectionM : gpsAltM;
```

`qnhHpa` уже парсится в `IGCParser.ParseResult` — нужно прокинуть его в `TrackPoint` или в `IGCAnalyzer` для пост-обработки.

#### 1.1.2 Корневая причина №2: слишком агрессивное сглаживание варио

**Файл:** `IGCAnalyzer.java:33, 222-230`

```java
private static final float VARIO_EMA_ALPHA = 0.25f;  // ← слишком много

// В advance():
float rawVario = altDelta / Math.max(dt, 0.01f);
// ... TE compensation ...
smoothVario = smoothVario * (1f - VARIO_EMA_ALPHA) + teVario * VARIO_EMA_ALPHA;
```

**Математика EMA:**

Время установления 63% (τ) для EMA с α=0.25 при Δt=1с равно `τ = -Δt / ln(1-α) ≈ 3.48 сек`. Время установления 95% ≈ 3τ ≈ 10.5 сек. Для **звукового** варио это **слишком медленно** (пилот чувствует термик с задержкой 3 сек). Для **отображения** это **слишком быстро** (10 сек — много шума).

Мировые стандарты (см. теор. часть в `chat3.txt` §7.5):

| Параметр | Окно | α (для Δt=1с) |
|----------|------|---------------|
| Instant vario (звук) | 0.5–1 сек | 0.7–0.9 |
| TE vario (пилотаж) | 1–2 сек | 0.4–0.6 |
| Average vario (термик) | 10–30 сек | 0.05–0.15 |

В коде используется **одно значение** `α=0.25` для всего, что соответствует «ни звуку, ни дисплею».

**Исправление:**

```java
// Два раздельных канала
private static final float VARIO_SOUND_ALPHA = 0.6f;   // 1.5 сек τ — для звука
private static final float VARIO_DISPLAY_ALPHA = 0.1f; // 9.5 сек τ — для дисплея
private static final float VARIO_AVG30_ALPHA = 0.05f;  // ~20 сек τ — для среднего

private float soundVario = 0f;       // быстрый, для VarioSoundManager
private float displayVario = 0f;     // медленный, для HUD
private float avgVario30 = 0f;       // очень медленный, реальное 30с окно

// В advance():
float teVario = rawVario + keDelta / Math.max(dt, 0.001f);
soundVario   += (teVario - soundVario)   * VARIO_SOUND_ALPHA;
displayVario += (teVario - displayVario) * VARIO_DISPLAY_ALPHA;
avgVario30   += (teVario - avgVario30)   * VARIO_AVG30_ALPHA;
```

```java
private static int varioToColor(float vario, float alpha) {
    final float[] keys = {-5f, -2f, -0.3f, 0f, 0.5f, 1.5f, 3f, 5f, 8f};
    final int[] colors = {
        0x001A00, 0x006600, 0x99CC66, 0xFFFF33,
        0xFFAA00, 0xFF4400, 0xFF0000, 0x660000, 0x330000
    };
    int hi = 1;
    while (hi < keys.length - 1 && vario > keys[hi]) hi++;
    int lo = hi - 1;
    float denom = keys[hi] - keys[lo];
    float t = (denom == 0f) ? 0f : (vario - keys[lo]) / denom;
    t = Math.max(0f, Math.min(1f, t));
    int r = (int)(((colors[lo] >> 16) & 0xFF) * (1f - t) + ((colors[hi] >> 16) & 0xFF) * t);
    int g = (int)(((colors[lo] >> 8)  & 0xFF) * (1f - t) + ((colors[hi] >> 8)  & 0xFF) * t);
    int b = (int)((colors[lo] & 0xFF) * (1f - t) + (colors[hi] & 0xFF) * t);
    int a = (int)(alpha * 200);
    if (a < 8) a = 8;
    if (a > 255) a = 255;
    return (a << 24) | (r << 16) | (g << 8) | b;
}
```

#### 1.1.5 Корневая причина №5: отсутствие Kalman-фильтра

В `algo.md` упоминаются «Kalman (baro+GPS+IMU) fusion», но в коде **нет ни одного фильтра Калмана**. Только EMA. Для параплана с его чувствительностью к термику 0.5 м/с это критично.

Минимальный 1D Kalman для высоты:

```java
public class AltitudeKalman {
    private float x;        // state: altitude
    private float v;        // state: vertical speed
    private float p00, p01, p10, p11;  // covariance 2x2
    private float q;        // process noise (turbulence)
    private float rBaro, rGps;  // measurement noise

    public AltitudeKalman() {
        x = 0; v = 0;
        p00 = 100; p01 = 0; p10 = 0; p11 = 1;
        q = 0.1f; rBaro = 0.25f * 0.25f; rGps = 10f * 10f;
    }

    public void predict(float dt) {
        x += v * dt;
        // F = [[1, dt], [0, 1]]
        // P = F P F^T + Q
        float new_p00 = p00 + p01*dt + p10*dt + p11*dt*dt + q*dt*dt*dt*dt/4f;
        float new_p01 = p01 + p11*dt + q*dt*dt*dt/2f;
        float new_p10 = p10 + p11*dt + q*dt*dt*dt/2f;
        float new_p11 = p11 + q*dt*dt;
        p00 = new_p00; p01 = new_p01; p10 = new_p10; p11 = new_p11;
    }

    public void updateBaro(float z, float dt) {
        // H = [1, 0], R = rBaro
        float y = z - x;              // innovation
        float s = p00 + rBaro;        // innovation covariance
        float k0 = p00 / s;           // Kalman gain
        float k1 = p10 / s;
        x += k0 * y;
        v += k1 * y;
        p00 -= k0 * p00;
        p01 -= k0 * p01;
        p10 -= k1 * p00;
        p11 -= k1 * p01;
    }

    public float getAltitude() { return x; }
    public float getVario()    { return v; }
}
```

Использовать в `IGCAnalyzer.advance()`:

```java
private final AltitudeKalman altKf = new AltitudeKalman();

// В advance(dt):
altKf.predict(dt);
if (newVarioAlt > 0) altKf.updateBaro(newVarioAlt, dt);
altitude = altKf.getAltitude();
float rawVario = altKf.getVario();  // уже отфильтровано
```

Это даст шум дисплея ±0.3 м и задержку < 0.3 сек — уровень коммерческих приборов.

### 1.2 Баг №2: «Неверно указано направление ветра на радаре»

#### 1.2.1 Корневая причина: сломанный алгоритм оценки ветра

**Файл:** `IGCAnalyzer.java:438-447` (метод `estimateWind()`)

```java
private void estimateWind() {
    float speed = getSpeedMs();
    if (speed < 0.5f) return;

    // Along-track wind (no compass needed)
    float alongWind = speed - AIRSPEED_MS;        // ← только продольная компонента
    float windSpd = Math.abs(alongWind);          // ← модуль, потеря знака
    if (windSpd < 0.3f) return;

    float windDir = headingDeg + 180f;             // ← ВСЕГДА «от хвоста»!
    if (windDir < 0) windDir += 360;
    if (windDir >= 360) windDir -= 360;

    // Crosswind rejection
    if (windCount > 0) {
        float avgDir = getWindFromDeg();
        if (avgDir >= 0) {
            float dirDiff = Math.abs(windDir - avgDir);
            if (dirDiff > 180) dirDiff = 360 - dirDiff;
            if (dirDiff > 90) return;              // ← выбрасывает реальные измерения
        }
    }

    // Просто сохраняем в smoothWindDir/Spd для внутренних расчётов
    if (smoothWindSpd < 0 || smoothWindDir < 0) {
        smoothWindDir = windDir;
        smoothWindSpd = windSpd;
    } else {
        smoothWindDir = windDir;                   // ← if/else ИДЕНТИЧНЫ!
        smoothWindSpd = windSpd;
    }
}
```

**Тут минимум 5 независимых багов:**

1. **`alongWind = speed - AIRSPEED_MS`** — это **продольная компонента** ветра, а не его модуль. Если у вас кросс-ветер 10 м/с и продольный 0, вы получите `alongWind=0` и метод выйдет по `if (windSpd < 0.3f) return`. Кросс-ветер полностью игнорируется.

2. **`windSpd = Math.abs(alongWind)`** — потеря знака. При встречном ветре (speed=7, airspeed=9.5) `alongWind=-2.5`, `windSpd=+2.5`. Знак теряется, и мы не знаем, ветер в лоб или в хвост.

3. **`windDir = headingDeg + 180f`** — направление **всегда** «от хвоста» (сзади). Если у вас встречный ветер, он должен быть `windDir = headingDeg` (спереди). Код даёт неправильное направление во всех случаях, кроме чистого попутного ветра.

4. **`if/else` с одинаковыми ветками** — `if (smoothWindSpd < 0 || ...)` и `else` делают **одно и то же присваивание**. Никакого сглаживания (EMA/MA) нет, каждое новое измерение перезаписывает предыдущее → ветер «скачет» от измерения к измерению.

5. **`Crosswind rejection` с порогом > 90°** — если новое измерение говорит о ветре с направления, отличающегося от среднего более чем на 90°, оно **выбрасывается**. Это блокирует коррекцию при реальном изменении ветра (например, при переходе на другую высоту с другим ветром).

#### 1.2.2 Как должно быть: Circling Wind Algorithm (XCSoar gold standard)

В `algo.md` (раздел 11.1) это описано подробно. Минимальная реализация:

```java
// Новый класс: com/termo1/radar/igc/CirclingWindEstimator.java

public class CirclingWindEstimator {

    /** Один семпл в развороте */
    private static class Sample {
        final float trackDeg;     // курс в момент семпла
        final float groundSpeed;  // путевая скорость (м/с)
        final float airspeed;     // воздушная скорость (м/с)
        final float timeSec;
        Sample(float t, float gs, float as, float ts) {
            trackDeg = t; groundSpeed = gs; airspeed = as; timeSec = ts;
        }
    }

    private static final int MAX_SAMPLES = 80;
    private final List<Sample> samples = new ArrayList<>(MAX_SAMPLES);
    private float prevTrack = -1f;
    private float accumulatedAngle = 0f;

    /** Добавить точку трека в развороте */
    public void addSample(float trackDeg, float groundSpeed, float airspeed, float timeSec) {
        if (prevTrack >= 0) {
            float d = trackDeg - prevTrack;
            while (d > 180) d -= 360;
            while (d < -180) d += 360;
            accumulatedAngle += Math.abs(d);
        }
        prevTrack = trackDeg;

        samples.add(new Sample(trackDeg, groundSpeed, airspeed, timeSec));
        if (samples.size() > MAX_SAMPLES) samples.remove(0);
    }

    /** Сбросить буфер (выход из разворота) */
    public void reset() {
        samples.clear();
        prevTrack = -1f;
        accumulatedAngle = 0f;
    }

    /** Готово ли к расчёту (минимум 270° набрано) */
    public boolean isReady() {
        return accumulatedAngle >= 270f && samples.size() >= 8;
    }

    /** Рассчитать ветер */
    public WindResult estimate() {
        if (!isReady()) return null;

        // 1. Offset: средняя разность (GS - TAS)
        double offset = 0;
        for (Sample s : samples) {
            offset += (s.groundSpeed - s.airspeed);
        }
        offset /= samples.size();

        // 2. Amplitude: avg(|GS - TAS - offset|) × π/2
        double sumAbs = 0;
        for (Sample s : samples) {
            sumAbs += Math.abs(s.groundSpeed - s.airspeed - offset);
        }
        double amplitude = (sumAbs / samples.size()) * (Math.PI / 2.0);
        if (amplitude > 30) amplitude = 30;  // реалистичный лимит

        // 3. Cosine fit: найти направление, минимизирующее
        //    sum((GS - TAS - offset) - amplitude*cos(track - wind_dir))^2
        //    Итеративное сужение: 6 шагов по кругу → 3+3+3 шага сужения
        double bestDir = 0;
        double bestErr = Double.MAX_VALUE;
        double step = 60.0;  // начальный шаг
        double center = 0;

        for (int iter = 0; iter < 4; iter++) {
            for (int i = -3; i <= 3; i++) {
                double dir = (center + i * step + 360) % 360;
                double err = 0;
                for (Sample s : samples) {
                    double model = amplitude * Math.cos(Math.toRadians(s.trackDeg - dir));
                    double diff = (s.groundSpeed - s.airspeed - offset) - model;
                    err += diff * diff;
                }
                if (err < bestErr) {
                    bestErr = err;
                    bestDir = dir;
                }
            }
            center = bestDir;
            step /= 2.0;
        }

        // 4. Скорость ветра
        double windSpeed = Math.sqrt(offset * offset + amplitude * amplitude);
        if (windSpeed > 30) windSpeed = 30;

        // 5. Направление «откуда дует»:
        //    Если offset > 0 — попутный (ветер сзади), направл = bestDir + 180
        //    Если offset < 0 — встречный (ветер спереди), направл = bestDir
        //    Универсально: windFrom = bestDir + 180, если сместить фазу правильно
        double windFrom = (bestDir + 180.0 + (offset > 0 ? 0 : 180)) % 360;

        return new WindResult((float) windFrom, (float) windSpeed, samples.size());
    }

    public static class WindResult {
        public final float windFromDeg;
        public final float windSpeedMs;
        public final int sampleCount;
        public WindResult(float d, float s, int n) {
            windFromDeg = d; windSpeedMs = s; sampleCount = n;
        }
    }
}
```

```java
    if (wakeLock != null && (now - lastWakeLockRefreshMs > WAKE_LOCK_TIMEOUT_MS / 2)) {
        releaseWakeLock();
        acquireWakeLock();
    }
}
```

Проблемы:
1. Если `acquireWakeLock()` вызывается повторно (например, при `refreshWakeLock()` → `releaseWakeLock()` не успел отработать из-за исключения), WakeLock не переарендуется, но и не освобождается — **утечка WakeLock**.
2. `wakeLock.acquire()` без таймаута — если приложение упадёт без `onDestroy`, WakeLock будет висеть до перезагрузки устройства (жрут батарею).
3. `releaseWakeLock()` не проверяет `isHeld()`:
   ```java
   void releaseWakeLock() {
       if (wakeLock != null && wakeLock.isHeld()) {
           wakeLock.release();
       }
       wakeLock = null;
   }
   ```
   Здесь OK, но нет `try/finally` — если `release()` выбросит исключение (что редко, но бывает при отзыве perm), `wakeLock` не занулится → при следующем `acquireWakeLock()` проверка `wakeLock == null` даст false → новый WakeLock не будет создан.

**Исправление:**

```java
void acquireWakeLock() {
    if (powerManager == null) return;
    if (wakeLock != null && wakeLock.isHeld()) return;  // уже держим
    if (wakeLock == null) {
        wakeLock = powerManager.newWakeLock(
                PowerManager.PARTIAL_WAKE_LOCK, "TERMO1:RadarLock");
    }
    if (wakeLock != null) {
        // Всегда с таймаутом как fallback, но большим
        wakeLock.acquire(WAKE_LOCK_TIMEOUT_MS);
        lastWakeLockRefreshMs = System.currentTimeMillis();
    }
}

void releaseWakeLock() {
    if (wakeLock == null) return;
    try {
        if (wakeLock.isHeld()) wakeLock.release();
    } catch (Exception e) {
        Log.w("TERMO1", "WakeLock release failed", e);
    } finally {
        wakeLock = null;
    }
}
```

1. **`MainActivity` — god object (2146 строк).** Содержит: lifecycle, UI, sensor callbacks, TTS, simulation, track replay, IGC bridge, logging callbacks, wakelock, calibration. Должно быть разбито минимум на 5 классов:

4. **Нет dependency injection.** Все зависимости передаются через конструкторы (что лучше, чем static singletons), но 14-параметрический конструктор `FlightController` (строки 88-114) — это code smell. Использовать Dagger Hilt или Koin:

   ```kotlin

       @Provides @Singleton
       fun provideFlightController(...) = FlightController(...)

   ```

5. **Нет ViewModel.** Конфигурационные изменения (поворот экрана — хотя сейчас запрещён, но в будущем) требуют сохранения состояния. `MainActivity` хранит всё в полях — при пересоздании всё теряется.

6. **Нарушение SRP в `FlightController`:** класс делает:
   - bg-поток lifecycle
   - FSM updates
   - circling + lift + thermal locator
   - thermal blip storage
   - wind drift calculation
   
   Это 5 разных ответственностей. Минимум — вынести wind в `WindEstimator`, thermal в `ThermalTracker`.

7. **Нет repository pattern.** SharedPreferences читается напрямую в 8 местах. Должно быть `SettingsRepository` с типизированными getters/setters.

### 4.1 PERF-1: `findSegment()` — линейный поиск O(n)

**Файл:** `IGCAnalyzer.java:823-831`

```java
private int findSegment(float targetTime) {
    for (int i = currentIdx; i < track.length - 1; i++) {
        if (targetTime >= track[i].timeSec && targetTime < track[i + 1].timeSec) {
            return i;
        }
    }
    return track.length - 2;
}
```

Для трека 1 час @ 1 Гц = 3600 точек. На каждом кадре реплея (50 мс) — цикл до 3600 итераций. Это 72000 итераций в секунду — терпимо, но **если трек 5 часов @ 5 Гц = 90000 точек**, то 1.8 млн итераций/сек.

**Исправление:** бинарный поиск:

```java
private int findSegment(float targetTime) {
    int lo = currentIdx;  // оптимизация: ищем только вперёд
    int hi = track.length - 1;
    if (targetTime >= track[hi].timeSec) return hi - 1;
    while (lo < hi) {
        int mid = (lo + hi) >>> 1;
        if (track[mid].timeSec <= targetTime) {
            if (track[mid + 1].timeSec > targetTime) return mid;
            lo = mid + 1;
        } else {
            hi = mid;
        }
    }
    return Math.max(0, lo - 1);
}
```

### 4.2 PERF-2: `IGCParser.haversineMeters()` вызывается в каждом кадре

**Файл:** `IGCAnalyzer.java:248` (advance)

```java
            track[i].lat, track[i].lon, track[i+1].lat, track[i+1].lon);

}
```

```java
package com.termo1.radar.igc;

/**
 * Точка трека (IGC B-record).
 *
 * Единое представление для всех источников: чужой IGC, живой полёт.
 * Содержит как pressure altitude (baro), так и GPS altitude.
 *
 * Приоритет высоты для отображения:
 *   1. Pressure altitude (P_ALT) — гладкая, приоритет при наличии бародатчика
 *   2. GPS altitude (G_ALT) — fallback когда P_ALT == 0 ("00000" в IGC)
 *
 * QNH-коррекция применяется в IGCAnalyzer при наличии qnhHpa из H-record.
 */
public class TrackPoint {
    public final double lat, lon;
    public final float pressAltM;     // barometric altitude (m), 0 если нет бародатчика
    public final float gpsAltM;       // GNSS altitude (m)
    public final float timeSec;       // seconds from start of flight
    public final boolean fixValid;    // 'A' = valid, 'V' = invalid

    /** Высота для отображения: баро приоритет (гладкая), GPS fallback (шумная) */
    public final float displayAltM;

    public TrackPoint(double lat, double lon,
                      float pressAltM, float gpsAltM,
                      float timeSec, boolean fixValid) {
        this.lat = lat;
        this.lon = lon;
        this.pressAltM = pressAltM;
        this.gpsAltM = gpsAltM;
        this.timeSec = timeSec;
        this.fixValid = fixValid;
        // БАГ-ФИКС: было (gpsAltM > 0) ? gpsAltM : pressAltM
        // Баро должна быть приоритет — она в 30-50 раз глаже GPS.
        // pressAltM == 0 означает "бародатчика нет" (IGC "00000").
        this.displayAltM = (pressAltM > 0f) ? pressAltM : gpsAltM;
    }

    /** Высота для расчёта vario: та же логика, что и для display */
    public float getVarioAltM() {
        return (pressAltM > 0f) ? pressAltM : gpsAltM;
    }

    /** Создать копию с QNH-коррекцией (для display) */
    public TrackPoint withQnhCorrection(float qnhCorrectionM) {
        if (pressAltM <= 0f || qnhCorrectionM == 0f) return this;
        return new TrackPoint(lat, lon, pressAltM, gpsAltM, timeSec, fixValid) {
            // Анонимный класс не может изменить final поле — нужен конструктор с qnh
        };
    }
}
```

### 12.3 Патч для `IGCAnalyzer.estimateWind()` (баг 1.2)

Заменить целиком (см. §1.2.2 — `CirclingWindEstimator` класс). Минимальный patch (без нового класса):

```java
private void estimateWind() {
    float speed = getSpeedMs();
    if (speed < 0.5f) return;

    // Полный вектор ветра через GPS vector и airspeed
    // gpsVx, gpsVy — компоненты путевой скорости в east/north
    double hdgRad = Math.toRadians(headingDeg);
    double gpsVx = speed * Math.sin(hdgRad);  // east
    double gpsVy = speed * Math.cos(hdgRad);  // north

    // Вектор воздушной скорости (предполагаем, что пилот летит по курсу)
    double asVx = airspeedMs * Math.sin(hdgRad);
    double asVy = airspeedMs * Math.cos(hdgRad);

    // Ветер = GPS - Airspeed (векторная разность)
    // windU — восточная компонента ветра (куда дует)
    // windV — северная компонента ветра (куда дует)
    double windU = gpsVx - asVx;
    double windV = gpsVy - asVy;

    double windSpeed = Math.sqrt(windU * windU + windV * windV);
    if (windSpeed < 0.3f) return;
    if (windSpeed > 30.0) return;  // нереалистично

    // Направление ОТКУДА дует:
    // windU > 0 → ветер на восток → с запада (270°)
    // windV > 0 → ветер на север → с юга (180°)
    // windFrom = atan2(-windU, -windV) в формате «0=север, 90=восток»
    double windFrom = Math.toDegrees(Math.atan2(-windU, -windV));
    if (windFrom < 0) windFrom += 360;

    // Сохраняем в WindStore (векторное усреднение)
    long nowMs = (long)(currentTimeSec * 1000);
    int quality = Math.min(5, (int)(windSpeed * 2));  // эвристика
    windStore.addMeasurement((float) windFrom, (float) windSpeed, quality,
                             altitude, nowMs);

    // Обновляем smooth-значения для обратной совместимости
    WindStore.WindMeasurement w = windStore.getWindAt(altitude, nowMs);
    if (w != null) {
        smoothWindDir = (float) w.bearing;
        smoothWindSpd = (float) w.speed;
    }
}
```

**ВАЖНО:** этот патч всё равно даёт только **мгновенную оценку** ветра по одной точке. Для качественной оценки нужен **Circling Wind Algorithm** (см. §1.2.2), который накапливает точки в развороте. Мгновенная оценка — это «zig-zag method» из теор. части, требует минимум 2 галса.

## 13. Заключение

Проект ThermalRadar v0.3.6 — это **серьёзная попытка** сделать профессиональный парапланерный прибор на Android, но он **не дотягивает до уровня XCSoar/LK8000** по качеству алгоритмов. Три пользовательских бага, описанные в задаче, имеют **конкретные кодовые причины**, ни один не является «магическим»:

1. **«Скачут показания»** — неправильный приоритет GPS над баро в `TrackPoint.displayAltM` + слишком агрессивный EMA + отсутствие Kalman-фильтра.
2. **«Неверно направление ветра»** — сломанный алгоритм `estimateWind()` (along-track only, всегда «от хвоста», идентичные if/else).
3. **«Автостарт в фоне»** — рассинх `flightController.trackMode` флага + отсутствие понятия «user paused» в lifecycle.

Все три бага **исправимы локально** (без переписывания архитектуры), но для предотвращения регрессий нужна **архитектурная работа** (Спринт 3 в плане рефакторинга): разбить god-object MainActivity, унифицировать live/replay pipeline, добавить CI.

**Ключевая рекомендация:** после исправления P0-багов выпустить v0.4.0, прогнать regression tests на 5 реальных IGC-треках от разных приборов (LXNav, Flytec, XCSoar, Naviter), сравнить показания с эталонными устройствами. Только после этого продолжать разработку новых фич.
