# Анализ репозитория ThermalRadar v0.1.0

## Общая оценка

**ThermalRadar** — это амбициозный и технически грамотный проект Android-приложения для парапланеристов, которое анализирует микрораскачку крыла через акселерометр телефона для обнаружения термиков, отслеживания крутки и помощи в центровке ядра. Версия 0.1.0 демонстрирует серьёзный подход к обработке сигналов и продуманную архитектуру.

---

## Что сделано хорошо

### 1. Алгоритмическая база
- **Полосовой фильтр Баттерворта 2-го порядка** (0.25–2.5 Гц) — корректный выбор диапазона для микрораскачки параплана. Фильтр реализован как каскад ФВЧ→ФНЧ с anti-windup защитой.
- **RMS-окно 64 отсчёта** (~1.3 с) + zero-crossing детектор частоты — логичная комбинация для оценки амплитуды и частоты турбулентности.
- **Адаптивный шумовой порог** с медленной подстройкой (τ ≈ 40 с) — позволяет адаптироваться к условиям без ложных срабатываний.
- **Верификация blip** через сравнение амплитуд первого и второго периода (ratio 1.2×/0.8×) — интересная эвристика для отсева ложных срабатываний.

### 2. Архитектура кода
- Чёткое разделение на модули: `core/`, `flight/`, `sensors/`, `gps/`, `logging/`, `ui/`, `model/`
- `SensorController` с `SensorDataListener` интерфейсом — правильный подход к декомпозиции.
- `VarioManager` выделен отдельно от `SensorController` (SRP).
- `CirclingManager` с секторным анализом варио (N/E/S/W) и оценкой ветра по дрейфу GPS-центров спиралей — физически обоснованный подход.

### 3. Полезные фичи для пилота
- **Слепой режим (карман)** — 15% яркость + голосовые подсказки. Критически важно для безопасности.
- **Голосовые подсказки TTS** с дебаунсом 8 сек и фильтрацией задних секторов.
- **Ветер по дрейфу спиралей** — наиболее точный метод для параплана (сравнение центроидов GPS-точек спиралей).
- **Центровка ядра** после каждого круга: «ядро на ветер» / «ядро по ветру» / кардинальные направления.
- **Солнечный режим** — макс. яркость + контрастный HUD.
- **Логирование 50 Гц** в ZIP-чанки по 10 минут с Events.csv — отлично для пост-полётного анализа.
- **Монотонное время** (`elapsedRealtime`) — защита от скачков NTP/перевода часов.
- **Тестовый режим** из 6 шагов — позволяет проверить все датчики перед полётом.

### 4. Безопасность и UX
- Кнопка СТОП с подтверждением — случайно не прервать запись.
- GPS всегда включён для логирования координат.
- WakeLock с таймаутом и периодическим обновлением.
- Автостарт полёта по высоте (+10 м за 10 с).

## Критические замечания и что надо доработать

### 🔴 Критические проблемы

#### 1. **Race conditions и thread-safety**
- `MainActivity.onDraw()` выполняется в UI-треде, но читает `volatile` поля из сенсорного потока. Хотя поля `volatile`, **списки `thermals` и `thermalsCopy` синхронизированы через `synchronized(thermalLock)`** — это хорошо. Но:
  - `CirclingManager` не синхронизирован — вызов `update()` из сенсорного потока (через `onLinearAccelSample`) и чтение `isCircling()`/`getWindFromDeg()` из UI-треда без синхронизации. **Потенциальный data race.**
  - `ThermalDetector.processSample()` вызывается из сенсорного потока, а `getCurrentBlip()` — из UI. `currentBlip` — `volatile`, но внутреннее состояние объекта `ThermalBlip` не immutable. **При одновременном чтении и обновлении полей `distance`/`strength`/`angle` в `onDraw()` возможно чтение неконсистентного состояния.**

#### 2. **Проблема с `SensorController` и `VarioManager`**
- В `SensorController` есть **дублирующийся код** вариометра: и в `onPressureSample()` (старый метод), и в `VarioManager.processPressureSample()`. `SensorController` создаёт `VarioManager`, но при этом сам тоже обрабатывает давление. Это запутанно и может приводить к двойной обработке.
- `SensorController.onPressureSample()` нигде не вызывается — метод есть, но вызова нет. Вместо этого `VarioManager` обрабатывает давление напрямую. Это мёртвый код.

#### 3. **GPS-локация и разрешения**
- `GpsManager.requestUpdates()` создаёт **новый `LocationRequest`** вместо использования того, что создан в `startGps()`. Параметры дублируются.
- Нет обработки случая, когда GPS permission denied — приложение просто покажет Toast и продолжит работу без GPS, но логи будут без координат.

#### 4. **Memory leaks**
- `MainActivity` создаёт `new Paint()` прямо в `onDraw()` для ветровой стрелки и подписи — **allocation в цикле рендеринга 30 FPS**. Это приведёт к постоянным GC-паузам.
  ```java
  Paint windLine = new Paint(Paint.ANTI_ALIAS_FLAG); // Создаётся каждый кадр!
  ```
- Аналогично `Paint windLabel`, `Paint windFill`, `Paint thermalLabelPaint` — все создаются в `onDraw()`.

#### 5. **Проблемы с компасом**
- `SensorManager.getRotationMatrix()` требует **калибровки магнитометра**. В полёте телефон на груди — магнитные помехи от подвески, рации, двигателя (если моторный параплан). Нет никакой компенсации магнитных аномалий.
- `HEADING_EMA_ALPHA = 0.3` — слишком агрессивное сглаживание. При быстром развороте в термике компас будет отставать на ~1 секунду, что критично для направления на термик.

#### 6. **ThermalDetector — логические проблемы**
- `TH_SUSPECT = TH_THERMAL = 0.020f` — пороги одинаковые. Статус `SUSPECT` никогда не будет достигнут, если `THERMAL` срабатывает одновременно. В `processSample()`:
  ```java
  if (level > insideThresh) status = STATUS_INSIDE;
  else if (level > thermalThresh) status = STATUS_THERMAL;
  else if (level > suspectThresh) status = STATUS_SUSPECT;
  ```
  При `TH_SUSPECT == TH_THERMAL` второй `else if` никогда не выполнится при `level > TH_THERMAL`. Но при `level == TH_THERMAL` первое условие `> thermalThresh` — false, второе `> suspectThresh` — true. То есть `SUSPECT` срабатывает только при точном равенстве. **Баг.**

#### 7. **CirclingManager — проблемы с ветром**
- Оценка ветра по дрейфу центров спиралей требует **минимум 2 полных спиралей**. На практике пилот может крутить 1.5–2 круга и уходить. Ветер не успеет оцениться.
- `windSpeedFromGps` — оценка по разнице GPS-скорости и воздушной 9.5 м/с. Это работает только при точном знании воздушной скорости, которая варьируется от 8 до 12 м/с в зависимости от крыла и нагрузки. **Нет настройки воздушной скорости в UI.**

#### 8. **Аудио — потенциальные проблемы**
- `VarioSoundManager` использует `AudioTrack.Builder()` с `AudioAttributes.USAGE_ALARM` — звук идёт через канал будильника. На некоторых устройствах это может игнорировать системную громкость медиа.
- Нет graceful degradation при `AudioTrack` creation failure (например, если аудиофокус занят другим приложением).

### 🟡 Важные доработки

#### 9. **Отсутствие настроек для пилота**
- Нет настройки **воздушной скорости** (жёстко 9.5 м/с).
- Нет настройки **чувствительности детектора** (пороги hardcoded).
- Нет настройки **языка TTS** (жёстко русский).
- Нет настройки **громкости варио** отдельно от системной.

#### 10. **BlindFlightMode — неполная реализация**
- В коде `BlindFlightMode.java` не был загружен, но в `MainActivity` есть вызов `blindFlightMode.processSample()`. Если этот модуль не реализован полностью, слепой режим может работать некорректно.
- Gravity tilt compensation упоминается, но без кода сложно оценить корректность.

#### 11. **Логирование и хранение**
- Логи пишутся в `getExternalFilesDir(null)` — это `/Android/data/com.termo1.radar/files/`, доступное только приложению. Но в README указан путь `Android/data/com.termo1.radar/files/logs/`. На Android 11+ (API 30+) `MANAGE_EXTERNAL_STORAGE` может потребоваться для доступа пользователя.
- Нет автоматической очистки старых логов — диск забивается.

#### 12. **UI/UX**
- Радар рисуется на `Canvas` вручную — это ок для прототипа, но для production лучше `OpenGL ES` или хотя бы `SurfaceView` с отдельным рендер-тредом. Сейчас рендеринг 30 FPS в UI-треде + обработка сенсоров 50 Гц в том же треде = потенциальные пропуски кадров.
- Нет индикации **качества GPS-сигнала** (количество спутников, DOP).
- Нет индикации **уровня заряда батареи** — критично для 3-часового полёта.

#### 13. **Термик-радар: физическая модель**
- Формула дистанции `dist = 150 × sqrt(0.05 / rmsMs²)` — эмпирическая. Нет валидации на реальных полётах с известными термиками (например, сравнение с данными с других пилотов или инструментов).
- Радар показывает **только один термик** (`currentBlip`). В реальности может быть несколько термиков в поле зрения. `THERMAL_LIMIT = 12` в `MainActivity`, но `ThermalDetector` создаёт только один `currentBlip`.

#### 14. **Симуляция**
- `SimulationManager` генерирует детерминированный seed (42) — хорошо для воспроизводимости, но плохо для тестирования edge cases.
- Нет симуляции **реальной турбулентности** (только синусоидальные пыхи). Реальная турбулентность имеет более сложный спектр.

## Заключение

**ThermalRadar v0.1.0** — это **очень перспективный проект** с продуманной алгоритмической базой. Для пилота-парапланериста он может быть полезен как:
- **Вспомогательный инструмент** для поиска термиков в начале полёта
- **Тренажёр** центровки (голосовые подсказки по секторам)
- **Логгер** полёта с высокой частотой дискретизации

**Однако для production-использования необходимо:**
1. Исправить thread-safety issues
2. Оптимизировать рендеринг (убрать allocation в onDraw)
3. Добавить калибровку под конкретное крыло и стиль полёта
4. Провести валидацию на реальных полётах с эталонными приборами (например, XCTracer, Flymaster)

Проект заслуживает внимания и дальнейшего развития. Рекомендую автору сфокусироваться на стабилизации и тестировании в реальных условиях перед добавлением новых фич.

Анализ от квена:
Изучив предоставленную информацию о репозитории **ThermalRadar**, я проанализировал проект с двух точек зрения: как пилот-парапланерист и как Android-разработчик. Проект вызывает огромное уважение: это не просто «еще один вариометр», а глубокая попытка оцифровать «чувство крыла» с помощью DSP (цифровой обработки сигналов).

Ниже представлен подробный разбор полезности и зон роста.

---

### 🪂 Взгляд Пилота-парапланериста

#### Полезность прибора (Сильные стороны)
1. **Поиск ядра через микрораскачку (Кивки крыла):** Это «святой грааль» для пилотов, которые пока не умеют считывать тактильные отклики крыла. При входе в термик крыло кивает вверх (pitch up), при выходе — опускается. Привязка этих колебаний к компасу для вывода вектора на радар — гениальная идея, которая может радикально ускорить прогресс новичков и помочь опытным пилотам в «нулевых» термиках.
2. **Голосовой интерфейс (TTS) и Слепой режим:** В термике смотреть на экран некогда и опасно (риск столкновения). Голосовые подсказки («ядро на ветер», «сектор 45°») и режим «в кармане» с тактильной отдачей (вибрация) — это именно то, что отличает авиационный инструмент от игрушки.
3. **Расчет ветра по дрейфу спиралей:** Отличная функция, которая обычно доступна только в топовых вариометрах (Flymaster, Flytec, Naviter). Понимание сноса критически важно для принятия решений на переходе.
4. **Звук Brauniger IQ:** Тёплый ламповый звук классического вариометра, реализованный через `AudioTrack`, снизит когнитивную нагрузку для пилотов старой школы, привыкших к этому пику.
5. **Фильтрация ложных срабатываний (SNR > 3 и TH_SUSPECT):** Автор уже столкнулся с главной проблемой акселерометров — шумом при спиралировании в роторе или при активном снижении, и грамотно отсек её пороговыми значениями.

#### Что надо доработать (Пилотская физика и UX)
1. **Точка крепления (Критично!):** В документации упомянута «калибровка наклона на груди». Грудь пилота работает как амортизатор: она демпфирует высокочастотные кивки крыла и добавляет паразитные колебания от дыхания и турбулентности.
   * *Решение:* Добавить алгоритмы компенсации задержки (delay compensation) или поддержать крепление на шлем / свободные концы (райзеры), где IMU-датчики телефона будут ближе к реальным осям крыла.
2. **Логика входа/выхода из термика:** Алгоритм должен четко дифференцировать фазу *входа* в термик (крыло кивает вперед/вверх) и *выхода* (крыло проваливается). Если радар перепутает эти фазы, он будет указывать ядро в противоположной стороне (в сторону ротора/снижения).
3. **Безопасность и Трафик:** Смотреть на экран радара в плотном потоке опасно. В идеале нужно добавить парсинг NMEA-строк от внешних GPS (или OpenNET) для отображения других пилотов на радаре.
4. **Автономность:** Полет с включенным GPS, экраном (даже на 15%) и опросом датчиков на 50 Гц высадит батарею смартфона за 1.5–2 часа. Для XC-полетов этого мало.

### 📱 Взгляд Android-разработчика

#### Архитектура и Код (Сильные стороны)
1. **Математика и DSP:** Использование полосового фильтра Баттерворта (0.25–2.5 Гц) в связке с RMS-окном и детектором пересечения нуля (Zero-Crossing) — это профессиональный инженерный подход к выделению периодического сигнала из шума дешевых MEMS-акселерометров.
2. **Монотонное время (`elapsedRealtime`):** Огромный плюс. Многие разработчики GPS-трекеров используют `System.currentTimeMillis()`, из-за чего лог ломается при скачках NTP или уходе устройства в сон. Использование `elapsedRealtime` гарантирует, что `dtMs` всегда будет корректным для интегрирования и фильтрации.
3. **Разделение ответственности:** Четкая структура пакетов (`core`, `flight`, `sensors`, `gps`) и паттерн State Machine (`FlightStateMachine`) говорят о зрелом подходе к проектированию.
4. **Сенсорный тест и Симуляция:** Наличие 75-секундной симуляции и тестового режима с real-time фидбеком позволит отлаживать алгоритмы без необходимости каждый раз прыгать с горки или висеть на турнике.

#### Что надо доработать (Технический долг и Android API)
1. **Foreground Service (Критично для Android 8.0+):** В требованиях указан API 26+. Если приложение работает просто как `Activity` или обычный `Service`, система **убьет его** через 10–15 минут из-за Doze Mode и ограничений фоновых процессов.
   * *Решение:* Обязательно реализовать `ForegroundService` с постоянным уведомлением и разрешением `FOREGROUND_SERVICE_LOCATION`. Для Android 14+ потребуются специфичные типы Foreground Service и запрос `ACCESS_BACKGROUND_LOCATION`.
2. **Переход на Kotlin + Coroutines / Flow:** Java-код с ручным управлением потоками и коллбэками `SensorEventListener` подвержен утечкам памяти и гонкам данных. Реактивные потоки (Kotlin Flow) идеально ложатся на стримы данных с датчиков, позволяя легко применять операторы `debounce`, `filter` и `map` для обработки сигналов.
3. **Sensor Batching и Оптимизация I/O:** Логирование 50 Гц в `Samples.csv` и `Events.csv` в реальном времени создаст колоссальную нагрузку на I/O подсистему.
   * *Решение:* Использовать `registerListener` с параметром `maxReportLatency` (Sensor Batching), чтобы система отдавала данные пачками, и писать лог через выделенный буфер в памяти с асинхронным сбросом на диск (например, через `Dispatchers.IO` или `ExecutorService` с очередью).
4. **Сенсорная фильтрация (Sensor Fusion):** Сырой магнитометр в горах и на фоне электроники телефона дает сильный дрифт и шумы. Для точного Heading (курса) на радаре лучше использовать `Sensor.TYPE_ROTATION_VECTOR` (встроенный Android Sensor Fusion) или реализовать свой фильтр Калмана/Маджвика, учитывающий крены подвески.
5. **UI / Рендеринг:** Отрисовка радара через `Canvas` в `onDraw` может вызывать просадки FPS при частых обновлениях. Стоит рассмотреть переход на **Jetpack Compose** с использованием `Canvas` API внутри Compose — это даст декларативный подход, лучшую производительность и упростит адаптацию HUD под разные диагонали и складные экраны.

### 🏆 Итоговое заключение

**ThermalRadar** — это невероятно амбициозный и технически подкованный проект. Автор не просто «написал приложение, которое пищит при наборе высоты», а решил сложную задачу биомеханики и DSP: попытался оцифровать тактильные ощущения пилота через шумный сенсор потребительского смартфона.

**Вердикт о полезности:**
Прибор имеет потенциал стать **«убийцей бюджетных вариометров»**. Пилоты часто не хотят тратить $400-$600 на Flymaster/Flytec, если летают только по выходным. Если приложение сможет стабильно находить ядро по кивкам крыла и не высаживать телефон за час, оно станет маст-хэвом в арсенале любого пилота-любителя.

**Главный приоритет для следующего релиза (Roadmap):**
1. **Защита от убийства системой:** Внедрение `Foreground Service` и оптимизация батареи (иначе полет закончится раньше, чем термик).
2. **Физика подвески:** Корректировка алгоритмов с учетом того, что телефон на груди «отстает» от реакций крыла на 0.5–1.5 секунды.
3. **Unit-тесты для DSP:** Написать тесты для `SignalProcessor`, прогоняя через них записанные CSV-логи реальных полетов, чтобы гарантировать, что фильтр Баттерворта не «режет» полезные сигналы на разных моделях телефонов.

# Заключение по репозиторию ThermalRadar (TERMO1 Radar v0.1.0)

## Что это за прибор (кратко)

Android-приложение для парапланеризма, превращающее телефон в «искатель термиков». Принцип — нетривиальный: по **микрораскачке крыла**, уловленной акселерометром телефона на груди пилота, определяется направление на термик. Дополнительно: баровариометр, компасный радар с кольцами 50/100/125/150 м, детекция крутки по гироскопу, оценка ветра по дрейфу GPS-центров спиралей, голосовые подсказки (TTS), логирование 50 Гц в ZIP-чанки.

Стек: Java, Android 8+, LINEAR_ACCELERATION/FusedLocationProvider, Canvas-рендеринг, AudioTrack для звука варио. Один автор, 0 звёзд, создан 2025-05-25 → **очень свежий, экспериментальный проект**.

## 1. Заключение с точки зрения пилота

### Полезность — концепция сильная, но требует валидации

**Что по-настоящему ценно:**
- Сама идея «термик-сниффера» по микрораскачке — научно обоснована (крыло раньше начинает качать со стороны термика), но почти не реализована в массовом софте. Если алгоритм реально работает в воздухе, это качественное преимущество перед обычными варио.
- Комбинация «радар + варио + голос + центровка + ветер» в одном бесплатном приложении — привлекательна для пилотов без дорогих приборов (Flymaster/Naviter/Oudie).
- Голосовые подсказки «ядро на ветер / по ветру / N-E-S-W» после каждого круга — правильная парадигма: пилот в спирали смотрит в центр, а не на экран.
- Слепой режим (15% яркости) + голос — грамотно для кармана, экономит батарею и не слепит в сумерках.
- Кнопка СТОП с подтверждением — защита от случайного прерывания лога в тербулентности (перчатки, тряска).
- Лог Events.csv с монотонным временем (elapsedRealtime) — отличная инженерная практика, без скачков от NTP.

### Критичные пилотские замечания

1. **Достоверность самого метода под вопросом.** Реальная турбулентность omnidirectional; качание тела в подвеске, работа клевантами, ускорителем, шаги при разбеге — всё попадает в полосу 0.25–2.5 Гц. Верификация «level вырос в 1.2× за 2 периода» отсекает часть ложняков, но в реальном воздухе это нужно проверять на десятках полётов. Риск ложных направлений, уводящих пилота от термика.
2. **Дальность 50/100/125/150 м — ложная точность.** Формула `dist = 150 × sqrt(0.05 / rms)` — эвристика, не подтверждённая физикой конкретного термика (размер, сдвиг ветра, lapse rate). Надо подписать «оценочно» и не показывать как истину.
3. **Нет AGL по рельефу.** AGL упомянут, но без цифровой модели рельефа (DEM) это просто GPS-высота над непонятно чем. В горах/у склона это **safety-critical пробел** — нельзя судить о высоте над землёй.
4. **Нет воздушного пространства (CTR, зоны).** Как primary-прибор непригоден — нет airspace/waypoints. Только как доп. «сниффер» к основному прибору.
5. **Нет экспорта IGC/KML/GPX.** Только CSV — пилот не сможет выгрузить трек в XContest/Leonardo/CMMC для анализа. Нужен конвертер.
6. **Порог крутки 18°/с.** Параплан в спирали обычно 12–20 с/круг = 18–30°/с — порог на грани. **Плоские/широкие спирали (>20 с/круг) не детектируются** — а в слабых широких термиках именно так и крутят.
7. **Секторный анализ всего 4 сектора по 90°.** Грубо для центровки. «Лучший сектор» по среднему варио может исказиться одним сильным bump-ом.
8. **Ветер: airspeed 9.5 м/с захардкожен.** Trim-speed зависит от загрузки крыла, скорости, ускорителя. Оценка ветра по прямой (GPS−airspeed) будет ошибаться на величину этой погрешности.
9. **Магнитный vs истинный север.** В коде направление/ветер — магнитные. Пилоты работают с картой в истинном севере. Нет упоминания магнитного склонения (`GeomagneticField`). Возможна путаница «ветер 270°» — магнитный или истинный?
10. **Маршрутизация звука в шлем.** Не упомянут audio focus / routing в BT-гарнитуру шлема. Если TTS или звук варио перебивается или не идёт в шлем — прибор бесполезен в воздухе.
11. **Единственная точка отказа.** Один телефон = при сбое теряешь всё. Приложение должно позиционироваться как **дополнение** к основному варио, не замена. В README этого честного предупреждения нет.
12. **Батарея.** 50 Гц сенсоры + GPS HIGH_ACCURACY + барометр + TTS + экран 3+ часа горного полёта — тяжело. Нет упоминания power-management.
13. **Русский только.** TTS и UI — только RU. Сужает аудиторию до СНГ.

**Вердикт пилота:** концепция перспективная и бесплатно, но **доверию как primary-прибору не подлежит** до массовой лётной валидации. Сейчас это «термик-сниффер второго эшелона» — дополнение к основному варио/GPS, интересный эксперимент. Реальная польза появится, когда наберётся статистика «направление с радара vs реальный подъём».

## 2. Заключение с точки зрения Android-разработчика

### Архитектура — главные риски

1. **🚨 Нет Foreground Service — критично.** Сенсоры, логирование и детекция привязаны к `MainActivity` («главный экран, сенсоры, рендеринг»). Android убьёт фоновые сенсоры в Doze и при выключенном экране. А именно в **слепом режиме** (экран 15%/выкл) приложение должно работать. Без foreground service + `PARTIAL_WAKE_LOCK` + persistent notification оно **молча перестанет писать лог**. Это архитектурный blocker №1.
2. **🚨 Нет WakeLock.** Без `PARTIAL_WAKE_LOCK` CPU спит, `SensorManager` перестаёт выдавать 50 Гц. Слепой режим фундаментально не работает.
3. **🚨 Storage на Android 11+.** Путь `{ExternalStorage}/logs/` — legacy. На Android 10+ Scoped Storage это сломается без `MANAGE_EXTERNAL_STORAGE` (Play Store такое отклонит). Надо `getExternalFilesDir(null)/logs` — без разрешений.
4. **God-Activity.** `MainActivity` совмещает сенсоры, рендеринг, UI — не масштабируется. Разнести: `FlightService` (сенсоры+детекция+лог, жизненный цикл независим от UI) + тонкий UI-слой.
5. **Нет восстановления после краша.** Текущий ZIP-чанк теряется при crash. Нужен `try/finally` + periodic flush + shutdown hook.

### Корректность алгоритмов

6. **Коэффициенты фильтра захардкожены под 50 Гц.** `SignalProcessor` использует константы Баттерворта, посчитанные для fs=50 Гц. Но Android **не гарантирует** ровно 50 Гц для `LINEAR_ACCELERATION` — реальная частота зависит от устройства (50/100/200/…). При другой fs полоса 0.25–2.5 Гц **сдвигается**, детекция деградирует. Надо: измерять реальный dt и пересчитывать коэффициенты, либо ресэмплировать в фиксированные 50 Гц.
7. **Численная устойчивость ФВЧ 0.25 Гц.** Коэффициент `HP_A1 = -1.9555` близок к единичной окружности → фильтр чувствителен. На `float` за часы полёта возможен дрейф состояния. Anti-windup clamp ±100 — пластырь. Лучше `double` для state или периодический reset.
8. **Усреднение угла ветра через EMA по скаляру.** Классический баг у 0/360: `windFromDeg += alpha*(new-from)` ломается на перепаде 359°→1°. В выгрузке кода операторы сравнения «съелись» HTML-парсером, поэтому точно не проверить, но нужен **векторный метод** (sin/cos среднее), а не скалярный.
9. **Усреднение lat/lon как плоских чисел** для центроида спирали — на малых дистанциях (<1 км) допустимо, но корректнее — проекция в локальный ENU/NED. Минор.
10. **`volatile` без атомарности.** `currentBlip` / `lastBlipUpdateMs` volatile — видимость есть, но составная проверка из двух полей потоково-небезопасна (UI читает, сенсор-поток пишет). Лучше один immutable snapshot или `synchronized`. Минор, но в safety-critical софте важно.

### Android-платформа / качество

11. **Нет модульных тестов.** Чистые функции (фильтр, RMS, ZC, haversine, bearing, wind EMA) тривиально тестируются синтетическими сигналами (синус известной частоты, известный дрейф). Для safety-relevant приложения это серьёзный пробел. Особенно учитывая, что SIMULATION.md уже описывает тестовые сценарии — их надо автоматизировать в JUnit.
12. **Нет CI.** Не видно GitHub Actions / lint-конфига. Нужен CI: lint + test + assembleRelease.
13. **Permissions.** Не видно обработки runtime-permissions: `ACCESS_FINE_LOCATION`, `FOREGROUND_LOCATION` (service), `POST_NOTIFICATIONS` (Android 13+), запрос на отключение battery-optimization (Doze白名单) для 50 Гц сенсоров.
14. **targetSdk.** Не видно; для Play Store нужно 34+. Высший targetSdk накладывает ограничения на запуск foreground service (Android 12+) и runtime-уведомления — нужно корректно обработать.
15. **Audio focus.** TTS + AudioTrack варио должны запрашивать `AudioManager.requestAudioFocus` и корректно реагировать на `AUDIOFOCUS_LOSS` (звонок, другое приложение). Не видно.
16. **Магнитное склонение и помехи.** Не используется `GeomagneticField`. Нет детекции магнитных аномалий (близость металла подвески, тросса, ЛЭП) — магнитометр может врать сильно, а от него зависит и компас, и матрица поворота R.
17. **I18n.** Все строки в коде по-русски — надо `strings.xml` с локалями.
18. **Распространение.** APK в Releases, sideload. Нет Play/F-Droid → нет автообновлений у пользователей.
19. **Деградация при отсутствии сенсоров.** README говорит «барометр/магнитометр/GPS опциональны», но fallback описан размыто. Нужны явные in-app предупреждения: «Нет барометра — варио отключён», «Нет магнитометра — направление может быть ошибочным».
20. **Airspeed 9.5 м/с как константа.** Должен быть настраиваемым (загрузка крыла, speedbar). Иначе ветер по прямой считается с систематической ошибкой.
21. **Memory/GC.** Чанк 10 мин × 50 Гц × 30 полей ≈ 30 000 строк держится в памяти до ZIP. Нормально, но лучше стримить в буфер/файл инкрементно, чтобы снизить GC-давление.

## Итоговый вердикт

**Полезность прибора:** концепция термик-сниффера по микрораскачке — **перспективная и малозанятая ниша**, бесплатная альтернатива дорогим приборам привлекательна. Но на v0.1.0 это **экспериментальный прототип**, который **нельзя считать primary-прибором**: нет AGL по рельефу, нет воздушного пространства, нет IGC-экспорта, метод детекции не валидирован в массовых полётах. Реальное место — **доп. «сниффер» к основному варио**, интересный для пилотов-энтузиастов, готовых тестировать.

**С точки зрения кода:** алгоритмическая база (фильтр, детектор, центровка, оценка ветра) проработана вдумчиво и хорошо документирована — чувствуется понимание DSP и парапланерной физики. Но **Android-инфраструктура критически отстаёт**: отсутствие Foreground Service + WakeLock + правильного storage — это не «детали для следующей версии», а **блокеры, из-за которых приложение молча перестанет работать в реальном полёте** (особенно в слепом режиме). Пока эти 3–4 архитектурных пробела не закрыты, лётная пригодность под большим вопросом.

**Рекомендация:** автору — закрыть блокеры из красной зоны (foreground service, wake-lock, scoped storage, flush-on-crash) и провести серию тестовых полётов с синхронной записью эталонного варио/GPS для верификации детектора. После этого проект имеет реальный потенциал стать полезным инструментом в нишевом, но лояльном сообществе парапланеристов.
