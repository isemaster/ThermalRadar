## 🔴 1. Критическая ошибка Android: Почему чужие треки не играются?

### Проблема: `FileInputStream` и Android URI
В `TrackReplayer.java` метод загрузки файла выглядит так:
```java
public void loadFile(String filePath) {
    try {
        loadFromIGC(new java.io.FileInputStream(filePath), false);
    } catch (Exception e) {
        // ... fallback to synthetic track
    }
}
```
**Суть бага:** В современном Android (API 26+, Scoped Storage) при выборе файла через системный пикер (`Intent.ACTION_GET_CONTENT` или `ActivityResultContracts`) вы получаете **`Uri`** (например, `content://com.android.providers.media.documents/...`), а не абсолютный путь в файловой системе (`/storage/emulated/0/...`).
Когда студент передает `uri.toString()` или `uri.getPath()` в `new FileInputStream()`, Java кидает `FileNotFoundException` или `IllegalArgumentException`. Срабатывает `catch`, и плеер молча подсовывает `generateFallbackTrack()`. Пользователь видит синтетический трек и думает, что его файл "сломан".

**✅ Как исправить:**
Необходимо работать с `Uri` через `ContentResolver`.
```java
public void loadFromUri(Context context, Uri uri) {
    try (InputStream is = context.getContentResolver().openInputStream(uri)) {
        loadFromIGC(is, false);
    } catch (Exception e) {
        Log.e(TAG, "Failed to open IGC URI", e);
    }
}
```

## 🔴 2. Фатальная ошибка навигации: Пересечение полуночи (Midnight Crossing)

### Проблема: Абсолютное время вместо дельт
```java
float timeSec = hh * 3600f + mm * 60f + ss; // Секунды от начала суток
// ...
float trackTime = track.get(0).timeSec + totalSimSec;
```
**Суть бага:** Время в B-записи парсится как секунды от 00:00:00. Если полет начался в 23:30 (`84600` сек) и закончился в 01:00 (`3600` сек), то в массиве `track` время пойдет: `84600, 84601 ... 86399, 0, 1 ... 3600`.
Но `trackTime` будет монотонно расти: `84600 + totalSimSec`. Как только `totalSimSec` перевалит за 30 минут, `trackTime` станет `86400+`.
Цикл поиска сегмента:
```java
if (trackTime >= track.get(i).timeSec && trackTime < track.get(i + 1).timeSec)
```
**мгновенно сломается**, так как `86400` не находится между `86399` и `0`. Плеер зависнет или завершится.

**✅ Как исправить:**
IGC-парсер должен читать `HFDTE` (дату) и формировать монотонный `long timestampMs` (Unix Epoch). Либо, что проще для реплеера, хранить `deltaTime` (разницу в секундах между соседними точками) и накапливать её, игнорируя абсолютное время суток.

## 🟠 3. Ошибки авионики и математики

### 3.1. Расчет курса без учета схождения меридианов
```java
double dLat = pilotLat - prevLat;
double dLon = pilotLon - prevLon;
// ...
float newHeading = (float) Math.toDegrees(Math.atan2(dLon, dLat));
```
**Суть бага:** Это формула для плоской земли. На широте Москвы (55°) или Санкт-Петербурга (60°) 1 градус долготы физически короче 1 градуса широты почти в 2 раза (`cos(60°) = 0.5`). `Math.atan2(dLon, dLat)` даст **чудовищную ошибку в курсе** (вплоть до 30-45 градусов) при полете на Восток/Запад.
**✅ Как исправить:**
```java
double dLonAdj = dLon * Math.cos(Math.toRadians(prevLat)); // Компенсация схождения
float newHeading = (float) Math.toDegrees(Math.atan2(dLonAdj, dLat));
```

### 3.2. Игнорирование GPS-высоты и хрупкий парсер
```java
int pressAlt = Integer.parseInt(line.substring(25, 30));
track.add(new TrackPoint(lat, lon, pressAlt, timeSec));
```
**Суть бага:**
1. Если у чужого логгера нет барометра, он пишет `00000` или пробелы. `Integer.parseInt` упадет с `NumberFormatException`, запись пропустится.
2. Если баро-высота `00000`, `vario` будет считаться от нуля, что убивает детекцию термиков.
3. Если строка короче 30 символов (нет GPS-высоты `GGGGG`), упадет `StringIndexOutOfBoundsException`.
**✅ Как исправить:** Парсить и `pressAlt` (25-29), и `gpsAlt` (30-34). Использовать GPS-высоту как фоллбэк, если баро-высота невалидна. Добавить проверки длины строки и `trim()`.

### 3.3. Деление на ноль и `NaN` координаты
```java
float segLen = p1.timeSec - p0.timeSec;
frac = (trackTime - p0.timeSec) / segLen;
```
**Суть бага:** В чужих IGC-файлах часто встречаются дублирующиеся таймштампы (например, при потере фикса или 10Hz логгерах). Если `segLen == 0`, `frac` становится `Float.NaN`. Далее `pilotLat` становится `NaN`, и OSM-карта крашится при попытке отрисовать маркер.
**✅ Как исправить:** `if (segLen < 0.001f) frac = 0;`

## 🟡 4. Архитектурные и логические проблемы

### 4.1. Нарушение стандарта IGC (Timezone)
В `IgcLogger.java`:
```java
private static final ThreadLocal<SimpleDateFormat> IGC_TIME_FMT_TL =
    ThreadLocal.withInitial(() -> new SimpleDateFormat("HHmmss", Locale.US));
```
В `TrackReplayer.java` студент пишет в комментариях: *"Формат времени в файле: UTC+4"*.
**Суть бага:** Стандарт IGC **СТРОГО** требует, чтобы все время было в **UTC**. Собственный логгер пишет локальное время (так как не вызван `setTimeZone(TimeZone.getTimeZone("UTC"))`), из-за чего треки не открываются в XCSoar/SeeYou. А в реплеере студент пытается "подогнать" время под MSK, что ломает логику.
**✅ Как исправить:** Жестко задать UTC во всех `SimpleDateFormat` и забыть про часовые пояса в навигации.

### 4.2. Изобретение велосипедов (Игнорирование собственных модулей)
В `TrackReplayer` написан свой хардкодный детектор крутки:
```java
boolean isCircling = headingAccum > CIRCLE_HEADING_ACCUM; // 360 градусов
```
**Суть бага:** `headingAccum` суммирует `Math.abs(delta)`. Если пилот летит прямо, но крыло болтает в турбулентности на ±5 градусов, `headingAccum` быстро наберет 360°, и плеер нарисует "красное ядро термика" на прямом участке!
При этом в проекте уже есть портированный **`CirclingManager`** (XCSoar CirclingComputer) и **`VarioThermalDetector`**.
**✅ Как исправить:** Удалить самописную логику из `TrackReplayer`. Генерировать `SensorEvent` и `Location` из B-записей и скармливать их в существующий `FlightStateMachine` и `CirclingManager`.

### 4.3. Утечка ресурсов
```java
BufferedReader br = new BufferedReader(new InputStreamReader(inputStream));
```
`InputStream` и `BufferedReader` не закрываются. При загрузке больших файлов (4+ часа полета) или частом выборе файлов через SAF это приведет к утечке файловых дескрипторов (`ParcelFileDescriptor`).
**✅ Как исправить:** Использовать `try-with-resources`.
