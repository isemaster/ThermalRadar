# TODO — ThermalRadar v0.1.0

По результатам рецензий от Квена, ГЛМ и третьего ревьюера (kimi.txt).
Приоритеты: 🔴 критично → 🟠 высокий → 🟡 средний → 🟢 низкий.

---

## 🔴 Критические

### 1. Race conditions — CirclingManager + ThermalBlip
`CirclingManager` обновляется из сенсорного потока (`update()`), а читается из UI-треда (`isCircling()`, `getWindFromDeg()`) — синхронизации нет.
`ThermalBlip` поля (`distance`, `strength`, `angle`) не immutables — UI читает, сенсорный поток пишет.

**Решение:** synchronized блоки или immutable snapshot объекты.

### 2. Paint создаётся в onDraw() — GC-паузы
```java
Paint windLine = new Paint(Paint.ANTI_ALIAS_FLAG);  // каждый кадр!
Paint windLabel = new Paint(...);
Paint thermalLabelPaint = new Paint(...);
```
На 30 FPS — 60+ аллокаций в секунду.

**Решение:** вынести все Paint в поля RadarView, инициализировать в конструкторе.

### 3. TH_SUSPECT == TH_THERMAL — SUSPECT недостижим
Оба порога = 0.020f. При `level > 0.020f` срабатывает `THERMAL`. SUSPECT срабатывает только при `level == 0.020f` (практически никогда).

**Решение:** сделать `TH_SUSPECT = 0.010f` или 0.015f.

### 4. Стрелка ветра привязана к телефону, а не к сторонам света
Стрелка ветра рисуется в `onDraw()` после `radarRenderer.draw()`, где canvas уже `restore()` — то есть в ориентации телефона, а не с вращением по heading.
Ветер определяется как абсолютный (градусы компаса), а рисуется в screen-координатах без поправки на heading. На экране стрелка стоит на месте при повороте телефона, хотя должна вращаться вместе с N/S/W/E.

**Решение:** переместить отрисовку стрелки ветра внутрь `RadarRenderer.draw()`, до `canvas.restore()`, чтобы она попала в north-up систему координат.

### 5. Приложение не работает после нажатия кнопки питания (выключения экрана)
Сенсоры, детекция и лог привязаны к `MainActivity`. При выключении экрана Android приостанавливает Activity. В Doze режиме — сенсоры отключаются, лог не пишется, TTS не работает.
Пользователь ожидает: нажал кнопку → экран погас → прибор продолжает писать лог и давать голосовые подсказки в кармане (слепой режим).

**Решение:** вынести сенсоры, детекцию, лог и TTS в `ForegroundService` с `PARTIAL_WAKE_LOCK`, persistent notification и запросом `REQUEST_IGNORE_BATTERY_OPTIMIZATIONS`. Activity — только UI-слой.

## 🟠 Высокий приоритет

### 4. Настройка воздушной скорости
Airspeed захардкожен 9.5 м/с. Реальная скорость зависит от крыла, загрузки, ускорителя (8–12 м/с). Оценка ветра по прямой на прямой `|GPS − 9.5|` даёт систематическую ошибку.

**Решение:** поле «Воздушная скорость» в настройках (Slider 8–15 м/с, шаг 0.5). Передавать в CirclingManager.

### 5. Усреднение направления ветра — баг 0/360
```java
windFromDeg += WIND_DIR_EMA_ALPHA * (newWindFrom - windFromDeg);
```
На перепаде 359° → 1° даст: `1 − 359 = −358`, EMA сдвинет на −358×0.3 ≈ −107° — резкий скачок.

**Решение:** заменить на векторное усреднение (sin/cos среднее) или корректную нормализацию разницы.

### 6. Адаптивная частота сэмплов
Коэффициенты Баттерворта посчитаны для fs = 50 Гц. Android не гарантирует ровно 50 Гц для LINEAR_ACCELERATION. Реальная частота может быть 50/100/200 Гц в зависимости от устройства → полоса пропускания 0.25–2.5 Гц сдвигается.

**Решение:** измерять реальный dt между сэмплами, пересчитывать коэффициенты фильтра или ресэмплировать до 50 Гц.

### 7. Магнитное склонение
Направления (heading, ветер) — магнитные. На картах пилоты работают в истинном севере.

**Решение:** добавить `GeomagneticField` для пересчёта, опция «магнитный / истинный север» в настройках.

### 8. AGL — честная подпись
AGL считается как `gpsAlt − startAlt`. В горах/у склона это не высота над землёй, а высота над точкой старта. Safety-critical для пилота.

**Решение:** подписать «над стартом», а не «над землёй». Или встроить DEM (долгая задача — отложить).

## 🟡 Средний приоритет

### 9. 8 секторов вместо 4
4 сектора по 90° — грубо для центровки. Один сильный bump может исказить среднее.

**Решение:** разделить на 8 секторов по 45° (N/NE/E/SE/S/SW/W/NW) с соответствующими голосовыми подсказками.

### 10. Порог крутки в настройки
18°/с захардкожен. Параплан в широкой спирали (>20 с/круг) даёт ~18°/с — на грани срабатывания. Плоские термики не детектятся.

**Решение:** ползунок «чувствительность крутки» (10–30°/с) в настройках, передавать в CirclingManager.

### 11. Экспорт в IGC / KML / GPX
Логи только в CSV. Пилот не может выгрузить трек в XContest / Leonardo.

**Решение:** конвертер (отдельный инструмент или встроенная кнопка). Для начала — простой скрипт на Python, потом в приложение.

### 12. Индикация GPS + батареи на HUD
Нет количества спутников, DOP, уровня заряда. Для 3-часового полёта критично.

**Решение:** добавить иконку батареи + кол-во спутников (или fixAge) на экран.

### 13. Множественные термики
`ThermalDetector` держит только один `currentBlip`. В поле зрения может быть несколько термиков. `THERMAL_LIMIT = 12` есть, но не используется полноценно.

**Решение:** хранить список активных `currentBlip`, каждый со своим направлением/силой.

## 🟢 Низкий приоритет / Полировка

### 14. Foreground Service + WakeLock для слепого режима
Сейчас сенсоры/лог привязаны к `MainActivity`. В Doze Android убьёт фоновые процессы. Для слепого режима (экран выключен) это фатально.

**Решение:** вынести сенсоры, детекцию и лог в `ForegroundService` с `PARTIAL_WAKE_LOCK` и уведомлением.

### 15. Flush-on-crash
Текущий ZIP-чанк теряется при падении приложения.

**Решение:** `try/finally` + периодический flush на диск (каждые N сэмплов или по таймеру).

### 16. Audio focus и BT-routing в шлем
TTS + AudioTrack варио не запрашивают `AudioManager.requestAudioFocus`. При звонке или уведомлении — конфликт звука. Нет явного роутинга в BT-гарнитуру шлема.

### 17. Sensor Fusion — Rotation Vector вместо компаса
`SensorManager.getRotationMatrix()` требует калибровки магнитометра. В полёте магнитные помехи от подвески, рации, ЛЭП.

**Решение:** использовать `TYPE_ROTATION_VECTOR` (встроенный Android Sensor Fusion) или фильтр Маджвика.

### 18. Модульные тесты для DSP
`SignalProcessor`, `ThermalDetector`, `CirclingManager` — чистые функции. Тривиально тестируются синтетическими сигналами. Нет ни одного юнит-теста.

### 19. Double для state фильтра
Коэффициент ФВЧ 0.25 Гц близок к единичной окружности. На float за часы полёта — дрейф. Anti-windup ±100 — пластырь.

### 20. Численная устойчивость Баттерворта
`HP_A1 = −1.9555`. На float возможно накопление ошибки за длительный полёт.

### 21. Формула дистанции — эвристика
`dist = 150 × sqrt(0.05 / rmsMs²)`. Не валидирована на реальных полётах. Кольца 50/100/125/150 м создают ложное ощущение точности.

**Решение:** подписать «оценочно» на радаре.

### 22. I18n / локализация
Все строки в коде — русские. TTS — только русский. Нет `strings.xml` с переводами.

### 23. CI — GitHub Actions
Нет автоматической сборки + линтинга + тестов.

### 24. ProGuard / R8 config для release
Сейчас release собирается с ProGuard по умолчанию. Нет кастомных правил для сохранения классов, используемых через рефлексию.
