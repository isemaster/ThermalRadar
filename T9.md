## 0. Краткое резюме (TL;DR)

Проект производит **очень сильное для студента впечатление**: реализованы порты ключевых алгоритмов XCSoar (`CirclingComputer`, `ThermalLocator`, `LiftDatabase`, `VarioSynthesiser`), EKF для ветра (названный EKF, но по факту не EKF — см. §6), IGC-логгер, фон-сервис, аудио-варио на `AudioTrack`. Это не «лаба на коленке», а осмысленный продукт.

Одновременно с этим код **категорически не готов к production-полёту**. Основные причины:

1. **Дрожание компаса на столе ±5° при резком повороте — плавно.** Это не «особенность сенсора», это прямое следствие неправильного выбора fusion-датчика, неверных коэффициентов α-β фильтра и слишком маленького deadband. Подробный разбор — §3.
2. **Архитектурный долг:** `MainActivity.java` — 2712 строк «God Object», в котором смешаны отрисовка Canvas, бизнес-логика, sensor wiring, GPS-трек, TTS, диалоги и кнопки. Это критическая проблема сопровождаемости.
3. **Нарушения потокобезопасности:** `rotationMatrix` — общий `float[9]`, который мутируется в sensor-треде и читается в UI-треде **без барьера**. При определённых таймингах можно прочитать наполовину обновлённую матрицу.
4. **Аллокации в `onDraw`:** `new Paint()`, `new float[2]`, `new float[...]` на каждой точке трека в каждом кадре → GC-pressure 30–60 раз в секунду.
5. **«EKF», который не EKF:** в `WindEKF.java` нет ковариационной матрицы, нет prediction step. Это градиентный спуск. Название вводит в заблуждение.
6. **Дед-код и рассинхрон настроек:** `vario_smooth` меняется в настройках, но в рантайме не применяется. `onPressureSample()` в `SensorController` — мёртвый код-дубликат `VarioManager`.
7. **Magnetometer calibration отсутствует.** Нет ни hard-iron, ни soft-iron. На столе у телефона магнитный фон искажён железом стола/ноутбука → плюс к джиттеру.

**Вердикт:** проекту нужен 1–2 спринта рефакторинга с упором на sensor fusion и разбор `MainActivity` на слои. После этого — полевые испытания.

## 2. Главная жалоба: компас дрожит на столе ±5°, при резком повороте — плавно

### 2.1. Симптом

> Прибор лежит на столе. Стрелка компаса дрожит в пределах 5 градусов. При резком повороте — компас плавно поворачивается.

Этот симптом — **классическая карта отказов α-β фильтра с завышенным α и слишком малым deadband**. «Плавность при резком повороте» объясняется тем, что в динамике дисперсия сенсора маскируется большим `vk` (turn-rate), а в статике `vk` ≈ 0 и весь шум сенсора уходит в `xk`.

### 2.3. Корневые причины (root causes)

#### RC-1. Используется наихудший из доступных fusion

```java
// SensorController.java:375
if (SensorManager.getRotationMatrix(rotationMatrix, null, gravBuf, geomBuf)) {
    SensorManager.getOrientation(rotationMatrix, orientationVals);
```

Здесь `gravBuf` берётся из `TYPE_ACCELEROMETER`, а не из `TYPE_GRAVITY`. Разница критическая:

- `TYPE_ACCELEROMETER` = gravity + linear accel. На столе микровибрации стола (вентилятор ноутбука, шаги в коридоре, HVAC) попадают в «gravity».
- `TYPE_GRAVITY` = низкочастотный фильтр акселерометра, который Android сам делает внутри `SensorFusion` без overhead.
- `TYPE_ROTATION_VECTOR` / `TYPE_GEOMAGNETIC_ROTATION_VECTOR` — fusion gyro+accel+mag, сглаженный ядром Android. Это **де-факто стандарт** для компасов с 2014 года.

В коде ни разу не запрашивается `TYPE_ROTATION_VECTOR`, при том что заявлена поддержка API 26+. Это означает, что приложение делает вручную то, что ядро Android делает лучше.

#### RC-2. α = 0.75 — это почти pass-through

```java
// HeadingFilter.java:48
public HeadingFilter() {
    this(0.75, 0.3);
}
```

Формула α-β:

```java
// HeadingFilter.java:122
xk = xkPred + alpha * innovation;
vk = vkPred + beta * innovation / dt;
```

При α = 0.75 фильтр доверяет новой measurement на 75 % и prediction — на 25 %. Шум сырого magnetometer имеет σ ≈ 1.5–3° на типичном MEMS-сенсоре телефона. После α-β с α=0.75 остаточный шум σ_out ≈ √(α²·σ² + (1-α)²·σ²) ≈ σ·0.79. То есть фильтр **гасит шум всего на ~21 %**.

Реалистичные значения для параплана на 50 Гц:
- α ∈ [0.05, 0.2] — для статичного HUD
- β ∈ [0.005, 0.02] — для turn-rate

В авиационной практике (AHRS) типичный коэффициент обновления для magnetometer в fusion — 0.05–0.1, для gyro — 0.5–0.9. Здесь же автор перепутал роли: 0.75 — это «коэффициент доверия к сенсору», и он должен быть маленьким для зашумлённого магнитометра.

#### RC-3. Deadband = 0.5° — фактически отсутствует

```java
// HeadingFilter.java:36
private static final double DEADBAND_DEG = 0.5;
```

При σ_out ≈ 2° (после α-β с α=0.75) P(|display_diff| > 0.5°) ≈ 1.0. То есть deadband почти никогда не срабатывает. Стрелка перерисовывается в каждом кадре (30 FPS), даже если реальный heading не изменился.

Правильное значение для статичного компаса — 2–3°. Лучше — **адаптивный deadband**: больше, когда |vk| < 1°/с (статика), меньше — в движении.

#### RC-4. CLAMP не влияет на статику

```java
// HeadingFilter.java:85
double maxDelta = MAX_DELTA_DEG_PER_SEC * dt;  // 70 * 0.02 = 1.4° на сэмпл
if (Math.abs(diff) > maxDelta) {
    clamped = xk + Math.signum(diff) * maxDelta;
```

При 50 Гц порог срабатывания — 1.4° за сэмпл. Шум ±5° редко достигает 1.4° за один шаг (σ_step ≈ 0.5°). Поэтому в статике clamp почти не работает. А вот при резком повороте на 90°/с он включается и **именно он обеспечивает «плавность»**, которую видит пользователь. Это объясняет асимметрию симптома.

#### RC-5. Отсутствует hard-iron / soft-iron калибровка магнитометра

В коде есть только `GeomagneticField.getDeclination()` (магнитное склонение — это другое):

```java
// SensorController.java:587
android.hardware.GeomagneticField gf = new android.hardware.GeomagneticField(
        (float) lat, (float) lon, 0f, now);
magneticDeclination = gf.getDeclination();
```

Это поправка между магнитным и истинным севером, **она не убирает локальные искажения** телефона (токопроводящие дорожки PCB, динамик, батарея). Без калибровки «восьмёркой» магнитометр имеет систематическую ошибку ±5–15°, **зависящую от самого heading** (sinusoidal). Эта ошибка на столе выглядит как дрожание, потому что шум сенсора модулируется этой синусоидой.

В авиацике это решается:
- Запросом `Sensor.TYPE_MAGNETIC_FIELD_UNCALIBRATED` → ручной эллиптической подгонкой (FitEllipsoid).
- Либо использованием `TYPE_ROTATION_VECTOR` (внутри Android уже откалибровано ядром).

#### RC-6. β·innovation/dt усиливает шум

```java
// HeadingFilter.java:123
vk = vkPred + beta * innovation / dt;
```

Деление на `dt` имеет смысл, когда `dt` постоянный и предсказуемый. На реальном магнетометре Android интервалы спама плавают (10–25 Гц вместо 50), и при `dt → 0.005 с` (200 Гц короткий всплеск) `vk` получает множитель 200, что мгновенно раздувает turn-rate в 10–100 раз. Дальше `xk + vk * dt` в следующем predict'е уносит heading в сторону.

Правильно: либо `vk += β · innovation` (без /dt, фиксируя dt в константу), либо преднастроить `dt` на номинал 1/50 и не менять.

#### RC-7. Render loop читает volatile heading без сглаживания отображения

```java
// MainActivity.java:2354
float headingDisplay = getCompassHeading();
// MainActivity.java:2367
radarRenderer.draw(canvas, nowMs, thermalsCopy,
        headingDisplay, varioDisplay, currentStatus, ...);

// RadarRenderer.java:329
canvas.rotate(-heading, cx, cy);
```

`headingDisplay` — это просто последнее значение `volatile float heading`. Между двумя кадрами (33 мс) значение может прыгнуть на 1–3°. `canvas.rotate` применяет это мгновенно — стрелка скачет.

Для плавности UI в авиацике делают **двойную буферизацию сглаживания**:
1. Внутренний фильтр (sensor → `headingFiltered`) — медленный, точный.
2. UI-фильтр (`headingDisplay = lerp(headingDisplay, headingFiltered, 0.15)`) — быстрый, сглаживает визуально.

В коде этого слоя нет.

### 2.4. Воспроизводимый эксперимент

Если воспроизвести симуляцию: положить телефон на стол, записать `rawHeading` и `heading` (filtered) 30 секунд, потом построить гистограмму.

Ожидаемый результат при текущей реализации:
- `rawHeading`: σ ≈ 2–3°, диапазон ±5°
- `heading`: σ ≈ 1.5–2.5°, диапазон ±4° (фильтр почти не подавляет)
- `headingDisplay` (30 Гц): каждый кадр отличается от предыдущего на 0.5–2° → визуальное дрожание

После применения рекомендаций из §2.5:
- `rawHeading`: σ ≈ 2–3° (не изменилось)
- `heading`: σ < 0.5°, диапазон ±1°
- `headingDisplay`: статика — 0 изменений в секунду; динамика — плавная интерполяция

### 2.5. Рекомендации по исправлению компаса (приоритет P0)

#### Fix B — пересчитать коэффициенты α-β (30 минут, эффект 15%)

```java
// Deadband — увеличить и сделать адаптивным
```

#### Fix C — добавить UI-слой сглаживания (15 минут, эффект 5%)

### 2.6. Почему «при резком повороте — плавно»

Это **побочный эффект трёх факторов**:

1. `MAX_DELTA_DEG_PER_SEC = 70` — clamp ограничивает производную, поэтому при повороте 180°/с реальный heading отстаёт и интерполируется.
2. В динамике `vk` (turn rate) становится большим (20–50°/с), и `xk_pred = xk + vk * dt` реально предсказывает следующее положение. Шум сенсора становится пренебрежимо малым по сравнению с `vk * dt`.
3. α = 0.75 в динамике работает «как надо» — быстро реагирует. Проблема только в статике, где `vk ≈ 0` и `xk_pred ≈ xk`, и innovation = весь шум сенсора.

То есть фильтр **скрывает проблему только тогда, когда она не видна**. Это типичный симптом «коэффициенты подобраны на динамическом тесте».

## 3. Построчный разбор `HeadingFilter.java`

Полный файл — 163 строки. Разбираю ключевые места.

### 3.1. Класс-документация и константы

```java
// HeadingFilter.java:9

*     → DEADBAND (0.5°) → не дёргать UI на микро-колебаниях
```

**Комментарий враньё.** В коде ниже:

```java
// HeadingFilter.java:20
private static final float MAX_DELTA_DEG_PER_SEC = 70f;
// ...
// HeadingFilter.java:36
private static final double DEADBAND_DEG = 0.5;
```

Константы совпадают с комментарием, но комментарий в строке 9 обещает `α=0.75, β=0.3`, а в README.md (строка 46) написано `Alpha-Beta(α=0.6, β=0.2)`. README и код расходятся. Это значит, что в какой-то момент правили коэффициенты и не обновили README — признак отсутствия CI-проверки соответствия docs/code.

### 3.2. CLAMP — корректный, но бесполезный в статике

```java
// HeadingFilter.java:82-89
double diff = rawHeading - xk;
while (diff > 180) diff -= 360;
while (diff < -180) diff += 360;
double maxDelta = MAX_DELTA_DEG_PER_SEC * dt;
if (Math.abs(diff) > maxDelta) {
    clamped = xk + Math.signum(diff) * maxDelta;
    clamped = (clamped % 360 + 360) % 360;
}
```

Unwrap сделан правильно (циклы while). `% 360 + 360) % 360` — стандартная нормализация, OK.

**Проблема:** clamp считает относительно `xk` (внутреннее состояние фильтра), а не `lastOutputDeg`. Это значит, что если α-β отстал (например, после резкого разгона), clamp будет ограничивать **скорость возврата**, создавая lasting lag. Правильно клампать относительно `lastOutputDeg` (то, что видит пользователь).

### 3.3. MEDIAN FILTER — работает, но окно 3 бесполезно

```java
// HeadingFilter.java:94-102
if (medianCount < MEDIAN_WINDOW) {
    medianBuf[medianCount++] = clamped;
} else {
    medianBuf[0] = medianBuf[1];
    medianBuf[1] = medianBuf[2];
    medianBuf[2] = clamped;
}
double median = (medianCount >= MEDIAN_WINDOW) ? computeMedian() : clamped;
```

Median(3) удаляет **точечные** выбросы (single outlier). Непрерывный шум (σ ≈ 2°) им не подавляется. На практике window=5 или window=7 даёт лучший эффект для импульсного шума.

Реализация через сдвиг массива — O(n), но при n=3 это 3 операции, пренебрежимо.

`computeMedian()`:

```java
// HeadingFilter.java:143-148
private double computeMedian() {
    double a = medianBuf[0], b = medianBuf[1], c = medianBuf[2];
    if ((a >= b && a <= c) || (a >= c && a <= b)) return a;
    if ((b >= a && b <= c) || (b >= c && b <= a)) return b;
    return c;
}
```

Корректно. Без аллокаций. ОК.

### 3.4. ALPHA-BETA — главная проблема

```java
// HeadingFilter.java:108-126
// Unwrap
double uDiff = median - unwrapped;
while (uDiff > 180) uDiff -= 360;
while (uDiff < -180) uDiff += 360;
unwrapped += uDiff;

// Predict
double xkPred = xk + vk * dt;
double vkPred = vk;

// Innovation
double innovation = unwrapped - xkPred;

// Update
xk = xkPred + alpha * innovation;
vk = vkPred + beta * innovation / dt;   // ← ПРОБЛЕМА
```

**Unwrap** корректный — поддерживает `unwrapped` в монотонной шкале.

**Predict** стандартный α-β.

**Update** для `xk` корректный.

**Update для `vk`** — содержит деление на `dt`. В классическом α-β фильтре:

```
xk = xk_pred + α · innovation
vk = vk_pred + (β / T) · innovation
```

Здесь `T = dt`. Математически — правильно. НО: при `dt → 0` (быстрый спам сенсора, например 200 Гц всплеск) `vk` взлетает. Это усугубляет джиттер.

**Решение:** зафиксировать `dt`:

```java
double dt = 1.0 / 50.0;  // ожидаемая частота сенсора
// либо: если реальный dt в [10ms, 30ms], использовать 20ms; иначе — реальный
```

### 3.5. DEADBAND — слишком мал

См. RC-3 выше. Значение 0.5° не выполняет свою функцию.

Дополнительно: deadband считает относительно `lastOutputDeg`, но возвращает `lastOutputDeg` без обновления `xk`. Это значит, что фильтр **накапливает** разницу между `xk` и `lastOutputDeg` — если пользователь медленно повернёт на 0.4°/с, фильтр будет отставать всё сильнее, пока разница не превысит 0.5°, и тогда произойдёт скачок. Лучше обновлять `lastOutputDeg = xk` в момент, когда возвращаем `lastOutputDeg` (то есть «садим» xk на lastOutputDeg):

```java
if (displayDiff < deadband) {
    xk = lastOutputDeg;  // подтягиваем состояние к видимому
    return lastOutputDeg;
}
```

## 6. Построчный разбор `WindEKF.java`

Файл 112 строк. **Самая проблемная часть с точки зрения корректности.**

### 6.1. Это не EKF

```java
// WindEKF.java:14
public class WindEKF {
```

**Расширенный фильтр Калмана (EKF)** требует:
1. Prediction step: `x_pred = f(x_k)`, `P_pred = F·P·F^T + Q`
2. Update step: `K = P_pred·H^T·(H·P_pred·H^T + R)^{-1}`, `x = x_pred + K·innovation`, `P = (I - K·H)·P_pred`

В коде:
- Нет матрицы ковариации `P`
- Нет prediction step (состояние постоянно)
- Нет `Q` (process noise), нет `R` (measurement noise)
- `k` — это не Kalman gain, а просто learning rate

Это **градиентный спуск** с эвристическим decay. Не EKF. Название вводит в заблуждение любого, кто читает код или документацию.

### 6.2. Сам алгоритм

```java
// WindEKF.java:38-66
public void update(double airspeed, double gpsVx, double gpsVy) {
    double dx = gpsVx - windU;
    double dy = gpsVy - windV;
    double mag = Math.sqrt(dx * dx + dy * dy);

    if (mag < 0.1 || airspeed < 1.0) return;

    double k0 = -scaleFactor * dx / mag * k;
    double k1 = -scaleFactor * dy / mag * k;
    double k2 = mag * 1e-5;

    double error = airspeed - scaleFactor * mag;

    windU += k0 * error;
    windV += k1 * error;
    scaleFactor += k2 * error;
    // ...
    k = k + 0.01 * (0.01 - k);
```

Модель: `airspeed_measured = scale_factor · |gps_velocity - wind|`. Это физически корректно — воздушная скорость равна путевой минус ветер.

`k0 = -scaleFactor * dx / mag * k` — это `∂airspeed/∂windU`, то есть якобиан. Корректно.

`error = airspeed - scaleFactor * mag` — невязка. Корректно.

`windU += k0 * error` — gradient descent step. Корректно для оптимизации `error²` по `windU`.

**НО:**

- `k2 = mag * 1e-5` — совсем другой масштаб для `scaleFactor`. Это потому, что `scaleFactor` безразмерный (1.0), а `windU` в м/с. Но `1e-5` — магическое число без обоснования.
- `k = k + 0.01 * (0.01 - k)` — exponential decay `k → 0.01` со скоростью 0.01 за шаг. Это означает, что через ~460 шагов `k` станет < 0.011, и фильтр перестанет обновляться (`getQuality()` вернёт 3 — "stabilized"). На 1 Гц GPS это 8 минут. После этого ветер **не обновляется**, даже если он изменился.

**Правильная реализация EKF** (упрощённо, для линейной модели):

```java
public class WindEKF {
    // State
    private final double[] x = new double[3]; // [windU, windV, scaleFactor]
    // Covariance
    private final double[][] P = new double[3][3];
    // Process noise
    private static final double Q_WIND = 0.5;     // м/с за шаг²
    private static final double Q_SCALE = 0.001;
    // Measurement noise
    private static final double R_AIRSPEED = 1.0; // м/с²

    public void update(double airspeed, double gpsVx, double gpsVy) {
        // Predict (constant state, only P grows)
        P[0][0] += Q_WIND;
        P[1][1] += Q_WIND;
        P[2][2] += Q_SCALE;

        // Measurement: airspeed = scale * |gps - wind|
        double dx = gpsVx - x[0];
        double dy = gpsVy - x[1];
        double mag = Math.sqrt(dx*dx + dy*dy);
        if (mag < 0.1 || airspeed < 1.0) return;

        // Jacobian H = [d(airspeed)/d(windU), d/d(windV), d/d(scale)]
        double[] H = new double[3];
        H[0] = -x[2] * dx / mag;
        H[1] = -x[2] * dy / mag;
        H[2] = mag;

        // Innovation
        double y = airspeed - x[2] * mag;

        // S = H·P·H^T + R
        double S = 0;
        for (int i = 0; i < 3; i++)
            for (int j = 0; j < 3; j++)
                S += H[i] * P[i][j] * H[j];
        S += R_AIRSPEED;

        // K = P·H^T / S
        double[] K = new double[3];
        for (int i = 0; i < 3; i++) {
            for (int j = 0; j < 3; j++) K[i] += P[i][j] * H[j];
            K[i] /= S;
        }

        // Update state
        for (int i = 0; i < 3; i++) x[i] += K[i] * y;

        // Update P = (I - K·H)·P
        for (int i = 0; i < 3; i++) {
            for (int j = 0; j < 3; j++) {
                double kh = 0;
                for (int l = 0; l < 3; l++) kh += K[i] * (l == j ? 1 : 0) * H[l];
                // упрощено: (I - K H^T) P
                P[i][j] = P[i][j] - K[i] * H[j] * P[i][j]; // аппроксимация
            }
        }

        // Clamp scale
        if (x[2] < 0.5) x[2] = 0.5;
        if (x[2] > 1.5) x[2] = 1.5;
    }
    // ...
}
```

### 6.3. `getWindDirectionDeg` — корректный

```java

    double dir = Math.toDegrees(Math.atan2(-windU, -windV));
    return (dir + 360) % 360;
}
```

`atan2(-windU, -windV)` — направление ОТКУДА дует ветер. Корректно, если `windU/windV` — это «куда дует» (компоненты вектора ветра). Но в коде `windU = gpsVx - airspeed_vector_x`, что является скоростью воздуха относительно земли = куда воздух движется. Так что инверсия через `-` даёт «откуда». OK.

### 6.4. `getQuality` — слабая метрика

```java
// WindEKF.java:93-99
public int getQuality() {
    if (updateCount < 5) return 0;
    if (k < 0.02) return 3;
    if (k < 0.05) return 2;
    if (k < 0.1) return 1;
    return 0;
}
```

Quality = функция от `k`. После 460 шагов `k` всегда станет < 0.02 → quality = 3. Это не метрика качества оценки, это метрика «сколько шагов прошло». Реальная quality должна учитывать:
- `P[0][0] + P[1][1]` (дисперсия ветра) — мала = хорошо
- Количество обновлений за последние N минут
- Невязку `error` в скользящем окне

### 6.5. Нет `predict` для ветра при изменении условий

Если ветер реально меняется (фронт, порыв), EKF должен через `Q` «открыть» ковариацию и адаптироваться. В текущем коде `k → 0.01` навсегда, и фильтр не сможет перестроиться.

`new Date()` + `String.format` — 2 аллокации на кадр для времени, которое меняется раз в секунду. Кэшировать:

```java
private long lastSysTimeUpdateMs;
private String lastSysTimeText;

public void drawSystemTime(Canvas c, float cx, float y) {
    long now = System.currentTimeMillis();
    if (lastSysTimeText == null || now - lastSysTimeUpdateMs > 1000) {
        lastSysTimeText = String.format("%tT", new java.util.Date(now));
        lastSysTimeUpdateMs = now;
    }
    c.drawText(lastSysTimeText, cx, y, sysTimePaint);
}
```

## 10. Конкурентность и потоки — сводка

### 10.1. Карта потоков

| Поток | Что делает |
|---|---|
| Main (UI) | `onDraw`, `onResume/onPause`, `renderHandler`, `bgHandler`, `simHandler`, `testHandler` |
| Sensor thread (SensorManager) | `onSensorChanged` → `SensorController` updates |
| FusedLocationProvider thread | `onLocationResult` → `GpsManager` updates |
| AudioTrack thread | `VarioSoundManager` playback |
| TTS thread | `TextToSpeech` callbacks |

Все колбеки FusedLocation и TTS приходят на main looper (потому что так зарегистрировано). SensorManager — на своём потоке.

### 10.2. Критические race conditions

| Где | Проблема | Влияние |
|---|---|---|
| `SensorController.rotationMatrix` (final float[9]) | мутируется в sensor thread, читается в UI thread через `getRotationMatrix()` | UI может прочитать наполовину обновлённую матрицу → некорректный accel-to-world transform |
| `SensorController.heading` (volatile float) | пишется из sensor thread (updateCompass) и UI thread (updateGpsHeading) | lost updates, divergence headingFilter state |
| `HeadingFilter.xk, vk, unwrapped` (не volatile) | мутируются из обоих потоков | разрушение внутреннего состояния фильтра |
| `GpsManager.gpsSpeed, gpsHeading, gpsLat, gpsLon` (volatile) | пишутся из FusedLocation callback, читаются из UI/sensor thread | ok (volatile), но не atomic — можно прочитать lat старую и lon новую |
| `MainActivity.thermals` (synchronized) | OK, но `ThermalBlip` поля mutable, читаются вне sync | partial reads |

### 10.3. Решение для rotationMatrix

```java
// Текущее:
private final float[] rotationMatrix = new float[9];
public float[] getRotationMatrix() { return rotationMatrix; }  // ← прямой доступ!

// Правильно:
private final Object rotMatrixLock = new Object();
private final float[] rotationMatrixInternal = new float[9];
private volatile float[] rotationMatrixSnapshot = new float[9];

private void updateCompass() {
    // ... fill rotationMatrixInternal ...
    synchronized (rotMatrixLock) {
        System.arraycopy(rotationMatrixInternal, 0, rotationMatrixSnapshot, 0, 9);
    }
}

public float[] getRotationMatrix() {
    // Возвращаем snapshot — immutable для caller
    return rotationMatrixSnapshot;
}
```

Или проще — `AtomicReference<float[]>` с заменой массива на новый при каждом обновлении (но это аллокация).

### 10.4. Решение для HeadingFilter

```java
// В SensorController:
private final Object headingLock = new Object();

private void updateCompass() {
    // ...
    synchronized (headingLock) {
        heading = (float) headingFilter.update(newHeading, SystemClock.elapsedRealtime());
    }
}

public void updateGpsHeading(float gpsHeading) {
    synchronized (headingLock) {
        // ...
        heading = (float) headingFilter.update(gpsHeading, SystemClock.elapsedRealtime());
    }
}
```

Слабое место — блокировка sensor thread на UI thread. На 50 Гц это допустимо (короткая критическая секция).

Лучше — выделить весь sensor-fusion в `HandlerThread` с одной очередью сообщений:

```java
private final HandlerThread fusionThread = new HandlerThread("fusion");
private Handler fusionHandler;

// onCreate:
fusionThread.start();
fusionHandler = new Handler(fusionThread.getLooper());

// все обновления heading — через fusionHandler.post(...)
```

## 11. Память и производительность — сводка

### 11.1. Аллокации на кадр (P1)

| Файл:строка | Что | Частота | Решение |
|---|---|---|---|
| `MainActivity.java:1949` | `new Paint(Paint.ANTI_ALIAS_FLAG)` в blind mode | 30/сек | вынести в поле |
| `MainActivity.java:2229` | `new float[2]` для trail distance | 200 точек × 30 FPS = 6000/сек | переиспользовать массив |
| `MainActivity.java:2245-2247` | `markerPxBuf/Py/IsEntry` arrays | 30/сек | вынести в поля фиксированного размера |
| `MainActivity.java:2290` | `new ThermalBlip(t)` копирование | 12 × 30 = 360/сек | unmodifiableList или copy-on-write |
| `RadarRenderer.java:340` | `new Rect(...)` | 30/сек | вынести в поле |
| `UiManager.java:351, 356` | `String.format("%.0fм", alt)` | 60/сек | StringBuilder + Formatter |
| `UiManager.java:337` | `new Date()` + `String.format("%tT", ...)` | 30/сек | кэшировать на 1 сек |

Суммарно: ~7000 аллокаций/сек на main thread. На свежих флагманах это незаметно, на 3–4 летних телефонах среднего сегмента — periodic GC pauses 5–20 мс, что видно как micro-stutters.

### 11.2. Sensor delay

`SensorManager.SENSOR_DELAY_GAME` = ~50 Гц (на самом деле 16.67 мс = 60 Гц на большинстве устройств). Для параплана это overkill — частота микрораскачки крыла 0.25–2.5 Гц (из README), достаточно 25 Гц сенсора. Это снизит CPU и батарею.

**Рекомендация:** `sensorManager.registerListener(this, accel, 40000);` — 25 Гц явно.

### 11.3. Барометр

На большинстве телефонов барометр выдаёт ~10 Гц. Если запрашивать `SENSOR_DELAY_GAME` (50 Гц), всё равно получим 10 Гц, но sensor manager будет пытаться будить поток чаще. Лучше `100000` (10 Гц явно).

## 12. Жизненный цикл Android — сводка

### 12.1. Утечки

| Что | Где | Утечка |
|---|---|---|
| `Handler.postDelayed` в `onCreate` (auto-calibration) | `MainActivity.java:354` | если активити уничтожится за 1 сек, Handler удержит Runnable → утечка MainActivity |
| `Handler.postDelayed` для tilt calibration | `MainActivity.java:448, 451` | вложенные postDelayed, ещё хуже |
| TTS инициализация | `MainActivity.java:360` | `TextToSpeech` удерживает context; в `onDestroy` вызывается `tts.shutdown()`, но если `onDestroy` не успеет — утечка |
| `staticMapLoader` callback с `runOnUiThread` | `MainActivity.java:288` | lambda удерживает `MainActivity` |
| `simHandler`, `bgHandler`, `testHandler` — поля класса, не static | `MainActivity.java:171, 179, 182, 210` | Handler удерживает Looper.MainLooper, а Runnable — outer class; OK, пока callback'и снимаются в `onDestroy` |

**Решение:** все `postDelayed` в `onCreate` должны либо:
1. Использовать `view.postDelayed` (привязка к View, автоматически очищается при разрушении View).
2. Или хранить `Runnable` в поле и снимать в `onDestroy`.

### 12.2. Foreground service lifecycle

```java
// MainActivity.java:374
startForegroundService(new Intent(this, ThermalRadarService.class));
```

`startForegroundService` требует, чтобы сервис в течение 5 секунд вызвал `startForeground(...)`. Если сервис не успеет (например, из-за тяжёлой инициализации), Android кидает `ForegroundServiceDidNotStartInTimeException` (CRASH на Android 12+). Нужен тест на медленных устройствах.

### 12.3. Permission flow

```java
// MainActivity.java:250-260
if (android.os.Build.VERSION.SDK_INT >= android.os.Build.VERSION_CODES.M) {
    if (checkSelfPermission(...) != GRANTED && ...) {
        requestPermissions(...);
    }
}
```

`Build.VERSION.SDK_INT >= M` (23) — `minSdk = 26`, всегда true. Дед-код.

`POST_NOTIFICATIONS` (Android 13+) не запрашивается runtime. На Android 13+ foreground service notification не покажется, и пользователь не увидит, что прибор работает в фоне. Нужно:

```java
if (Build.VERSION.SDK_INT >= 33) {
    if (checkSelfPermission(Manifest.permission.POST_NOTIFICATIONS) != GRANTED) {
        requestPermissions(new String[]{Manifest.permission.POST_NOTIFICATIONS}, 1002);
    }
}
```

### 13.1. Тесты


Один файл юнит-тестов. Не проверял содержимое детально, но размер кодовой базы (5000+ строк) и наличие критичных алгоритмов (CirclingManager, WindEKF, ThermalLocator, HeadingFilter) требуют **минимум**:

- `UnusedResources`, `HardcodedText`
- `ObsoleteLintCustomCheck`
- `UnsafeNativeCodeLocation`

Рекомендуется добавить в CI:

```bash

```


`proguard-rules.pro` — нужно проверить, что keep-правила есть для:

Без keep-правил релизный APK может крашиться на старте сервиса.

## 14. Безопасность и разрешения

### 14.1. Manifest

```xml
<!-- AndroidManifest.xml -->
<uses-permission android:name="android.permission.INTERNET" />
<uses-permission android:name="android.permission.ACCESS_NETWORK_STATE" />
```

INTERNET добавлен в v0.2.0 для OSM-карты. **НО:** StaticMapLoader обращается к `https://tile.openstreetmap.org/` напрямую, без user-agent. OSM Tile Usage Policy требует:

```
User-Agent: TERMO1-Radar/0.2.12 (contact@example.com)
```

Без UA OSM может забанить IP после нескольких сотен запросов. Нужен `HttpURLConnection.setRequestProperty("User-Agent", ...)`.

### 14.2. NETWORK_STATE

`ACCESS_NETWORK_STATE` запрошен, но в коде я не нашёл проверки `ConnectivityManager`. Скорее всего, добавлен «на всякий случай». Если не используется — удалить.

### 14.3. Cleartext traffic

`AndroidManifest.xml` не содержит `android:usesCleartextTraffic`. По умолчанию на `targetSdk = 34` cleartext запрещён, что правильно. Но StaticMapLoader должен ходить только по HTTPS — проверить.

### 14.4. Backup

```xml
<!-- AndroidManifest.xml:21 -->
android:allowBackup="false"
```

OK — чувствительные данные (IGC-логи с GPS-координатами) не пойдут в Google Drive backup.

## 17. Что в коде действительно хорошо (для баланса)

Чтобы ревью было конструктивным, отмечу решения, которые **не нужно трогать** и которые можно брать за образец в других частях кода:

1. **`VarioManager`** — хорошая декомпозиция: SRP, чистый API, без аллокаций в hot path, volatile поля для cross-thread reads. Это пример, на который `SensorController` должен быть похож.
2. **`IgcLogger`** — стандартный IGC-формат реализован корректно (FR/B/H/L/G записи). Совместимость с XCSoar/SeeYou — это серьёзный плюс.
3. **`HeadingFilter.computeMedian`** — без аллокаций, через a/b/c. Эталонная реализация для window=3.
4. **`SensorController.updateDeclination`** — правильное использование `GeomagneticField`, кэширование на 1 минуту.
5. **`WindStore`** — слоистое хранение ветра по высоте (100 м слои) с weighted average. Это правильная авиационная модель (ветр меняется с высотой).
6. **`CirclingManager` state machine** — порт XCSoar CirclingComputer сделан добросовестно: 4 состояния, гистерезис, поворот по total angle. Не业余.
7. **Foreground service с `foregroundServiceType="location"`** — правильно для Android 14+.
8. **Кэширование Paint в `UiManager`** — `private final Paint ... = new Paint(...)` в полях, не в onDraw. Кроме слепого режима (§7.4).
9. **`ThermalBlip`** — clean model class с жизненным циклом (`isAlive(now)`).
10. **Логика вычисления магнитного склонения и `getTrueHeading`** — корректная.
